Handout: Maven-Plugins – Die Werkzeuge des Build-Prozesses

6.1 Was sind Plugins?

  • Plugins sind zentrale Bausteine in Maven.

  • Sie erledigen die eigentliche Arbeit in den verschiedenen Phasen des Build-Lifecycles:

    • Kompilieren des Codes
    • Ausführen von Tests
    • Erzeugen von JAR-/WAR-Dateien
    • Erstellen von Berichten, u. v. m.

Wie funktionieren Plugins?

  • Jede Phase im Maven-Lifecycle (wie compile, test, package) ruft eins oder mehrere Goals von Plugins auf.

  • Beispiel:

    • In der Phase compile wird das Goal compile vom maven-compiler-plugin ausgeführt.
    • In der Phase test wird das Goal test vom maven-surefire-plugin ausgeführt.

Beispiele wichtiger Plugins

Plugin Zweck
maven-compiler-plugin Kompiliert den Quellcode
maven-surefire-plugin Führt die Unit-Tests aus
maven-assembly-plugin Baut ausführbare JARs, ZIPs usw.
maven-jar-plugin Erstellt das JAR-Artefakt
maven-clean-plugin Löscht Build-Verzeichnisse

6.2 Plugins konfigurieren in der pom.xml

Grundaufbau eines <plugin>-Blocks:

<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-compiler-plugin</artifactId>
      <version>3.10.1</version>
      <configuration>
        <source>17</source>
        <target>17</target>
      </configuration>
    </plugin>
    <!-- Weitere Plugins ... -->
  </plugins>
</build>

Hinweise:

  • Plugins werden im <build>-Abschnitt unter <plugins> eingetragen.
  • Viele Plugins laufen out of the box – für spezielle Anforderungen gibt es <configuration>.

Wo findet man passende Plugins?

  • Offiziell: Maven Plugin Documentation
  • Suche: search.maven.org – nach „maven-plugin“ filtern.
  • Tipp: Die wichtigsten Plugins sind von org.apache.maven.plugins (offiziell, stabil, gut dokumentiert).

6.3 Das Maven-Surefire-Plugin

Wofür ist das Surefire-Plugin zuständig?

  • Es führt alle Unit-Tests in der Phase test aus.
  • Erkennt automatisch alle Testklassen, die bestimmte Namensmuster haben (z. B. *Test.java, *Tests.java, Test*.java).

Standardverhalten:

  • Läuft automatisch bei mvn test, mvn package, etc.

  • Benötigt ab JUnit 5 eine kleine Anpassung:

    • JUnit 5 ist modular aufgebaut.
    • Es muss mindestens die Engine junit-jupiter-engine als Dependency vorhanden sein.
    • Teilweise muss das Surefire-Plugin in der pom.xml angepasst werden, um Tests zu erkennen.

Beispiel: Surefire-Konfiguration für JUnit 5

1. Dependencies in der pom.xml:

<dependency>
  <groupId>org.junit.jupiter</groupId>
  <artifactId>junit-jupiter</artifactId>
  <version>5.10.0</version>
  <scope>test</scope>
</dependency>

2. Plugin-Konfiguration (empfohlen):

<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-surefire-plugin</artifactId>
      <version>3.2.5</version>
      <configuration>
        <useModulePath>false</useModulePath>
      </configuration>
    </plugin>
  </plugins>
</build>

Erläuterung:

  • Die Version ab 3.0.0-M5 unterstützt JUnit 5 offiziell.
  • <useModulePath>false</useModulePath> deaktiviert das Java-Modulsystem beim Testen – das ist bei manchen Projekten oder Bibliotheken hilfreich, falls du z. B. noch kein module-info.java verwendest oder Bibliotheken auf dem Classpath brauchst.

Übung: Surefire für JUnit 5 selbst konfigurieren

  1. Lege ein Maven-Projekt mit JUnit 5 als Dependency an.

  2. Füge das Surefire-Plugin wie oben in deine pom.xml ein.

  3. Schreibe eine einfache Testklasse in src/test/java:

    import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class BeispielTest {
    @Test
    void testAddition() {
        assertEquals(4, 2 + 2);
    }
}

  4. Führe den Test aus:

    mvn test

  5. Beobachte die Konsole: Siehst du „BUILD SUCCESS“ und „Tests run: ...“? Glückwunsch, deine JUnit-Tests wurden erfolgreich mit Surefire ausgeführt!

Zusammenfassung

  • Plugins sind die „Werkzeuge“ im Maven-Build, sie führen die einzelnen Aufgaben aus.
  • Konfiguration erfolgt im <build>-Abschnitt der pom.xml.
  • Surefire ist für Tests zuständig; bei JUnit 5 sollte man Version und ggf. Optionen wie <useModulePath> beachten.
  • Eigene Anpassungen an Plugins sind einfach möglich und oft für spezielle Projekte nötig.
