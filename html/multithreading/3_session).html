Handout: Kommunikation und Zusammenarbeit von Threads in Java

3.1 Gemeinsame Nutzung von Ressourcen (Shared Objects, Membervariablen)

Was bedeutet „gemeinsame Nutzung“?

  • Mehrere Threads greifen auf dieselben Variablen, Objekte oder Ressourcen zu (z. B. Datenbanken, Dateien, Speicherobjekte).

  • Beispiele:

    • Zwei Threads bearbeiten dieselbe Liste.
    • Ein Thread schreibt einen Wert in eine Variable, ein anderer liest ihn.

In Java:

  • Alle Membervariablen und Objekte, die von mehreren Threads referenziert werden können, sind potenziell „Shared Objects“.

  • Typisches Szenario:

    class Counter {
    private int value = 0;

    public void increment() {
        value++;
    }

    public int getValue() {
        return value;
    }
}

    Mehrere Threads teilen sich dasselbe Counter-Objekt.

Risiko: Ohne Synchronisation können Inkonsistenzen oder Datenverluste auftreten!

3.2 Probleme ohne Synchronisation: Race Conditions, inkonsistente Zustände

a) Race Conditions (Wettlaufsituationen)

  • Definition: Ein Race Condition entsteht, wenn der Programmausgang davon abhängt, wie der Scheduler (die JVM/das Betriebssystem) die Threads abwechselnd ausführt.

  • Beispiel (gefährlich):

    class UnsafeCounter {
    private int count = 0;

    public void increment() {
        count++; // Nicht thread-sicher!
    }
}

// Zwei Threads rufen increment() "gleichzeitig" auf.

  • Gefahr: Zwei Threads lesen denselben Wert, inkrementieren getrennt und schreiben beide das gleiche Ergebnis zurück → ein Inkrement geht verloren.

b) Inkonsistente Zustände

  • Beschreibung: Wenn ein Thread eine Ressource liest, die gerade von einem anderen Thread verändert wird, können „kaputte“ oder unvollständige Werte entstehen.

  • Beispiel:

    • Thread A liest eine Liste, während Thread B gerade ein Element hinzufügt oder entfernt.
    • Führt oft zu ConcurrentModificationException oder fehlerhaften Ergebnissen.

c) Sichtbarkeitsprobleme (Visibility Problems)

  • Problem: Änderungen eines Threads an gemeinsamen Variablen werden von anderen Threads nicht oder nicht rechtzeitig gesehen.

  • Ursache: CPU-Caches und Optimierungen der JVM.

  • Beispiel-Lösung:

    • Mit dem Schlüsselwort volatile können Sichtbarkeitsprobleme bei einfachen Variablen verhindert werden.

3.3 Thread-Zustände und Status-Abfragen

a) Wichtige Zustände (vgl. 2.3):

  • NEW: Thread-Objekt existiert, wurde aber noch nicht gestartet (start() nicht aufgerufen).
  • RUNNABLE: Thread ist bereit oder läuft gerade.
  • BLOCKED: Thread wartet auf einen Lock (Monitor).
  • WAITING/TIMED_WAITING: Thread wartet auf ein Signal oder eine bestimmte Zeitspanne (wait(), join(), sleep()).
  • TERMINATED: Thread hat die run()-Methode verlassen, ist also fertig.

Status-Abfrage in Java:

  • Mit der Methode getState() kann der aktuelle Zustand eines Threads abgefragt werden:

    Thread t = new Thread();
System.out.println(t.getState()); // NEW
t.start();
System.out.println(t.getState()); // RUNNABLE oder anderes, je nach Timing

  • Mit isAlive() kannst du prüfen, ob ein Thread aktuell noch läuft:

    if (t.isAlive()) {
    System.out.println("Thread läuft noch.");
}

Praktischer Tipp:

  • Beobachte mit kleinen Testprogrammen, wie sich die Zustände beim Starten, Warten und Beenden von Threads ändern.

Zusammenfassung

  • Threads können gemeinsam Ressourcen nutzen – das macht Programme leistungsfähig, aber auch fehleranfällig.
  • Ohne Synchronisation treten Race Conditions und Inkonsistenzen auf.
  • Das Verständnis von Thread-Zuständen und Statusabfragen ist Grundlage für stabile Multithread-Programme.
  • Geeignete Synchronisationsmechanismen (Thema der nächsten Einheit) sind essenziell für die korrekte Zusammenarbeit von Threads in Java.
