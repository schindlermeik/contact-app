Handout: Synchronisation von Threads in Java

4.1 Notwendigkeit der Synchronisation

Warum ist Synchronisation wichtig?

  • Wenn mehrere Threads gleichzeitig auf gemeinsame Ressourcen (z. B. Variablen, Listen, Bankkonten) zugreifen und mindestens einer davon die Ressource verändert, kann es zu Race Conditions und inkonsistenten Zuständen kommen.

Beispiele:

a) Zähler (Counter)

class Counter {
    private int count = 0;
    public void increment() {
        count++; // Nicht thread-sicher!
    }
}
  • Zwei Threads könnten denselben Wert lesen und überschreiben → Inkrement geht verloren.

b) Banking

class BankAccount {
    private double balance = 0;
    public void deposit(double amount) {
        balance += amount; // Race Condition möglich!
    }
}

c) Listen

  • Wenn mehrere Threads gleichzeitig einer nicht-synchronisierten Liste Elemente hinzufügen oder entfernen, kann die Liste „kaputt“ gehen oder es entstehen Exceptions (ConcurrentModificationException).

Fazit:

  • Synchronisation schützt den Zugriff auf kritische Abschnitte des Codes, in denen gemeinsam genutzte Ressourcen verändert werden.

4.2 Synchronisierte Methoden und Blöcke mit dem Schlüsselwort synchronized

Synchronisierte Methoden:

public synchronized void increment() {
    count++;
}
  • Der gesamte Methodenrumpf ist geschützt – es kann immer nur ein Thread gleichzeitig diese Methode für ein Objekt ausführen.

Synchronisierte Blöcke:

public void increment() {
    synchronized(this) {
        count++;
    }
}
  • Nur der Block im synchronized-Abschnitt wird geschützt.
  • Praktisch, wenn nur ein Teil der Methode synchronisiert werden muss oder wenn auf andere Monitore als this synchronisiert werden soll.

Synchronisieren auf beliebige Objekte:

private final Object lock = new Object();
public void doSomething() {
    synchronized(lock) {
        // kritischer Bereich
    }
}

Achtung:

  • Synchronisation kann zu Performanceeinbußen führen – immer nur wirklich notwendige Bereiche schützen!
  • Bei statischen Methoden synchronisiert synchronized auf die Klasse, nicht auf das Objekt.

4.3 volatile-Variablen: Wann und warum verwenden?

Was macht volatile?

  • Das Schlüsselwort volatile stellt sicher, dass eine Variable immer direkt aus dem Hauptspeicher gelesen und dorthin geschrieben wird – nicht aus CPU- oder Thread-eigenen Caches.
  • Damit sind Änderungen einer volatile-Variable für alle Threads sofort sichtbar.

Beispiel:

private volatile boolean running = true;

public void stop() {
    running = false; // Änderung wird für alle Threads sichtbar
}

Wann verwenden?

  • Bei einfachen Variablen (z.B. Flags), die von mehreren Threads gelesen und beschrieben werden, aber bei denen kein zusammengesetzter Vorgang (z.B. „lesen, rechnen, schreiben“) erforderlich ist.
  • Achtung: volatile reicht NICHT aus, um komplexe Operationen (wie count++) thread-sicher zu machen!

4.4 Java-Objektmonitore (wait(), notify(), notifyAll())

Was ist ein Monitor?

  • Jeder Java-Objekt besitzt einen sog. „Monitor“, den Threads verwenden können, um sich zu koordinieren.

Wozu dienen diese Methoden?

  • Ermöglichen die Kommunikation zwischen Threads („Warte, bis eine Bedingung erfüllt ist“).

Beispiel: Producer-Consumer-Problem

class Buffer {
    private int value;
    private boolean available = false;

    public synchronized void put(int v) throws InterruptedException {
        while (available) wait();
        value = v;
        available = true;
        notifyAll();
    }

    public synchronized int get() throws InterruptedException {
        while (!available) wait();
        available = false;
        notifyAll();
        return value;
    }
}
  • wait() setzt den aufrufenden Thread in den WAITING-Zustand und gibt den Monitor frei.
  • notify() weckt einen wartenden Thread.
  • notifyAll() weckt alle wartenden Threads.

Wichtig:

  • wait(), notify(), notifyAll() dürfen nur innerhalb synchronisierter Kontexte aufgerufen werden (Methoden/Blöcke mit synchronized).

4.5 Fortgeschrittene Synchronisation

4.5.1 ReentrantLock und das java.util.concurrent.locks-Paket

  • ReentrantLock bietet gegenüber synchronized mehr Kontrolle, z. B. Lock-Timeouts, Interruptibilität, Fairness.

Beispiel:

import java.util.concurrent.locks.ReentrantLock;

class SafeCounter {
    private int count = 0;
    private final ReentrantLock lock = new ReentrantLock();

    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }
}

4.5.2 Weitere Mechanismen

Semaphore:

  • Begrenzen den gleichzeitigen Zugriff auf eine Ressource (z. B. maximal 5 gleichzeitige Zugriffe).
import java.util.concurrent.Semaphore;
Semaphore sem = new Semaphore(5);
sem.acquire(); // Zugang holen
sem.release(); // Zugang freigeben

CountDownLatch:

  • Synchronisiert mehrere Threads, die auf ein Ereignis warten sollen (z. B. „Starte, wenn alle bereit sind“).
import java.util.concurrent.CountDownLatch;
CountDownLatch latch = new CountDownLatch(3);
latch.await(); // warten, bis latch.countDown() dreimal aufgerufen wurde
latch.countDown(); // von anderen Threads aufgerufen

CyclicBarrier:

  • Ähnlich wie CountDownLatch, aber mehrfach wiederverwendbar. Threads warten, bis eine bestimmte Anzahl angekommen ist.

4.6 Deadlocks in Java: Erkennen, vermeiden, beheben

Was ist ein Deadlock?

  • Zwei oder mehr Threads warten gegenseitig auf Ressourcen, sodass keiner mehr weiterlaufen kann.

Typisches Beispiel:

// Thread 1:
synchronized(obj1) {
    Thread.sleep(50);
    synchronized(obj2) { ... }
}
// Thread 2:
synchronized(obj2) {
    Thread.sleep(50);
    synchronized(obj1) { ... }
}
  • Jeder Thread hält ein Lock und wartet auf das andere → Stillstand.

Erkennen:

  • Programm „hängt“ fest.
  • Mit Tools wie jstack, VisualVM, IDE-Thread-Dumps analysierbar.

Vermeiden:

  • Immer in derselben Reihenfolge auf Ressourcen locken!
  • Möglichst wenige und kleine kritische Abschnitte verwenden.
  • Lock-Hierarchien festlegen und dokumentieren.
  • Hochwertige Synchronisationsklassen (Locks, Semaphoren) bevorzugen.

Beheben:

  • Lock-Logik überprüfen und ggf. Reihenfolge vereinheitlichen.
  • Zeitlimits (tryLock(long timeout)) einsetzen.

Zusammenfassung

  • Synchronisation ist in Java essenziell, wenn mehrere Threads auf gemeinsam genutzte Ressourcen zugreifen.
  • Java bietet verschiedene Möglichkeiten: synchronized, volatile, Objektmonitore (wait, notify), sowie fortgeschrittene Klassen wie ReentrantLock, Semaphore, CountDownLatch, CyclicBarrier.
  • Deadlocks sind ein zentrales Risiko bei Multithreading – sie müssen durch Design und Tools erkannt und vermieden werden.
  • Die richtige Strategie (und möglichst wenig gemeinsam genutzte Daten!) macht Multithread-Programme robust und performant.
