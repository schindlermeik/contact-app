Handout: Best Practices und typische Stolperfallen in Java (Multithreading)

6.1 Thread-Leaks vermeiden (Threads korrekt beenden)

Was ist ein Thread-Leak?

  • Ein Thread-Leak bezeichnet Threads, die nach Beendigung ihrer eigentlichen Aufgabe weiterlaufen und nicht korrekt beendet werden.
  • Folge: Erhöhter Ressourcenverbrauch (CPU, Speicher), potenzieller Absturz durch „OutOfMemoryError“ bei zu vielen Threads.

Ursachen:

  • Endlosschleifen oder fehlerhafte Abbruchbedingungen im Thread.
  • Threads warten unbegrenzt auf Ressourcen/Ereignisse, die nie eintreten.

Best Practices:

  • Abbruch-Bedingungen prüfen: Setze eine Abbruch-Flag (z. B. volatile boolean running), das regelmäßig geprüft wird:

    class Worker extends Thread {
    private volatile boolean running = true;
    public void run() {
        while (running) {
            // Arbeit
        }
    }
    public void stopWorker() { running = false; }
}

  • Interrupts verwenden: Ermögliche dem Thread, durch interrupt() sanft beendet zu werden. Prüfe regelmäßig Thread.interrupted():

    while (!Thread.currentThread().isInterrupted()) { ... }

  • Thread Pools bevorzugen: Vermeide direkte Thread-Erzeugung, nutze stattdessen ExecutorService und rufe am Ende shutdown() auf.

6.2 Fehlerquellen beim Synchronisieren: zu grobe/zu feine Sperren, unnötige Synchronisation

a) Zu grobe Sperren (Coarse-grained Locking)

  • Problem: Zu große synchronisierte Bereiche verringern Parallelität und führen zu Performance-Einbußen.

  • Beispiel:

    public synchronized void processAll() { 
    // großer, langsamer Codeblock
}

  • Lösung: Nur wirklich kritische Codeabschnitte schützen.

b) Zu feine Sperren (Fine-grained Locking)

  • Problem: Viele, kleine Locks können zu Deadlocks und komplexen Fehlerbildern führen.
  • Beispiel: Verschachtelte Synchronisation auf mehreren Objekten.

c) Unnötige Synchronisation

  • Problem: Synchronisation auf Code, der nur lokale Variablen verwendet (gar keine gemeinsame Ressource).
  • Lösung: Synchronisation nur für gemeinsam genutzte Daten verwenden.

Faustregeln:

  • So wenig wie nötig synchronisieren, aber so viel wie nötig!
  • Kritische Abschnitte so klein wie möglich halten.
  • Synchronisation möglichst immer auf das gleiche Objekt/Monitor.

6.3 Debugging von Multithreading-Problemen

Typische Probleme:

  • Deadlocks
  • Race Conditions (unerwartete Werte)
  • Livelocks (Threads arbeiten, aber kommen nie zum Ziel)

Werkzeuge und Methoden:

  • Thread Dumps: Zeigen alle laufenden Threads, deren Zustände und Locks.

    • In der IDE: „Dump Threads“-Funktion (meist bei „Run“/„Debug“).

    • Über Konsole:

      jstack <PID>

  • VisualVM: Kostenloses Java-Tool, zeigt laufende Threads, Locks und CPU-Auslastung.

  • Logging: Logge kritische Aktionen, Zustände und Locking-Verhalten zur Laufzeit (ggf. mit Thread-IDs).

  • Timeouts setzen: Vermeide endloses Warten, indem du Zeitlimits für Locking-Operationen (tryLock(timeout)) oder wait(timeout) verwendest.

Tipps:

  • Suche nach „WAITING“ oder „BLOCKED“ Threads im Dump – oft Anzeichen für Deadlocks.
  • Prüfe, ob Locks immer wieder freigegeben werden (finally-Block bei lock.lock()!).

6.4 Performance-Überlegungen: Synchronisations-Overhead, Contention, False Sharing

a) Synchronisations-Overhead

  • Synchronisation kostet Zeit (Lock holen/freigeben) – insbesondere bei hoher Thread-Anzahl und geringem „echten“ Parallelismus.
  • Übermäßige Synchronisation macht Programme langsam.

b) Contention (Lock-Konkurrenz)

  • Wenn viele Threads auf denselben Lock warten, kommt es zu „Lock Contention“.

  • Folge: Threads stehen „Schlange“, statt parallel zu arbeiten.

  • Lösung:

    • Arbeitslast auf mehrere unabhängige Ressourcen (Sharding, Partitioning) verteilen.
    • Nur das Nötigste synchronisieren.

c) False Sharing

  • Definition: Verschiedene Threads schreiben auf Variablen, die auf demselben CPU-Cache-Line liegen.

  • Obwohl sie logisch unabhängig sind, stören sich die Threads gegenseitig durch den Hardware-Cache.

  • Folge: Unerwartete Performance-Einbrüche.

  • Lösung:

    • Variablen, die oft von unterschiedlichen Threads beschrieben werden, räumlich im Speicher trennen (Padding).
    • Im Normalfall selten problematisch, aber in hochperformanten Programmen relevant.

  2. Wann ist Synchronisation notwendig – und wann überflüssig?
  3. Was ist der Unterschied zwischen Deadlock, Livelock und Contention?
  4. Welche Tools oder Methoden nutzt du, um Multithreading-Probleme zu analysieren?
  5. Warum kann zu viel Synchronisation die Performance sogar verschlechtern?

Zusammenfassung

  • Korrektes Beenden von Threads ist essentiell, um Ressourcenlecks zu vermeiden.
  • Synchronisiere nur kritische Abschnitte und mit so wenigen Locks wie möglich.
  • Nutze Tools wie VisualVM und Thread Dumps, um Fehler zu erkennen und zu analysieren.
  • Berücksichtige Performance-Aspekte wie Synchronisations-Overhead, Contention und False Sharing, um skalierbare Multithread-Anwendungen zu schreiben.
  • Überprüfe regelmäßig, ob dein Code thread-sicher UND performant ist – das schließt sich nicht aus!
