Handout: Grundlagen – Future und Callable in Java

2.1 Das Future-Interface: Limitierungen und typische Anwendungsfälle

Was ist ein Future?

  • Das Interface Future<V> (seit Java 5) repräsentiert das Ergebnis einer asynchronen Berechnung, die eventuell noch nicht abgeschlossen ist.
  • Future wird typischerweise verwendet, wenn du eine Aufgabe in einem separaten Thread ausführen willst und später das Ergebnis benötigst.

Wie verwendet man ein Future?

  • Eine Aufgabe wird (meist über einen Executor) gestartet und liefert ein Future-Objekt zurück.
  • Über Methoden wie get() kannst du auf das Ergebnis warten (blockierend).

Beispiel:

import java.util.concurrent.*;

ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Integer> future = executor.submit(() -> {
    Thread.sleep(1000);
    return 42;
});

System.out.println("Berechnung läuft im Hintergrund...");

// Hole das Ergebnis (blockiert ggf. bis das Ergebnis bereit ist)
Integer result = future.get();
System.out.println("Ergebnis: " + result);

executor.shutdown();

Typische Anwendungsfälle für Future

  • Aufgaben mit langer Laufzeit (z. B. Netzwerkanfragen, rechenintensive Jobs).
  • Aufgaben, deren Ergebnis erst später benötigt wird.
  • Einfache Parallelisierung ohne explizite Thread-Verwaltung.

Wichtige Methoden von Future

  • get(): Holt das Ergebnis, wartet ggf. blockierend.
  • get(long timeout, TimeUnit unit): Holt das Ergebnis, aber nur für maximal die angegebene Zeit.
  • cancel(boolean mayInterruptIfRunning): Versucht, die Aufgabe abzubrechen.
  • isDone(): Prüft, ob die Berechnung abgeschlossen ist.
  • isCancelled(): Prüft, ob die Aufgabe abgebrochen wurde.

2.2 Das Callable-Interface und Unterschied zu Runnable

Das Runnable-Interface

  • Runnable ist ein funktionales Interface mit der Methode void run().
  • Rückgabewert: keiner (void).
  • Kann keine geprüften Ausnahmen (Checked Exceptions) werfen.

Das Callable-Interface

  • Callable<V> ist ebenfalls ein funktionales Interface, eingeführt mit Java 5.
  • Methode: V call() throws Exception
  • Rückgabewert: beliebiger Typ V.
  • Kann Ausnahmen werfen.

Vergleich:

Runnable Callable
Rückgabewert nein (void) ja (beliebiger Typ V)
Exceptions ungeprüft (unchecked) geprüft (checked)
Verwendung Thread, Executor Executor, Future

Beispiel:

Callable<String> task = () -> {
    Thread.sleep(500);
    return "Fertig!";
};

ExecutorService executor = Executors.newSingleThreadExecutor();
Future<String> future = executor.submit(task);
System.out.println(future.get()); // Gibt "Fertig!" aus
executor.shutdown();

2.3 Probleme mit klassischen Futures (z.B. Blockieren, fehlendes Chaining)

1. Blockierende get()-Aufrufe

  • Die Methode get() blockiert den aktuellen Thread, bis das Ergebnis vorliegt.
  • Folge: Das Programm wartet untätig – das widerspricht oft der Idee asynchroner Programmierung!

Beispielproblem:

Future<String> f = ...;
String result = f.get(); // Blockiert bis Ergebnis da ist
  • Währenddessen kann der Thread nichts anderes tun.

2. Kein Chaining/Verketten von Aufgaben

  • Mit dem klassischen Future kann man keine Verarbeitungskette („Wenn fertig, dann…“) definieren.
  • Das macht komplexere asynchrone Abläufe (z. B. mehrere abhängige Berechnungen) sehr umständlich.

Beispielproblem:

  • Du kannst nicht einfach sagen: „Wenn das Ergebnis fertig ist, dann führe Methode X aus, sonst Y.“

3. Keine eingebaute Fehlerbehandlung

  • Fehler müssen separat über try/catch nach dem get()-Aufruf behandelt werden.
  • Keine Möglichkeit, direkt auf Erfolg oder Fehler asynchron zu reagieren.

4. Kein Zusammenschalten mehrerer Futures

  • Es ist nicht möglich, mehrere Futures einfach zu kombinieren (z. B. „Wenn alle Aufgaben fertig sind, dann…“).

Zusammenfassung

  • Das klassische Future-Interface erlaubt einfache asynchrone Berechnungen mit Ergebnis.
  • Callable ist flexibler als Runnable, da es Rückgabewerte und Ausnahmen unterstützt.
  • Die klassische Future-API ist jedoch eingeschränkt: kein echtes Chaining, keine einfache Fehlerbehandlung, blockierende Abfragen.
  • Für moderne, flexible asynchrone Programmierung ist CompletableFuture die bessere Wahl – dazu mehr im nächsten Abschnitt!
