Handout: Integration von CompletableFuture in bestehende Architekturen

5.1 Thread-Management mit Executor

Warum ist Thread-Management wichtig?

  • Standardmäßig nutzt CompletableFuture den ForkJoinPool.commonPool(), der für viele Anwendungen ausreicht, aber nicht immer optimal ist (z. B. bei limitierten Ressourcen oder besonderen Anforderungen).

Eigenen Executor verwenden

  • Du kannst steuern, in welchem Thread-Pool die asynchronen Aufgaben laufen.

Beispiel:

import java.util.concurrent.*;

ExecutorService pool = Executors.newFixedThreadPool(4);

CompletableFuture<Integer> future = CompletableFuture.supplyAsync(() -> {
    // Code läuft im eigenen Thread-Pool
    return 42;
}, pool);

// Nach Abschluss: Pool sauber herunterfahren
future.thenRun(pool::shutdown);

Wann eigenen Executor?

  • Für Anwendungen mit besonderen Lastprofilen (z. B. Webserver).
  • Wenn du Prioritäten, Ressourcen oder Isolation genauer steuern möchtest.
  • Für bessere Kontrolle über das Thread-Verhalten, Logging, Monitoring etc.

5.2 Zusammenspiel von CompletableFuture mit Blocking- und Non-Blocking-APIs

a) Integration mit Blocking-APIs

  • Häufig müssen bestehende, blockierende Methoden in asynchronen Workflows genutzt werden.

  • Lösung: Solche Methoden in einen eigenen Thread auslagern:

    CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    return blockingCall();
});

b) Integration mit Non-Blocking-APIs

  • Moderne Frameworks (z. B. HTTP-Clients ab Java 11) unterstützen non-blocking (asynchrone) APIs, die selbst Futures oder Callbacks liefern.
  • Diese lassen sich direkt mit CompletableFuture verkettet weiterverarbeiten.

Beispiel mit Java 11 HTTP-Client:

import java.net.URI;
import java.net.http.*;
import java.util.concurrent.*;

HttpClient client = HttpClient.newHttpClient();
HttpRequest request = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com"))
    .build();

CompletableFuture<HttpResponse<String>> response =
    client.sendAsync(request, HttpResponse.BodyHandlers.ofString());

response.thenAccept(res -> System.out.println(res.body()));

Best Practice:

  • Blockierende APIs möglichst kapseln, um die Vorteile von asynchroner Verarbeitung nicht zu verlieren.
  • Wenn möglich, auf moderne non-blocking Bibliotheken umsteigen.

5.3 Abwärtskompatibilität: Migration von klassischen Future-Konzepten zu CompletableFuture

a) Übergang von Future zu CompletableFuture

  • Viele ältere Libraries liefern noch Future<T>.
  • Mit Java 8+ kannst du ein CompletableFuture „manuell“ aus einem klassischen Future befüllen:

Beispiel:

Future<String> legacyFuture = ...; // Von alter API
CompletableFuture<String> cf = new CompletableFuture<>();

// Ergebnis in separatem Thread holen und im CompletableFuture setzen
Executors.newSingleThreadExecutor().submit(() -> {
    try {
        cf.complete(legacyFuture.get());
    } catch (Exception e) {
        cf.completeExceptionally(e);
    }
});
  • Vorteil: Du kannst dann alle Methoden von CompletableFuture (Chaining, Fehlerbehandlung usw.) nutzen!

b) Eigene API modernisieren

  • Für eigene Methoden, die bisher Future geliefert haben, biete eine CompletableFuture-Variante an.
  • Schrittweise Migration möglich: Zunächst Adapter, dann Refactoring.

5.4 Kombination mit Streams und Collections

a) Mit Collections viele asynchrone Aufgaben abfeuern

  • Typischer Anwendungsfall: Für viele Daten asynchrone Tasks starten und alle Ergebnisse einsammeln.

Beispiel:

List<String> urls = List.of("https://a", "https://b", "https://c");
List<CompletableFuture<String>> futures = urls.stream()
    .map(url -> CompletableFuture.supplyAsync(() -> fetchData(url)))
    .toList();

CompletableFuture<Void> all = CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]));
all.thenRun(() -> {
    List<String> results = futures.stream()
        .map(CompletableFuture::join)
        .toList();
    System.out.println("Alle fertig: " + results);
});

b) Asynchrone Verarbeitung in Streams

  • Verkettung von Futures mit Streams erlaubt kompakte und leistungsfähige asynchrone Verarbeitung.
  • thenApply, thenCompose, usw. lassen sich pro Element anwenden.

Zusammenfassung

  • Mit CompletableFuture lassen sich moderne, asynchrone Architekturen elegant aufbauen.
  • Der Einsatz eines eigenen Executors sorgt für Flexibilität und bessere Ressourcenkontrolle.
  • Bestehende (blocking) APIs können sauber integriert oder schrittweise abgelöst werden.
  • Mit Streams und CompletableFutures sind leistungsstarke, parallele Datenverarbeitungen einfach umsetzbar.
  • Für eine nachhaltige Migration: Schrittweise Übergänge, Adapter und Refactoring nutzen!
