Handout: Grundlagen – Panache-Entity und Panache-Repository

3.1 Verwendung der Panache-Entity (extends PanacheEntity / PanacheEntityBase)

Was ist eine Panache-Entity?

Eine Panache-Entity ist eine Java-Klasse, die direkt als Datenbanktabelle abgebildet wird (wie bei JPA), aber zusätzlich von Panache-Klassen erbt. Dadurch bekommst du zahlreiche Methoden zum Arbeiten mit der Datenbank „geschenkt“.

Es gibt zwei Möglichkeiten:

  1. Active Record Pattern: Vererbung von io.quarkus.hibernate.orm.panache.PanacheEntity

    • Du schreibst:

      @Entity
public class User extends PanacheEntity {
    public String name;
    public String email;
}

    • Vorteil: Das Feld id (Long, Primary Key) ist schon dabei!

    • Du nutzt sofort Methoden wie persist(), delete(), listAll() direkt an der Entity.

  2. Custom ID oder mehr Flexibilität: Vererbung von PanacheEntityBase

    • Du schreibst z. B.:

      @Entity
public class Product extends PanacheEntityBase {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    public Long productId;
    public String name;
}

    • Vorteil: Du kannst den Primärschlüssel oder Felder nach Bedarf selbst gestalten.

3.2 Automatisierte CRUD-Methoden durch Vererbung

Sobald deine Entity von PanacheEntity (oder PanacheEntityBase) erbt, hast du viele Methoden für CRUD-Operationen automatisch zur Verfügung:

Typische Methoden

  • persist() Speichert ein neues Objekt in der Datenbank.
  • delete() Löscht das aktuelle Objekt aus der Datenbank.
  • listAll() Liefert alle Objekte der Entity als Liste.
  • find() / findById() Sucht nach Objekten mit bestimmten Werten.
  • count() Gibt die Anzahl der Einträge zurück.

Beispiel:

// Neuen Benutzer speichern:
User u = new User();
u.name = "Anna";
u.email = "anna@mail.de";
u.persist();

// Alle Benutzer laden:
List<User> users = User.listAll();

// Einen Benutzer mit bestimmter ID finden:
User anna = User.findById(1L);

// Benutzer nach Name suchen:
List<User> list = User.find("name", "Anna").list();

// Einen Benutzer löschen:
u.delete();

Vorteil: Du brauchst keinen EntityManager, keine eigenen CRUD-Methoden – alles ist schon da!

3.3 Verwendung von Panache Repository Pattern (PanacheRepository<T>)

Nicht immer möchtest du das Active Record Pattern nutzen, z. B. wenn du Fachlogik trennen oder Entitäten "dumm" halten willst.

Dafür gibt es das Repository Pattern mit Panache:

Vorgehen:

  1. Entität ganz normal schreiben (wie bei klassischem JPA):

    @Entity
public class Book {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    public Long id;
    public String title;
}

  2. Repository-Interface/Bean anlegen:

    @ApplicationScoped
public class BookRepository implements PanacheRepository<Book> {
    // Optional: Eigene Methoden für spezielle Abfragen
}

    Oder mit PanacheRepositoryBase (für benutzerdefinierte IDs):

    public class BookRepository implements PanacheRepositoryBase<Book, String> { ... }

  3. Verwendung im Code (per Dependency Injection):

    @Inject
BookRepository bookRepo;

public void foo() {
    List<Book> all = bookRepo.listAll();
    Book found = bookRepo.findById(5L);
    bookRepo.persist(new Book("Panache 101"));
}

Vorteil:

  • Trennung zwischen Datenmodell und Datenzugriff.
  • Bessere Testbarkeit und fachliche Kapselung.
  • Keine statischen Methoden – Instanzbasiert.

3.4 Unterschiede und Einsatzszenarien (Active Record vs. Repository Pattern)

Active Record Pattern (PanacheEntity)

  • Vorteile:

    • Extrem kompakt, alles an einem Ort.
    • Einfach für kleine bis mittlere Projekte oder schnelle Prototypen.
    • Wenig Overhead, schnelle Entwicklung.

  • Nachteile:

    • Geschäftslogik und Persistenz sind vermischt.
    • Für große Projekte kann die Wartbarkeit leiden.

Repository Pattern (PanacheRepository)

  • Vorteile:

    • Saubere Trennung von Datenmodell und Datenzugriff.
    • Gut für große und wartbare Systeme.
    • Einfachere Unit-Tests (Mocking).

  • Nachteile:

    • Etwas mehr Code und Struktur notwendig.
    • Leicht weniger „magisch“ als Active Record.

Wann welches Pattern?

  • Active Record:

    • Microservices mit simpler Domäne
    • CRUD-zentrierte Anwendungen
    • Prototyping

  • Repository:

    • Komplexere Domänenmodelle
    • Unternehmen mit klaren Architekturvorgaben
    • Projekte mit Fokus auf Testbarkeit und Erweiterbarkeit

Fazit

  • Mit Panache kannst du entweder das Active Record Pattern (schnell, kompakt) oder das Repository Pattern (sauber, testbar) nutzen.
  • Beide Varianten bieten dir viele Methoden für CRUD-Operationen „out-of-the-box“.
  • Wähle das passende Pattern nach Projektgröße und Anforderungen!

Praxis-Tipp: Starte mit Active Record für kleine Demos und steige auf Repository um, wenn das Projekt größer wird.
