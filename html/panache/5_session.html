Handout: Erweiterte Features von Panache

5.1 Sortierung, Paging und Filterung

A) Sortierung

Mit Panache kannst du Ergebnisse komfortabel sortieren, z. B. aufsteigend oder absteigend nach einem bestimmten Feld.

Beispiel:

// Alle User alphabetisch sortiert nach Name:
List<User> users = User.list("ORDER BY name ASC");

// Mit dem Sort-Objekt:
List<User> users = User.findAll(Sort.by("name")).list();

// Absteigend nach Erstellungsdatum:
List<User> users = User.findAll(Sort.by("createdAt").descending()).list();

B) Paging (Paginierung)

Große Ergebnismengen kannst du mit Panache bequem seitenweise („paginiert“) abfragen.

Beispiel:

// Hole die erste Seite mit 10 Einträgen:
List<User> firstPage = User.findAll().page(Page.ofSize(10)).list();

// Zu einer bestimmten Seite springen (Seitenzählung beginnt bei 0!):
List<User> secondPage = User.findAll().page(Page.of(1, 10)).list();

Weitere Methoden:

  • .page() setzt die aktuelle Seite und Größe.
  • .nextPage() und .previousPage() blättern.
  • .hasNextPage(), .hasPreviousPage() prüfen auf Nachbarseiten.
  • .pageCount() liefert die Gesamtseitenzahl.

C) Filterung

Du kannst Ergebnisse auch nach bestimmten Kriterien filtern:

// Benutzer mit bestimmtem Namen:
List<User> maxUsers = User.find("name", "Max").list();

// Benutzer mit bestimmtem Alter und Name:
List<User> result = User.find("name = ?1 AND age > ?2", "Max", 18).list();

Mit Sortierung kombinieren:

List<User> result = User.find("age > ?1", Sort.by("name").descending(), 18).list();

5.2 Arbeiten mit optionalen Rückgabewerten (z. B. Optional<T>)

Manchmal gibt es Suchabfragen, bei denen es entweder genau ein oder kein Ergebnis gibt. Panache nutzt hierfür das Optional-Pattern aus Java 8+.

Beispiel:

// Suche nach einem Benutzer mit bestimmter E-Mail:
Optional<User> maybeUser = User.find("email", "max@example.com").firstResultOptional();

if (maybeUser.isPresent()) {
    User user = maybeUser.get();
    // Weiterverarbeitung
} else {
    // Kein Treffer, entsprechend reagieren
}

Vorteile:

  • NullPointerExceptions werden vermieden.
  • Klar definierter Umgang mit "keine Treffer".

5.3 Umgang mit Beziehungen (One-to-Many, Many-to-One)

Auch komplexere Datenmodelle mit Beziehungen lassen sich mit Panache und klassischen JPA-Annotationen abbilden.

A) One-to-Many & Many-to-One

Beispiel: Ein Benutzer hat mehrere Bestellungen (User → Order).

@Entity
public class User extends PanacheEntity {
    public String name;

    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    public List<Order> orders;
}

@Entity
public class Order extends PanacheEntity {
    public String product;
    public double price;

    @ManyToOne
    @JoinColumn(name = "user_id")
    public User user;
}

Verwendung:

User user = new User();
user.name = "Anna";

Order order1 = new Order();
order1.product = "Buch";
order1.price = 12.99;
order1.user = user;

user.orders = Arrays.asList(order1);

// Persistiert User und die Bestellungen dank Cascade:
user.persist();

B) Weitere Beziehungen

Auch Many-to-Many oder One-to-One lassen sich wie gewohnt mit den JPA-Annotationen realisieren.

5.4 Nutzung von Transaktionen in Panache

Datenbankänderungen sollten immer in Transaktionen erfolgen, damit Konsistenz und Rollbacks bei Fehlern gewährleistet sind.

A) Automatische Transaktionen in Quarkus

In Quarkus werden Datenbankoperationen standardmäßig innerhalb von Transaktionen ausgeführt, wenn sie in Methoden stehen, die mit @Transactional annotiert sind.

Beispiel:

import io.quarkus.hibernate.orm.panache.PanacheEntity;
import javax.transaction.Transactional;

@ApplicationScoped
public class UserService {

    @Transactional
    public void createUserWithOrder() {
        User user = new User();
        user.name = "Tom";
        user.persist();

        Order order = new Order();
        order.product = "PC";
        order.user = user;
        order.persist();
        // Beide Persist-Vorgänge laufen in EINER Transaktion
    }
}

B) Rollback bei Fehlern

Wenn innerhalb einer @Transactional-Methode eine Exception auftritt, werden alle bis dahin vorgenommenen Änderungen automatisch zurückgerollt.

C) Tipp: Explizite Transaktionen vermeiden

Vertraue dem Transaktionsmanagement von Quarkus und Panache, schreibe nur dann eigene Logik, wenn du sehr spezielle Anforderungen hast.

Zusammenfassung – Erweiterte Features von Panache

  • Sortierung & Paging erleichtern die Arbeit mit großen Datenmengen und machen APIs nutzerfreundlicher.
  • Optionale Rückgabewerte verhindern Fehler und machen die Codebasis robuster.
  • Beziehungen sind wie bei JPA abbildbar und werden durch Panache nicht eingeschränkt.
  • Transaktionen sorgen für konsistente Daten und sicheres Arbeiten – meist „out of the box“ durch Quarkus.

Praxis-Checkliste:

  1. Ergebnisse nach bestimmten Kriterien sortieren, filtern und paginieren.
  2. Optional verwenden, um sicher auf Einzelergebnisse zuzugreifen.
  3. Beziehungen zwischen Entitäten abbilden und testen.
  4. Transaktionen mit @Transactional absichern – Fehler sicher abfangen.
