Handout: Optional in der Praxis

1. Optionals als Rückgabewert von Methoden

Der wichtigste und sinnvollste Einsatz von Optional ist als Rückgabewert von Methoden, die möglicherweise keinen Wert liefern können.

Vorteile:

  • Macht für Aufrufer sofort klar, dass der Wert fehlen kann – explizit im Methodenkopf sichtbar.
  • Der Umgang mit dem Fehlen eines Wertes wird erzwingbar und ist keine „böse Überraschung“ mehr.

Beispiel:

public Optional<String> findeBenutzernameById(String id) {
    if (datenbank.containsKey(id)) {
        return Optional.of(datenbank.get(id));
    } else {
        return Optional.empty();
    }
}

// Aufrufer:
Optional<String> benutzername = findeBenutzernameById("4711");
benutzername.ifPresent(System.out::println);

Oder mit Defaultwert:

String name = findeBenutzernameById("4711").orElse("unbekannt");

2. Optional als Alternative zu Rückgabe von null

Warum nicht einfach null zurückgeben?

  • null ist unsichtbar und wird leicht vergessen → führt zu NullPointerException.
  • Mit Optional wird das Fehlen eines Werts typisiert und dokumentiert.
  • Der Code wird wartbarer und sicherer – zwingt zum bewussten Umgang.

Refactoring-Beispiel:

Vorher (unsicher):

public String findeEmail(String name) {
    if (db.contains(name)) {
        return db.getEmail(name);
    } else {
        return null; // Risiko!
    }
}

Nachher (sicher):

public Optional<String> findeEmail(String name) {
    if (db.contains(name)) {
        return Optional.of(db.getEmail(name));
    } else {
        return Optional.empty();
    }
}

Jetzt ist klar: Der Aufrufer muss mit beiden Fällen rechnen und kann nicht versehentlich eine NPE auslösen.

3. Optionals als Methode-Parameter? (Best Practice!)

Sollte man Optional als Parameter-Typ in Methoden verwenden?

Klare Empfehlung:

Optionals sind für Rückgabewerte gedacht, NICHT als Parameter für Methoden!

Warum?

  • Es gibt keinen echten Vorteil, einen Optional als Parameter zu nutzen.
  • Für optionale Parameter (die fehlen können), ist es besser, überladene Methoden oder Standardwerte zu verwenden.
  • Die Methode selbst sollte entscheiden, wie sie mit null/fehlenden Werten umgeht und ggf. Optional zurückgeben – aber nicht erwarten.

Schlecht:

public void bearbeiteBenutzer(Optional<String> name) { ... }

Besser:

public void bearbeiteBenutzer(String name) { ... }
// und dokumentieren, wie mit null umzugehen ist, falls möglich.

Oder:

public Optional<String> findeBenutzer(String id) { ... }

4. Praxis-Tipps

  • Verwende Optional immer für Rückgabewerte, wenn das Ergebnis fehlen kann.

  • Nutze für Parameter lieber klassische Ansätze:

    • Überladene Methoden
    • Standardparameter
    • Eindeutige Dokumentation und ggf. NotNull-Annotationen

  • Vermeide Optional in DTOs, Entities und bei Serialisierung – das kann zu Problemen führen.

5. Übungen zum Selbststudium

  1. Schreibe eine Methode, die einen optionalen Wert als Rückgabewert verwendet. Teste alle Varianten des Auslesens (z. B. mit ifPresent, orElse, etc.).
  2. Refactore eine Methode, die bisher null zurückgibt, auf Optional.
  3. Überlege für Methoden, die mehrere optionale Eingaben bekommen könnten, wie man diese eleganter gestaltet (z. B. durch Builder-Pattern oder Überladen).

6. Merksätze

  • Optional ist für Rückgabewerte, NICHT für Parameter gedacht!
  • Optional ersetzt unsichtbare null-Werte durch explizite, sichere API.
  • Optional erzwingt einen bewussten Umgang mit leeren Rückgabewerten.
