2.3 Arten von Exceptions: Checked vs. Unchecked

2.3.1 Unchecked Exceptions (RuntimeExceptions)

Was sind Unchecked Exceptions?

  • Unchecked Exceptions sind Laufzeit-Ausnahmen, die von der Klasse RuntimeException (bzw. ihren Unterklassen) erben.
  • Sie werden nicht vom Compiler dazu gezwungen, explizit behandelt zu werden (weder durch try-catch noch durch throws).
  • Treten typischerweise durch Programmierfehler oder logische Fehler im Code auf.

Typische Unchecked Exceptions

  • NullPointerException: Zugriff auf ein Objekt, das null ist.
  • ArrayIndexOutOfBoundsException: Zugriff auf ein Array außerhalb seiner Grenzen.
  • IllegalArgumentException: Ein ungültiges Argument wird an eine Methode übergeben.
  • ArithmeticException: Rechenfehler wie Division durch null.
  • ClassCastException: Fehlerhafter Typumwandlungsversuch.

Umgang mit Unchecked Exceptions

  • Prävention statt Behandlung: Der Code sollte so gestaltet werden, dass diese Ausnahmen im Normalfall nicht auftreten.
  • Beispiel: Vor dem Zugriff prüfen, ob eine Variable null ist.
public int getStringLength(String s) {
    if (s == null) {
        throw new IllegalArgumentException("String darf nicht null sein!");
    }
    return s.length();
}

2.3.2 Checked Exceptions

Was sind Checked Exceptions?

  • Checked Exceptions sind geprüfte Ausnahmen, die von Exception erben, aber nicht von RuntimeException.
  • Der Compiler erzwingt, dass diese Exceptions entweder abgefangen (try-catch) oder weitergegeben (throws) werden.
  • Sie treten häufig bei externen Fehlerquellen auf, die nicht durch einen Programmierfehler verursacht werden (z.B. I/O, Netzwerk).

Typische Checked Exceptions

  • IOException: Fehler beim Lesen/Schreiben von Daten (z.B. Datei nicht gefunden).
  • SQLException: Fehler beim Zugriff auf eine Datenbank.
  • ClassNotFoundException: Klasse zur Laufzeit nicht gefunden.
  • FileNotFoundException: Eine bestimmte Datei existiert nicht.

Umgang mit Checked Exceptions

  • Fehlerbehandlung erzwingen: Der Programmierer muss entscheiden, wie auf diese Ausnahme reagiert werden soll – z.B. Nutzer informieren, Alternative anbieten, Fehler weiterreichen.
  • Beispiel:
public void readFile(String filename) throws IOException {
    FileReader reader = new FileReader(filename); // Kann IOException werfen
    // Datei lesen ...
}

try {
    readFile("daten.txt");
} catch (IOException e) {
    System.out.println("Dateifehler: " + e.getMessage());
}

2.3.3 Praktische Beispiele für beide Arten

Beispiel 1: Unchecked Exception

public int divide(int a, int b) {
    return a / b; // Kann ArithmeticException werfen (b = 0)
}

public void run() {
    divide(10, 0); // Führt zu ArithmeticException, wenn b = 0
    // Wird meist nicht abgefangen, sondern als Programmierfehler betrachtet
}

Beispiel 2: Checked Exception

public void saveToFile(String filename, String data) throws IOException {
    FileWriter writer = new FileWriter(filename); // Kann IOException werfen
    writer.write(data);
    writer.close();
}

public void run() {
    try {
        saveToFile("output.txt", "Hallo Welt!");
    } catch (IOException e) {
        System.out.println("Fehler beim Schreiben: " + e.getMessage());
    }
}

Beispiel 3: Checked und Unchecked kombiniert

public void addTask(String title) {
    if (title == null || title.isEmpty()) {
        throw new IllegalArgumentException("Titel darf nicht leer sein!"); // Unchecked
    }
    // Checked Exception könnte z.B. beim Speichern auftreten:
    // saveToFile(title); // throws IOException
}

Zusammenfassung

Merkmal Checked Exception Unchecked Exception (Runtime)
Muss behandelt werden? Ja (try/catch oder throws) Nein
Ableitung von Exception (nicht Runtime) RuntimeException
Typische Ursache Externe Faktoren Programmierfehler
Beispiele IOException, SQLException NullPointerException, IllegalArgumentException
