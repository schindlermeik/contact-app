2.2 Die Java-Exception-Hierarchie

2.2.1 Überblick: Throwable, Error, Exception, RuntimeException

Die Wurzel: Throwable

  • In Java sind alle Fehler und Ausnahmen Objekte, die von der Basisklasse java.lang.Throwable erben.
  • Das ermöglicht es, Fehlerzustände objektorientiert zu behandeln und Details über die Art des Fehlers weiterzugeben.

Direkte Unterklassen von Throwable:

  • Error
  • Exception

Was ist ein Error?

  • Error repräsentiert schwerwiegende Fehler, die von der Java Virtual Machine (JVM) oder dem Laufzeitsystem ausgelöst werden.

  • Diese Fehler entstehen meist durch Systemprobleme, nicht durch einen Fehler im Anwendungsprogramm.

  • Beispiele:

    • OutOfMemoryError: Der Speicher ist erschöpft.
    • StackOverflowError: Der Aufruf-Stack ist zu tief (z.B. Endlosschleife mit Rekursion).

  • Wichtig:

    • Errors zeigen an, dass das Programm sich in einem Zustand befindet, aus dem es sich meistens nicht erholen kann.

Was ist eine Exception?

  • Exception steht für „gewöhnliche“ Ausnahmen, mit denen dein Programm rechnen sollte.

  • Sie können durch Fehler im Code, fehlerhafte Benutzereingaben oder Probleme mit externen Ressourcen (Dateien, Netzwerke) ausgelöst werden.

  • Beispiele:

    • IOException: Fehler beim Datei- oder Netzwerkzugriff.
    • FileNotFoundException: Datei existiert nicht.
    • NullPointerException: Zugriff auf eine Variable, die null ist.
    • IllegalArgumentException: Falsche Methoden-Parameter.

Checked vs. Unchecked Exceptions

  • Checked Exceptions:

    • Müssen im Code behandelt werden (try-catch oder throws).
    • Erben von Exception, aber nicht von RuntimeException.
    • Beispiele: IOException, SQLException.

  • Unchecked Exceptions:

    • Müssen nicht explizit behandelt werden.
    • Erben von RuntimeException.
    • Beispiele: NullPointerException, IllegalArgumentException, ArrayIndexOutOfBoundsException.

Was ist eine RuntimeException?

  • RuntimeException ist die Basisklasse für Exceptions, die während der Programmlaufzeit auftreten und meist durch Programmierfehler verursacht werden.

  • Beispiele:

    • NullPointerException
    • ArithmeticException (z.B. Division durch null)
    • ArrayIndexOutOfBoundsException

  • Sie müssen nicht abgefangen werden und werden häufig nicht behandelt, sondern durch Verbesserung des Codes vermieden.

Legende:

  • Von oben nach unten: Vererbungsbeziehungen.
  • Checked Exceptions: Von Exception abgeleitet, aber nicht von RuntimeException.
  • Unchecked Exceptions: Von RuntimeException abgeleitet.
  • Errors: Von Error abgeleitet.

2.2.3 Bedeutung von Error und warum man Fehler nicht behandeln sollte

Was genau ist ein Error?

  • Error-Objekte zeigen an, dass ein Problem im System oder der JVM selbst aufgetreten ist.

  • Beispiele:

    • OutOfMemoryError: Der JVM-Speicher reicht nicht mehr aus.
    • StackOverflowError: Zuviele Methodenaufrufe (z.B. Endlosrekursion).
    • InternalError: Interner Fehler der JVM.

Warum sollte man Error nicht behandeln?

  • Error signalisiert katastrophale Zustände, die das Programmieren außerhalb deiner Kontrolle liegen lassen.

  • Fehlerbehandlung (catch) ist zwar technisch möglich, aber fast nie sinnvoll:

    • Die JVM ist meist nicht mehr in der Lage, zuverlässig weiterzulaufen.
    • Ressourcen können inkonsistent oder beschädigt sein.
    • Ein Weiterarbeiten würde zu undefiniertem Verhalten führen.

  • Best Practice:

    • Schreibe niemals catch (Error e) – überlasse Errors der JVM.
    • Konzentriere dich auf die Behandlung von Exceptions, nicht Errors.

Merksatz:

Exceptions sind für Fehlerfälle, auf die dein Programm reagieren kann. Errors zeigen, dass etwas sehr Grundlegendes schiefgegangen ist – und sollten das Programm (meist) beenden.

Zusammenfassung & Reflexionsfragen

  • Alle Exceptions und Errors sind von Throwable abgeleitet.
  • Error: Schwerwiegende Fehler (z.B. OutOfMemory), nicht behandeln!
  • Exception: „Normale“ Ausnahmen, die im Programm behandelt werden können.
  • RuntimeException: Unchecked, meist Programmierfehler.
