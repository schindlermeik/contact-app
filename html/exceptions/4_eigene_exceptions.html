2.4 Eigene Exceptions schreiben

2.4.1 Warum eigene Exceptions?

Wozu braucht man eigene Exception-Klassen?

Java stellt bereits viele Exception-Klassen bereit. Trotzdem stößt man in der Praxis schnell an Grenzen, bei denen die Standard-Exceptions zu allgemein oder unpassend sind:

  • Klarheit & Lesbarkeit: Mit einer eigenen Exception kannst du spezifisch und aussagekräftig beschreiben, was im Programm schiefgelaufen ist. Beispiel: TaskNotFoundException ist viel verständlicher als eine allgemeine RuntimeException oder gar eine NullPointerException.
  • Sinnvolle Fehlerunterscheidung: Verschiedene Fehlerfälle können unterschiedlich behandelt werden. Mit eigenen Exception-Typen kannst du gezielt darauf reagieren.
  • Bessere Wartbarkeit: Fehlerursachen sind für dich und andere Entwickler*innen sofort nachvollziehbar.
  • Feingranulares Fehler-Handling: In komplexen Anwendungen (z.B. mit vielen Modulen oder Bibliotheken) ist eine gezielte Fehlerbehandlung oft nur mit eigenen Exception-Klassen möglich.

Praxisbeispiel: Du entwickelst eine ToDo-App. Wenn eine Aufgabe mit einer bestimmten ID nicht gefunden wird, willst du nicht einfach null zurückgeben oder eine allgemeine Exception werfen, sondern explizit einen eigenen Fehlerfall kennzeichnen: → TaskNotFoundException

2.4.2 Implementierung einer eigenen Exception

Schritt 1: Auswählen des Basistyps

  • Soll deine Exception checked oder unchecked sein?

    • Checked Exception: Erbt von Exception
    • Unchecked Exception: Erbt von RuntimeException

Schritt 2: Eigene Klasse erstellen

Beispiel: Eine Unchecked Exception

// Neue Exception: Unchecked, weil sie von RuntimeException erbt
public class TaskNotFoundException extends RuntimeException {
    // Standard-Konstruktor mit Nachricht
    public TaskNotFoundException(String message) {
        super(message);
    }
    // Optional: Konstruktor mit Ursache (Exception Chaining)
    public TaskNotFoundException(String message, Throwable cause) {
        super(message, cause);
    }
}

Beispiel: Eine Checked Exception

// Neue Exception: Checked, weil sie von Exception erbt
public class DataImportException extends Exception {
    public DataImportException(String message) {
        super(message);
    }
    public DataImportException(String message, Throwable cause) {
        super(message, cause);
    }
}

Schritt 3: Die Exception im Code verwenden

  • Exception werfen:

    if (!taskExists(id)) {
    throw new TaskNotFoundException("Aufgabe mit ID " + id + " wurde nicht gefunden!");
}

  • Exception behandeln:

    try {
    updateTask(5, "Neue Aufgabe");
} catch (TaskNotFoundException e) {
    System.out.println("Fehler: " + e.getMessage());
}

2.4.3 Anwendung und Vorteile im Code

Typische Anwendungsfälle für eigene Exceptions

  • Spezielle Fehlerfälle: Wenn ein bestimmtes Problem häufig und eindeutig auftritt, z.B. „Benutzer nicht gefunden“, „Ungültige Bestellung“, „Keine Berechtigung“.
  • Fein abgestufte Fehlerbehandlung: In komplexeren Anwendungen können so verschiedene Ausnahmen unterschiedlich behandelt oder an die Benutzeroberfläche gemeldet werden.
  • Bibliotheken/Frameworks: Eigene Exception-Klassen ermöglichen es Nutzern deiner Bibliothek, gezielt auf Fehlerfälle zu reagieren.

Vorteile im Code

  • Erhöhte Lesbarkeit und Verständlichkeit

    catch (TaskNotFoundException e) {
    // Jetzt ist klar, was passiert ist!
    ...
}

  • Bessere Fehleranalyse und Logging Durch spezifische Fehlertypen ist das Finden und Beheben von Fehlern deutlich einfacher.

  • Vermeidung von zu allgemeinen Fehlerbehandlungen Es wird verhindert, dass unterschiedliche Fehlerarten „in einen Topf“ geworfen werden.

Best Practice

  • Aussagekräftige Namen wählen: Nutze einen Namen, der den Fehlerfall exakt beschreibt.
  • Konstruktoren anbieten: Mindestens mit String message, besser auch mit Throwable cause (für Exception Chaining).
  • Javadoc hinzufügen: Dokumentiere, wann und warum diese Exception geworfen wird.

Zusammenfassung

  • Eigene Exceptions verbessern die Lesbarkeit, Wartbarkeit und Fehlerbehandlung deines Codes.
  • Sie sollten immer dann verwendet werden, wenn Standard-Exceptions zu ungenau sind oder du spezielle Fehlerfälle unterscheiden möchtest.
  • Die Implementierung ist einfach: Erbe von Exception oder RuntimeException und definiere sinnvolle Konstruktoren.
