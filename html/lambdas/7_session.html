Handout: Best Practices und Fallstricke bei Lambda-Ausdrücken

1. Lesbarkeit und Verständlichkeit von Lambdas

a) Klarheit geht vor Kürze

  • Schreibe Lambda-Ausdrücke so einfach und lesbar wie möglich.
  • Nutze aussagekräftige Parameternamen, besonders bei komplexeren Ausdrücken.
  • Halte Lambdas kurz: Ideal sind einzeilige Ausdrücke. Bei mehr Logik lieber einen Block { ... } verwenden – oder in eine Methode auslagern.

Schlecht:

list.forEach(x -> { if (x > 0) System.out.println(x); else if (x == 0) System.out.println("Null"); else ... });

Besser:

list.forEach(x -> System.out.println(x));

Oder für komplexere Fälle:

list.forEach(MyClass::druckeMitInfo);

b) Verwende Methodenreferenzen, wo sinnvoll

Wenn der Lambda-Ausdruck einfach nur eine vorhandene Methode aufruft, ist eine Methodenreferenz oft noch lesbarer:

namen.forEach(System.out::println);

2. Typkompatibilität und Fehlerquellen

a) Functional Interface erforderlich

  • Lambdas können nur verwendet werden, wo ein Functional Interface erwartet wird (also ein Interface mit genau einer abstrakten Methode).
  • Sonst: Compilerfehler!

Fehlerbeispiel:

interface MehrereMethoden { void a(); void b(); }
// MehrereMethoden f = () -> System.out.println("Test"); // Compilerfehler!

b) Typinferenz und explizite Typen

  • Der Compiler erkennt den Typ der Lambda-Parameter meist automatisch (Typinferenz).
  • Manchmal ist der Typ mehrdeutig – dann musst du ihn explizit angeben.

Beispiel:

Comparator<String> cmp = (a, b) -> a.length() - b.length();
// Wenn nicht klar: (String a, String b) -> ...

c) Zugriff auf Variablen (effectively final)

  • Lambdas dürfen nur auf lokale Variablen zugreifen, die „effectively final“ sind, also nach der Zuweisung nicht mehr geändert werden.

Fehlerbeispiel:

int faktor = 2;
list.forEach(x -> System.out.println(x * faktor));
faktor = 3; // Compilerfehler!

Lösung: Wert vor dem Lambda festlegen und nicht mehr ändern.

d) Checked Exceptions

  • Lambda-Ausdrücke können nur solche Ausnahmen werfen, die das Functional Interface zulässt.
  • Viele Standard-Functional-Interfaces deklarieren keine checked Exceptions.

Fehlerbeispiel:

Consumer<String> saver = s -> {
    try { writeToFile(s); }
    catch (IOException e) { ... } // Muss im Lambda gefangen werden
};

3. Wann Lambda-Ausdrücke und wann klassische Methoden verwenden?

a) Lambdas einsetzen, wenn ...

  • Das Verhalten kurz und einfach beschreibbar ist.
  • Die Funktion an eine andere Methode übergeben werden soll (z. B. forEach, filter, sort).
  • Es um eine einzelne, überschaubare Aktion geht.

b) Klassische Methoden oder Methodenreferenzen nutzen, wenn ...

  • Der Code länger oder kompliziert ist (mehr als 1–2 Zeilen Logik).
  • Du den Code mehrmals oder an verschiedenen Stellen brauchst.
  • Die Funktion benannt, dokumentiert oder getestet werden soll.
  • Die Funktion Zustand oder mehrere Schritte enthält.

Beispiel:

// Für einfache Filter:
list.stream().filter(s -> s.length() > 5);

// Für komplexe Filter besser:
list.stream().filter(MyClass::aufwendigeFilterLogik);

4. Übungen zum Selbststudium

  1. Schreibe einen Lambda-Ausdruck, der mehrere Bedingungen prüft. Lagere diese Bedingungen dann in eine separate Methode aus und vergleiche Lesbarkeit.
  2. Experimentiere mit einer lokalen Variable, die du im Lambda verwendest und dann ändern willst. Was passiert?
  3. Probiere einen Lambda-Ausdruck mit einem Interface, das mehr als eine abstrakte Methode hat – was sagt der Compiler?

5. Merksätze

  • Lesbarkeit geht vor Kürze: Nicht jeder Einzeiler ist besser als eine Methode!
  • Nur Functional Interfaces können mit Lambdas genutzt werden.
  • Typinferenz ist praktisch, aber manchmal muss der Typ explizit genannt werden.
  • Komplexere Logik lieber in Methoden auslagern und per Methodenreferenz nutzen.
  • Sei vorsichtig beim Zugriff auf lokale Variablen und bei Checked Exceptions.
