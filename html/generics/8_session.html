Handout: Best Practices beim Einsatz von Generics

1. Lesbarkeit und Verständlichkeit verbessern

Generics erhöhen die Flexibilität deines Codes, können aber bei unüberlegtem Einsatz auch schnell unübersichtlich werden. Beachte daher:

  • Kurze Methoden und Klassen: Halte generische Methoden möglichst kurz und verständlich. Lange Methoden mit vielen Typ-Parametern sind schwer zu überblicken.

  • Aussagekräftige Kommentare: Erkläre, wofür Typ-Parameter stehen und wie sie verwendet werden sollen. Beispiel:

    // T steht für den Element-Typ der Box
public class Box<T> { ... }

  • Klare Schnittstellen: Verwende Generics vor allem dort, wo sie den Code wirklich flexibler und sicherer machen – z. B. bei Datenstrukturen, Bibliotheken, Utility-Klassen.

  • Vermeide übermäßige Verschachtelung: Typen wie Map<String, List<Pair<Integer, Set<User>>>> sind schwer lesbar und sollten ggf. über Hilfsklassen/Typaliases vereinfacht werden.

2. Sinnvolle Namensgebung für Typ-Parameter

Typ-Parameter werden üblicherweise mit einem Großbuchstaben bezeichnet. Es gibt für bestimmte Fälle Konventionen:

Typ-Parameter Bedeutung / Verwendungszweck
T Type (beliebiger Typ)
E Element (Element einer Collection)
K Key (Schlüssel in einer Map)
V Value (Wert in einer Map)
N Number (häufig für Zahlenklassen)
S, U, V Weitere Typ-Parameter (Secondary, usw.)

Beispiele:

public class Box<T> { ... }
public interface List<E> { ... }
public interface Map<K, V> { ... }
public class Pair<T, U> { ... }
  • Merke: Typ-Parameter sollen kurz sein, aber ihre Funktion ausdrücken. In seltenen Fällen sind auch längere Bezeichner erlaubt (<ElementType>), aber das ist unüblich.

3. Wann Wildcards verwenden – und wann nicht?

Wildcards (?, ? extends T, ? super T) machen Methoden noch flexibler, erhöhen aber auch die Komplexität. Hier einige Faustregeln:

Wann solltest du Wildcards verwenden?

  • Wenn deine Methode mit beliebigen Subtypen eines Typs arbeiten soll, ohne sich für den genauen Typ zu interessieren.

    • Beispiel:

      public void printAll(List<?> list) { ... }
public double sum(List<? extends Number> numbers) { ... }

  • Wenn du eine API oder Library schreibst, die möglichst generisch und flexibel sein soll.

Wann solltest du Wildcards NICHT verwenden?

  • Wenn der Typ der Collection oder des Arguments bekannt und einheitlich ist. Beispiel:

    List<String> namen = ...;
// Verwende Methoden mit List<String> statt List<?>

  • Wenn du Elemente aus einer Collection herausschreiben möchtest und sicher sein musst, welchen Typ du verwendest. (→ Lieber festen Typ-Parameter verwenden.)

Die PECS-Regel (Merksatz):

  • Producer Extends, Consumer Super

    • Wenn du aus einer Collection liest (sie ist Datenproduzent), verwende <? extends T>.
    • Wenn du in eine Collection schreibst (sie ist Datenkonsument), verwende <? super T>.

Beispiele:

// Producer: List<? extends Fruit>
public void printFruits(List<? extends Fruit> fruits) { ... }

// Consumer: List<? super Apple>
public void addApple(List<? super Apple> apples) { apples.add(new Apple()); }

Achtung vor Übertreibung!

  • Zu viele Wildcards machen Code schwer verständlich und können zu Missverständnissen führen.
  • Für den „normalen“ Gebrauch reichen oft einfache Typ-Parameter.

4. Zusammenfassung – Best Practices im Überblick

  • Nutze Generics da, wo sie wirklich gebraucht werden (Flexibilität, Typsicherheit).
  • Halte Namen der Typ-Parameter kurz und nach Konvention.
  • Kommentiere den Zweck von Typ-Parametern, wenn er nicht offensichtlich ist.
  • Verwende Wildcards bewusst – und nach der PECS-Regel.
  • Vermeide tiefe Verschachtelung bei generischen Typen – lieber strukturieren oder aufteilen.

Tipp: Am Anfang lieber weniger Typ-Parameter/Wildcards und dafür den Code nachvollziehbar halten. Mit wachsender Erfahrung kannst du die Mächtigkeit von Generics gezielt ausreizen!
