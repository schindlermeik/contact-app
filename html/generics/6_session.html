Handout: Typsichere Programmierung mit Generics

1. Definition und Bedeutung der Typsicherheit

Typsicherheit bedeutet, dass im Code garantiert wird, dass Variablen, Objekte, Methoden etc. nur mit den erlaubten Datentypen arbeiten. Das Ziel: Fehler vermeiden, die entstehen, wenn ein Wert einen anderen Typ hat als erwartet.

Beispiele für typsicheren Code:

int zahl = 5;             // Nur ganze Zahlen erlaubt
String text = "Hallo";    // Nur Text erlaubt

Hier ist ausgeschlossen, dass z. B. zahl = "abc"; funktioniert – der Compiler verhindert das.

Warum ist Typsicherheit wichtig?

  • Fehlervermeidung: Viele Programmierfehler entstehen durch falsche Typen.
  • Klarer und wartbarer Code: Andere (und du selbst später) wissen sofort, welche Typen erwartet werden.
  • Früherkennung: Fehler werden früh entdeckt, oft schon beim Schreiben, spätestens beim Kompilieren.

2. Wie Generics Typsicherheit fördern

Vor der Einführung von Generics (Java 5) waren viele Methoden und Collections auf den Typ Object ausgelegt. Dadurch konnten Fehler erst zur Laufzeit auffallen, weil alles zu allem konvertiert werden konnte.

Beispiel – ohne Generics:

List namen = new ArrayList(); // "raw type"
namen.add("Anna");
namen.add(123); // Erlaubt – aber unsinnig!

for (Object o : namen) {
    String name = (String) o; // ClassCastException bei 123!
}

Fehler wie der Versuch, eine Zahl als String zu behandeln, zeigen sich erst zur Laufzeit – und das oft an ganz anderer Stelle im Programm.

Mit Generics:

List<String> namen = new ArrayList<>();
namen.add("Anna");
// namen.add(123); // Compilerfehler: Typsicherheit!

for (String name : namen) {
    System.out.println(name.toUpperCase());
}

Hier erkennt der Compiler sofort, wenn du einen falschen Typ hinzufügst. Typsicherheit ist gewährleistet.

3. Vergleich: Generics vs. „raw types“

Raw Types Generics
Deklaration List liste = new ArrayList(); List<String> liste = new ArrayList<>();
Typprüfung Nur auf Object-Basis Auf konkreten Typ
Casts nötig? Ja – beim Zugriff Nein
Fehlerart Laufzeitfehler Compilerfehler
Lesbarkeit Schlechter, unklarer Typ Klarer, besser lesbar

Code-Beispiel:

// Raw type
List daten = new ArrayList();
daten.add("Wert");
daten.add(42);
String s = (String) daten.get(1); // Laufzeitfehler

// Mit Generics
List<String> daten2 = new ArrayList<>();
daten2.add("Wert");
// daten2.add(42); // Compilerfehler: Nur Strings erlaubt!

4. Kompilierzeit-Prüfung statt Laufzeit-Fehler

Generics ermöglichen, dass viele Fehler früh erkannt werden, nämlich beim Kompilieren – bevor das Programm überhaupt startet.

Beispiel:

List<Integer> zahlen = new ArrayList<>();
zahlen.add(1);
zahlen.add(2);
// zahlen.add("drei"); // Compilerfehler

for (Integer zahl : zahlen) {
    System.out.println(zahl * 2);
}
  • Ohne Generics: "drei" hätte eingefügt werden können – und hätte erst beim Zugriff einen Fehler verursacht.
  • Mit Generics: Der Fehler wird sofort angezeigt, und das Programm startet gar nicht erst mit dem Fehler.

Vorteile der Kompilierzeit-Prüfung:

  • Früher Fehler erkennen: Keine Überraschungen im laufenden Programm.
  • Bessere Wartbarkeit: Andere Entwickler sehen sofort, welche Typen zulässig sind.
  • Keine unnötigen Casts: Weniger Schreibarbeit und weniger Fehlerquellen.

5. Übungen zum Selbststudium

  1. Schreibe eine Liste ohne Generics, füge verschiedene Typen ein, und versuche beim Zugriff einen Cast – was passiert?
  2. Schreibe denselben Code mit Generics – welche Fehler treten auf, und wann?
  3. Überlege: Wo in deinem bisherigen Java-Code haben Generics dich schon einmal vor Fehlern bewahrt?

6. Merksätze

  • Typsicherheit verhindert viele typische Programmierfehler schon beim Kompilieren.
  • Generics sind ein Werkzeug, das Typsicherheit für Collections und andere Klassen bereitstellt.
  • Generics = Kompilierzeit-Prüfung statt Laufzeit-Überraschung.
  • Benutze keine „raw types“ mehr – sie sind unsicher und veraltet!
