Handout: Praxis – Hands-On und Übungen zu Dependency Injection (DI)

6.1 Refactoring einer „hard-coded“ Implementierung zu DI

Ziel:

Lerne, bestehende festverdrahtete („hard-coded“) Abhängigkeitsbeziehungen in deinem Code auf Dependency Injection umzustellen.

Ausgangssituation (ohne DI):

public class OrderService {
    private final OrderRepository repository = new OrderRepository();

    public void createOrder(Order order) {
        repository.save(order);
    }
}
  • Problem: OrderService hängt fest von einer konkreten OrderRepository-Implementierung ab.
  • Konsequenz: Schwer testbar, schlecht erweiterbar (z. B. für Mocking oder alternative Repositories).

Refactoring mit DI (Konstruktor-Injektion):

public class OrderService {
    private final OrderRepository repository;

    // Abhängigkeit wird injiziert!
    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }

    public void createOrder(Order order) {
        repository.save(order);
    }
}

Verwendung:

OrderRepository repo = new DatabaseOrderRepository();
OrderService service = new OrderService(repo);

Teste selbst:

  • Führe ein Refactoring in deinem eigenen Code durch: Suche eine Klasse mit fest eingebauter Abhängigkeit und stelle auf Konstruktor-Injektion um.

6.2 Testen von Komponenten mit Dependency Injection

Ziel:

Verstehe, wie DI das Testen durch das Einsetzen von Mock-Objekten ermöglicht.

Beispiel:

public interface PaymentGateway {
    void pay(double amount);
}

public class OrderService {
    private final PaymentGateway paymentGateway;

    public OrderService(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }

    public void processPayment(double amount) {
        paymentGateway.pay(amount);
    }
}

Mock-Implementierung für den Test:

public class MockPaymentGateway implements PaymentGateway {
    public boolean called = false;
    public double lastAmount = 0;

    public void pay(double amount) {
        called = true;
        lastAmount = amount;
    }
}

Unit Test (z. B. mit JUnit):

@Test
public void testProcessPayment() {
    MockPaymentGateway mockGateway = new MockPaymentGateway();
    OrderService service = new OrderService(mockGateway);

    service.processPayment(42.0);

    assertTrue(mockGateway.called);
    assertEquals(42.0, mockGateway.lastAmount, 0.01);
}

Übung:

  • Schreibe selbst eine Komponente, injiziere ein Mock-Objekt, und teste, ob eine Methode korrekt aufgerufen wurde.

6.3 Diskussion: Typische Stolperfallen und Lösungen

Problem/Stolperfalle Lösung/Empfehlung
Zu viele Abhängigkeiten in einer Klasse SRP beachten, Klasse aufteilen, Verantwortung prüfen
Mischung von hard-coded und DI-Abhängigkeiten Immer einheitlich vorgehen, keine „versteckten“ new-Aufrufe
Vergessene/fehlende Injektion Unit Tests schreiben, Code Reviews, Konstruktor-Pflichtfelder
Zirkuläre Abhängigkeiten (A→B, B→A) Redesign, Einführung von Schnittstellen oder Factories
Unübersichtlicher „Zusammensteck-Code“ „Composition Root“ zentral halten, ggf. Factory nutzen

  • Wo finden sich in deinem Code solche Stolperfallen?
  • Wie kannst du durch DI und Refactoring die Architektur verbessern?

6.4 Kurzer Ausblick auf Dependency Injection Frameworks (Spring, Guice, Quarkus)

Warum und wann Frameworks sinnvoll werden

Manuelle DI ist ausreichend…

  • …bei kleinen bis mittleren Projekten.
  • …wenn die Anzahl der Komponenten überschaubar ist.

Aber:

  • Mit wachsendem Projekt werden das Management und die Konfiguration von Abhängigkeiten schnell unübersichtlich.
  • Lebenszyklus, Konfiguration (z. B. für verschiedene Umgebungen), automatische Erzeugung, Ressourcenverwaltung etc. sind aufwendig.

Vorteile von DI-Frameworks

  • Automatische Verdrahtung: Framework übernimmt die Erzeugung und das Verbinden der Komponenten.
  • Lebenszyklus-Verwaltung: Singleton, Prototype, etc.
  • Konfigurierbarkeit: Unterschiedliche Implementierungen für verschiedene Umgebungen (z. B. Test, Produktion).
  • Weitere Features: Scopes, Events, Aspektorientierung, etc.

Kurzüberblick bekannter Frameworks

  • Spring (Java):

    • Marktführer, sehr umfangreich, viele Zusatzfeatures (Web, Data, Security, …)
    • @Component, @Autowired, XML- und Java-Konfiguration möglich.

  • Guice (Google, Java):

    • Leichtgewichtiger, nur Java-Konfiguration, Annotation-basiert.

  • Quarkus (Java, für Cloud/Container):

    • Moderne, schnelle Microservices-Entwicklung, setzt auf CDI (Contexts and Dependency Injection, JSR 330).

Beispiel (Spring Boot):

@Service
public class OrderService {
    private final OrderRepository repository;

    @Autowired
    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }
}
  • Spring „verdrahtet“ automatisch das passende Repository-Bean.

Praxis-Aufgaben für das Selbststudium

  1. Refactore eine Klasse: Wähle eine fest verdrahtete Klasse und stelle sie auf Dependency Injection um. Erzeuge die Objekte im Main-Programm und injiziere sie.

  2. Schreibe einen Test mit Mock: Erstelle eine Mock-Implementierung eines Interfaces und teste, ob die zu testende Methode korrekt darauf zugreift.

  3. Reflektiere Stolperfallen: Analysiere deinen Code auf die oben genannten Stolperfallen. Wie könntest du sie beheben?

  4. Recherchiere Frameworks: Informiere dich über mindestens eines der genannten Frameworks und notiere Vor- und Nachteile für deine Projekte.

Zusammenfassung

  • Refactoring auf DI und der Einsatz von Mocks verbessern Flexibilität, Testbarkeit und Wartbarkeit.
  • Typische Probleme können durch klare Prinzipien und Clean Code gelöst werden.
  • Frameworks werden bei wachsender Projektgröße unverzichtbar und bieten viele Zusatzfunktionen.
