Handout: Einführung in Dependency Injection (DI)

1.1 Was ist Dependency Injection?

Dependency Injection (DI) ist ein Entwurfsmuster aus der Softwareentwicklung, das dazu dient, die Abhängigkeiten (Dependencies) eines Objekts nicht selbst innerhalb des Objekts zu erzeugen, sondern von außen bereitzustellen (zu „injizieren“).

Definition: DI bedeutet, dass eine Komponente die Objekte, die sie benötigt, nicht selbst erstellt, sondern diese von außen – meist durch den Konstruktor, eine Methode oder ein Interface – übergeben bekommt.

Beispiel:

Ohne Dependency Injection:

class Service {
    private Repository repository = new Repository();
    // ...
}

Mit Dependency Injection:

class Service {
    private Repository repository;

    public Service(Repository repository) {
        this.repository = repository;
    }
    // ...
}

Vorteile:

  • Die Klasse Service ist nicht mehr dafür verantwortlich, wie das Repository erzeugt wird.
  • Die Abhängigkeit (Repository) kann ausgetauscht, gemockt oder konfiguriert werden.

1.2 Begriffserklärung: Abhängigkeiten und ihre Rolle in Software-Projekten

Was sind Abhängigkeiten (Dependencies)?

Eine Abhängigkeit ist jedes Objekt oder jede Komponente, auf das eine Klasse oder ein Modul angewiesen ist, um korrekt zu funktionieren.

Beispiel: Eine Klasse UserController benötigt ein UserRepository, um Benutzerdaten aus einer Datenbank zu laden. UserRepository ist in diesem Fall eine Abhängigkeit von UserController.

Rolle von Abhängigkeiten in Projekten

  • Kopplung: Viele direkte Abhängigkeiten führen zu einer engen Kopplung („Tight Coupling“).
  • Austauschbarkeit: Starke Abhängigkeiten erschweren das Ersetzen einzelner Komponenten.
  • Testbarkeit: Eng gekoppelte Komponenten sind schwerer zu testen, da echte Abhängigkeiten (z.B. Datenbank, Services) verwendet werden müssen.

Merksatz: Jede Komponente, die ein anderes Objekt benutzt, ist von diesem abhängig – je loser die Kopplung, desto flexibler das System.

1.3 Motivation: Warum sollte man DI verwenden?

Die wichtigsten Gründe für Dependency Injection:

  1. Reduzierung der Kopplung:

    • Komponenten wissen nicht, wie ihre Abhängigkeiten erzeugt werden, sondern bekommen sie bereitgestellt.
    • Das macht das System flexibler und weniger anfällig für Änderungen.

  2. Verbesserte Testbarkeit:

    • Durch DI kann man leicht sogenannte Mocks oder Stubs einsetzen, um z.B. eine Datenbank zu simulieren.
    • Tests können isoliert geschrieben werden (Unit-Tests).

  3. Förderung von Wiederverwendbarkeit:

    • Komponenten, die ihre Abhängigkeiten von außen bekommen, sind einfacher in anderen Kontexten wiederverwendbar.

  4. Einfacherer Austausch von Implementierungen:

    • Man kann verschiedene Implementierungen (z.B. für Repository) verwenden, ohne die abhängige Klasse zu ändern.

  5. Bessere Wartbarkeit:

    • Änderungen an Abhängigkeiten betreffen nur die Stelle, an der sie eingebunden werden, nicht aber jede Komponente einzeln.

Praxisbeispiel:

  • Wenn ein PaymentService plötzlich mit PayPal statt mit Kreditkarte arbeiten soll, genügt es, eine neue Implementierung zu „injizieren“.

1.4 Vergleich: Traditionelle Abhängigkeitsverwaltung vs. Dependency Injection

Traditionelle Abhängigkeitsverwaltung

  • Komponenten erzeugen oder „wissen“ selbst, wie ihre Abhängigkeiten erzeugt werden.
  • Änderungen an Abhängigkeiten führen oft zu Änderungen an vielen Stellen im Code.
  • Schwer zu testen (da z.B. konkrete Datenbankverbindungen genutzt werden).

Beispiel:

class OrderService {
    private OrderRepository repository = new OrderRepository();
    // ...
}
  • Hier ist die Implementierung des Repositories fest verdrahtet.

Dependency Injection

  • Abhängigkeiten werden von außen gesetzt (Konstruktor, Setter, Interface).
  • Komponenten benötigen nur das Wissen über das „Was“, nicht über das „Wie“ einer Abhängigkeit.
  • Erleichtert das Austauschen von Abhängigkeiten, z.B. im Test.

Beispiel:

class OrderService {
    private OrderRepository repository;

    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }
    // ...
}
  • Hier kann OrderRepository im Test z.B. durch ein MockOrderRepository ersetzt werden.

Vergleich in Stichpunkten

Kriterium Traditionell Dependency Injection
Kopplung Hoch Gering
Testbarkeit Gering Hoch
Austauschbarkeit Schwierig Einfach
Wartbarkeit Schlechter Besser
Wiederverwendbarkeit Eingeschränkt Hoch

Zusammenfassung

  • Dependency Injection trennt die Erstellung und Nutzung von Abhängigkeiten.
  • Dadurch werden Kopplung, Wartbarkeit und Testbarkeit deutlich verbessert.
  • DI kann auch ohne Framework einfach im Code umgesetzt werden.

Tipp: Probiere aus, eine eigene kleine Klasse zu schreiben, die eine Abhängigkeit von außen über den Konstruktor erhält – z.B. ein PrinterService, der ein OutputDevice injected bekommt.
