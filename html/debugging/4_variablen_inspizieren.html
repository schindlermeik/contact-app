Handout: Variablen und Speicher inspizieren im IntelliJ Debugger

1. Ansehen und Überwachen von Variablenwerten

  • Während des Debuggens hält dein Programm an einem Breakpoint oder bei einem Schritt an.

  • Im Debugger-Fenster (meist unten oder seitlich) siehst du sofort:

    • Alle lokalen Variablen der aktuellen Methode
    • Parameter und Rückgabewerte

  • Wie:

    • Werte werden in einer Baumstruktur dargestellt (Name, Typ, aktueller Wert).
    • Klick auf das kleine Dreieck vor einer Variable zeigt bei Objekten deren Felder/Unterobjekte.

Praxisbeispiel: Nach int result = add(x, y); → Im Debugger-Fenster stehen jetzt: x = 10, y = 5, result = 15

Tipp: Hebe im Code eine Variable hervor – sie wird auch im Debugger sichtbar hervorgehoben.

2. Werte „on the fly“ verändern

  • Oft hilfreich, um gezielt zu testen, wie sich ein anderer Wert auf den weiteren Ablauf auswirkt – ohne den Code zu ändern.

  • So geht’s:

    1. Im Debugger-Fenster auf die Variable rechtsklicken → „Set Value…“
    2. Neuen Wert eingeben (z. B. result = 100)
    3. Das Programm läuft ab jetzt mit dem neuen Wert weiter.

Anwendungsfälle:

  • Fehler nachstellen, die im Normalfall schwer zu erreichen sind.
  • Effekte von Variablenänderungen sofort beobachten.

3. Watches und Expressions hinzufügen

a) Watches (Überwachte Ausdrücke)

  • Mit Watches kannst du beliebige Variablen oder Ausdrücke dauerhaft im Blick behalten – selbst wenn sie nicht mehr im aktuellen Scope sind.

  • So geht’s:

    1. Im Debugger-Fenster auf den Reiter „Watches“ klicken (oder Rechtsklick → „Add to Watches“).
    2. Ausdruck eintragen (z. B. x + y, myObject.getStatus()).
    3. Watch bleibt erhalten, bis du sie entfernst.

  • Sehr nützlich bei komplexen Ausdrücken oder über mehrere Methodenaufrufe hinweg!

b) Evaluate Expression

  • Siehe voriges Handout: Drücke Alt + F8 oder wähle im Debug-Fenster „Evaluate Expression“.
  • Gibt dir das Ergebnis einmalig zurück, ohne Watch zu setzen.

4. Anzeigen von Call Stack und Thread-Übersicht

a) Call Stack

  • Der Call Stack zeigt, wie das Programm zur aktuellen Stelle gelangt ist – also die „Kette“ aller Methodenaufrufe bis hierher.

  • Im Debug-Fenster (Reiter „Frames“ oder „Call Stack“):

    • Jede Zeile zeigt eine Methode und ihre Position im Code.
    • Klick auf einen Eintrag bringt dich direkt an die jeweilige Stelle im Code.

  • Wofür?

    • Rückverfolgung, wer eine Methode aufgerufen hat.
    • Ursachen für Fehler/Exceptions leichter finden.

b) Thread-Übersicht

  • Programme können mit mehreren Threads (Nebenläufigkeit) laufen.
  • Im Debugger werden alle laufenden Threads angezeigt.
  • Du kannst zwischen Threads wechseln, Breakpoints thread-spezifisch setzen oder analysieren, ob Threads „hängen“ (z. B. bei Deadlocks).

5. Objekt-Inspektion (Struktur, Felder, Werte)

  • Bei komplexen Objekten kannst du die Struktur aufklappen und alle Felder, Unterobjekte und deren Werte sehen.

  • Unterstützt auch verschachtelte Strukturen und Sammlungen (Listen, Maps, Sets).

  • Klick auf das Dreieck vor dem Objektnamen → Unterelemente werden angezeigt.

  • Mit Rechtsklick auf ein Objekt:

    • „View as…“ (z. B. Hexadezimal, Dezimal, String etc.)
    • „Copy Value“ oder „Set Value“

Praxis: Beispiel: Ein Breakpoint in einer Methode, die ein User-Objekt verarbeitet → Im Debugger kannst du user.getName(), user.getId(), und alle weiteren Felder einsehen.

6. Praxis-Tipps

  • Nutze Watches für wiederkehrende, komplexe Ausdrücke.
  • Verändere Werte „on the fly“, um Bedingungen gezielt zu testen.
  • Überwache Call Stack und Threads besonders bei komplexen Fehlern, Exceptions oder Multithreading.
  • Verwende die Objekt-Inspektion, um Datenmodelle oder verdächtige Werte zu analysieren.
  • Notiere dir während des Debuggens auffällige Werte und Abläufe, um den Fehler schneller einzugrenzen.
