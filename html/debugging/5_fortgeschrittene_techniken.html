Handout: Fortgeschrittene Debugging-Techniken in IntelliJ

1. Conditional Breakpoints (mit Bedingungen und Hit Counts)

a) Was sind Conditional Breakpoints?

  • Conditional Breakpoints stoppen den Programmlauf nur, wenn eine bestimmte Bedingung erfüllt ist – z. B. eine Variable hat einen bestimmten Wert.
  • Sehr hilfreich bei Schleifen oder bei Fehlern, die nur bei speziellen Daten auftreten.

b) Wie setzt man sie?

  1. Breakpoint normal setzen (Klick auf den Rand).
  2. Rechtsklick auf den roten Punkt → „More…“ oder „Breakpoint Properties“.
  3. Im Feld „Condition“ einen Java-Ausdruck eingeben, z. B. i == 42 || user != null && user.getName().equals("Alice").
  4. Nur wenn die Bedingung „true“ ist, hält das Programm an.

c) Hit Count nutzen

  • Du kannst einstellen, dass das Programm erst nach x Treffern anhält.
  • Z. B. „Stopp erst beim 100. Durchlauf einer Schleife“.
  • Feld „Hit Count“ setzen (z. B. 100).

2. Log Breakpoints (ohne Anhalten, nur Protokollieren)

  • Statt den Ablauf zu stoppen, kann ein Breakpoint nur eine Nachricht ins Debug-Fenster loggen – das Programm läuft weiter.
  • Besonders nützlich, wenn du wissen willst, wie oft und mit welchen Werten eine bestimmte Zeile ausgeführt wird, ohne das Programm zu unterbrechen.

So geht’s:

  1. Breakpoint normal setzen, dann rechtsklicken.
  2. Haken bei „Log message to console“ setzen.
  3. Optional: Variablen im Log ausgeben, z. B. "i = " + i
  4. Das Programm schreibt bei jedem Erreichen die Nachricht ins Debug-Fenster, läuft aber normal weiter.

3. Exception Breakpoints gezielt einsetzen

  • Exception Breakpoints stoppen das Programm genau dann, wenn eine bestimmte Exception auftritt – unabhängig davon, wo im Code das passiert.
  • Besonders praktisch bei schwer auffindbaren Fehlern.

So geht’s:

  1. Öffne das Breakpoints-Fenster (Ctrl + Shift + F8).
  2. Klicke auf das Plus-Symbol und wähle „Java Exception Breakpoint“.
  3. Suche z. B. nach NullPointerException oder einer eigenen Exception.
  4. Das Programm hält immer dann an, wenn diese Exception ausgelöst wird – du siehst sofort den fehlerhaften Zustand.

4. Multithreaded Debugging

a) Thread-Wechsel im Debugger

  • Programme mit mehreren Threads (z. B. in Webanwendungen oder parallelen Prozessen) können gleichzeitig an verschiedenen Stellen laufen.
  • Im Debugger siehst du unter „Threads“ alle laufenden Threads.
  • Wechsle zwischen Threads, um den Zustand und den Call Stack je Thread zu untersuchen.

b) Deadlocks und Race Conditions analysieren

  • Deadlock: Zwei oder mehr Threads warten gegenseitig auf Ressourcen – das Programm „hängt“.
  • Race Condition: Ergebnis hängt davon ab, wie schnell/bestimmte Threads laufen – führt zu schwer vorhersagbaren Fehlern.

Debugging-Tipps:

  • Beobachte, ob Threads im „WAITING“- oder „BLOCKED“-Status sind (erscheint im Thread-Fenster).
  • Analysiere, an welcher Stelle sich Threads gegenseitig blockieren.
  • Lasse dir im Debugger die Thread States anzeigen und prüfe Synchronisationsobjekte.

c) Thread States und Synchronisation überwachen

  • Thread State:

    • „RUNNABLE“ (läuft), „WAITING“, „BLOCKED“, „TERMINATED“

  • Im Debug-Fenster werden alle States live angezeigt.

  • Überprüfe bei Synchronisation (z. B. mit synchronized, Locks), wo Threads aufeinander warten.

5. Debugging von Lambdas und Streams

  • Lambdas/Streams machen das Debuggen schwieriger, weil sie oft anonymer und weniger sichtbar sind.

  • IntelliJ kann dennoch:

    • Breakpoints innerhalb von Lambdas/Streams setzen (z. B. in .forEach(x -> …)).
    • Variablen und Zwischenwerte anzeigen – probiere das im Debuggerfenster bei laufender Session.

  • Tipp: Nutze Conditional oder Log Breakpoints, um zu sehen, wie oft die Lambda wirklich aufgerufen wird und mit welchen Werten.

Beispiel:

List<String> list = Arrays.asList("A", "B", "C");
list.stream().map(s -> s.toLowerCase()) // Setze Breakpoint auf die Lambda-Zeile!
    .forEach(System.out::println);

6. HotSwap: Code während der Debugging-Session verändern

  • HotSwap ermöglicht, kleine Code-Änderungen vorzunehmen, während das Debugging läuft – ohne Neustart.
  • Funktioniert für einfache Änderungen (z. B. Methoden-Body ändern, aber keine Methoden hinzufügen/entfernen).
  • Sehr nützlich, um direkt eine kleine Korrektur zu testen, ohne den kompletten Prozess neu zu starten.

So geht’s:

  1. Debugging läuft.
  2. Ändere den Code (z. B. return-Wert oder Print-Ausgabe).
  3. IntelliJ erkennt die Änderung und bietet an, den Code zu „Reload“/HotSwap (oft per Popup oder über das Blitz-Symbol in der Debug-Leiste).
  4. Die Änderung wird sofort übernommen, der Debugger bleibt aktiv.

7. Tipps & Zusammenfassung

  • Conditional und Log Breakpoints helfen, bei großen Datenmengen und Schleifen gezielt zu debuggen.
  • Exception Breakpoints sind ideal für die Fehlersuche bei Ausnahmen.
  • Im Multithread-Debugging immer den Thread-State und den Call Stack beachten.
  • Lambdas und Streams lassen sich mit modernen IDEs ebenfalls effektiv debuggen.
  • Mit HotSwap sparst du Zeit bei kleinen Änderungen – nutze es, wo möglich!
