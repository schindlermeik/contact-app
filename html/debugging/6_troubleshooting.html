Handout: Praxisfälle und Troubleshooting beim Debugging in IntelliJ

1. Typische Fehlerquellen beim Debuggen

a) Fehler beim Setzen von Breakpoints

  • Breakpoint an der falschen Stelle (z. B. auf einer Leerzeile oder Kommentar).

  • Breakpoint im nicht erreichten Code (z. B. nie aufgerufene Methode, if-Zweig, der nie „true“ ist).

  • Tipp:

    • Achte auf die farbige Markierung im Editor (roter Punkt = aktiv).
    • Vergewissere dich, dass der Codeweg wirklich durch die Breakpoint-Zeile führt.

b) Variablen nicht sichtbar oder „optimized away“

  • Variablen erscheinen als „not available“, besonders bei Release-Builds mit Optimierung.

  • Tipp:

    • Für das Debuggen immer im Debug- oder Development-Modus kompilieren.
    • Prüfe Compiler-Settings und baue ggf. neu.

c) Falsche Debug-Konfiguration

  • Falscher Startpunkt (andere Methode oder Konfiguration gewählt).

  • Umgebungsvariablen, Argumente oder Profile stimmen nicht.

  • Tipp:

    • Überprüfe die „Run/Debug Configurations“ in IntelliJ.
    • Teste, ob der Code im normalen „Run“-Modus wie erwartet startet.

d) Nebenläufigkeit und Zeitpunkte

  • Fehler treten nur bei bestimmten Timing-Konstellationen oder mehreren Threads auf.

  • Breakpoints können das Timing verändern („Heisenbug“).

  • Tipp:

    • Mit „Log Breakpoints“ und Thread-Übersicht arbeiten.
    • Nutze zusätzliche Logs, um Timing-Probleme besser nachvollziehen zu können.

2. Debugging von Unit-Tests und Integrationstests

a) Debugging von Unit-Tests

  • Direkt in IntelliJ:

    • Rechtsklick auf einen Test oder eine Testklasse → „Debug Test“.
    • Setze Breakpoints wie gewohnt im Produktiv- und Testcode.

  • Während der Test läuft, hält der Debugger an jedem Breakpoint.

  • Du kannst Variablen in Test- und Produktivcode gleichzeitig untersuchen.

Tipp: Setze Conditional Breakpoints, um gezielt bei bestimmten Testdaten oder Fehlerfällen zu stoppen.

b) Debugging von Integrationstests

  • Meist komplexer, da mehrere Komponenten zusammenspielen (z. B. Datenbank, Services).

  • Auch hier: „Debug Test“ verwenden und Breakpoints an passenden Schnittstellen setzen.

  • Achte auf parallele Ausführung: Viele Testframeworks führen Tests parallel aus. Prüfe, ob der richtige Test anhält!

  • Tipp:

    • Ggf. parallele Ausführung deaktivieren (Test-Framework-Settings).

3. Debugging von Remote-Anwendungen (Remote Debugging konfigurieren)

a) Was ist Remote Debugging?

  • Debugging eines Programms, das auf einem anderen Rechner oder in einer anderen Umgebung läuft (z. B. Server, Container, Cloud).
  • Sehr hilfreich bei Fehlern, die nur „im echten Betrieb“ auftreten.

b) Remote Debugging in Java/IntelliJ

Vorgehen:

  1. Starte das entfernte Programm mit Debug-Optionen (Java: -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005)

  2. In IntelliJ:

    • Menü: Run → Edit Configurations… → + → „Remote JVM Debug“ auswählen
    • Trage die Adresse und Port (z. B. localhost:5005) ein

  3. Starte die Remote Debugging-Session.

  4. Setze Breakpoints wie gewohnt – der Debugger stoppt am Code der entfernten Anwendung.

Tipp: Nutze Remote Debugging sparsam und nur in sicheren Netzwerken – offene Debug-Ports sind ein Sicherheitsrisiko!

4. Debugging von Webanwendungen und Microservices

a) Debugging von Webanwendungen

  • Starte die Anwendung im Debug-Modus (oft im Spring Boot/Maven/Gradle-Run-Config).
  • Setze Breakpoints im Controller, Service oder Repository.
  • Simuliere HTTP-Requests (z. B. mit Postman oder Browser) und beobachte die Ausführung.

Beispiel: Breakpoint im REST-Controller → Sende eine Anfrage → Debugger hält bei jedem Request.

b) Debugging von Microservices

  • Microservices laufen oft in mehreren Prozessen/Containern.

  • Für Debugging:

    • Einzelnen Service im Debug-Modus starten (wie Remote Debugging).
    • Ggf. mehrere Debug-Sessions parallel in IntelliJ öffnen (für verschiedene Ports).

  • Kommunikation und Fehler zwischen Services analysieren:

    • Breakpoints an Schnittstellen
    • Logs kombinieren
    • Überwache Events und Messaging-Protokolle.

Tipp: Nutze Distributed-Tracing-Tools (z. B. Jaeger, Zipkin) für einen besseren Überblick.

5. Umgang mit „unsichtbaren“ Ausnahmen (z. B. in Third-Party-Bibliotheken)

  • „Unsichtbare“ Exceptions = Fehler, die im Code von Drittanbietern, Frameworks oder Bibliotheken entstehen und nicht direkt im eigenen Code sichtbar sind.

a) Exception Breakpoints nutzen

  • Setze Exception Breakpoints auf die gesuchte Exception-Klasse (z. B. NullPointerException).
  • Das Programm hält an der ersten Auslösestelle, egal wo der Fehler auftritt.

b) Call Stack analysieren

  • Wenn das Programm an einer Exception hält:

    • Sieh dir den gesamten Call Stack an.
    • Suche den letzten Aufruf aus deinem eigenen Code – meist liegt dort die Ursache.

c) Quellcode von Bibliotheken einbinden

  • Viele Bibliotheken bieten „Quellanhänge“ (Source Attachments).
  • Binde sie in IntelliJ ein, um auch fremden Code im Debugger zu sehen.
  • Über „External Libraries“ → Rechtsklick → „Download Sources“ (falls verfügbar).

d) Logging ergänzen

  • Ergänze eigenes Logging, um festzustellen, welche Eingaben und Zustände zum Fehler geführt haben.

6. Tipps & Best Practices

  • Debugge mit einfachen Szenarien und steigere die Komplexität schrittweise.
  • Kombiniere Debugger, Logging und (wenn möglich) automatisierte Tests, um Fehlerursachen effizient einzugrenzen.
  • Bei schwer reproduzierbaren Fehlern: Nutze Conditional und Exception Breakpoints!
  • Dokumentiere Erkenntnisse und typische Fehlerbilder für dich und das Team.
  • Nutze die IntelliJ-Hilfe! Viele Tipps sind direkt in der IDE hinterlegt (Shift + Shift → „Debugging“ suchen).
