Handout: Advanced Mocking-Techniken mit Mockito

1. Mocking von void-Methoden

  • Methoden ohne Rückgabewert („void“) können in Mockito trotzdem beobachtet und gesteuert werden.
  • Nützlich, wenn die Methode z.B. etwas speichert, loggt oder weiterleitet.

a) Verifikation

mockService.doSomething();
verify(mockService).doSomething(); // Prüft, ob Methode aufgerufen wurde

b) Verhalten ändern mit doNothing(), doThrow(), doAnswer()

  • doNothing() ist Standard, aber man kann auch Exceptions werfen oder eigenes Verhalten definieren:
doNothing().when(mockService).logEvent();
mockService.logEvent(); // Nichts passiert

doThrow(new RuntimeException("Fehler")).when(mockService).deleteFile();
assertThrows(RuntimeException.class, () -> mockService.deleteFile());

2. Mocking von Exceptions

  • Methoden mit Rückgabewert:

    when(mockService.loadData()).thenThrow(new IOException("Fehler beim Laden"));

  • Void-Methoden: Mit doThrow() – siehe oben.

  • Praxis-Tipp: So kann gezielt das Verhalten von Fehlerfällen getestet werden, ohne echte Fehler herbeizuführen.

3. Teil-Mocking (spy)

  • Spies sind „halbe“ Mocks: Sie überwachen echte Objekte, können aber gezielt Methoden überschreiben.
  • Praktisch, wenn ein Teil des Objekts normal arbeitet, andere Methoden aber gemockt werden sollen.

a) Erstellen eines Spies

List<String> realList = new ArrayList<>();
List<String> spyList = spy(realList);

spyList.add("Hallo");
verify(spyList).add("Hallo"); // Aufruf wird überwacht

// Optional: Verhalten überschreiben
doReturn(42).when(spyList).size();
assertEquals(42, spyList.size());

b) Einsatzfälle

  • Echte Logik nutzen, aber bestimmte Nebenwirkungen verhindern oder das Verhalten gezielt beeinflussen.
  • Nützlich für Legacy-Code oder komplexe Klassen.

Achtung: Bei Spies wird das echte Objekt verwendet! Fehler im Konstruktionscode wirken sich also weiterhin aus.

4. Mocking von finalen Klassen und statischen Methoden

  • Standard-Mockito (bis Version 2.x): Konnte nur Interfaces und nicht-finale Klassen mocken.

  • Seit Mockito 3.x (mit mockito-inline oder mockito-extended):

    • Auch finale Klassen und Methoden können gemockt werden.
    • Statische Methoden erfordern noch spezielle Unterstützung.

a) Setup für finales/statisches Mocking

In der Maven pom.xml:

<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-inline</artifactId>
    <version>5.2.0</version>
    <scope>test</scope>
</dependency>

b) Mocking von finalen Klassen

final class FinalClass {
    String getValue() { return "real"; }
}

FinalClass mockFinal = Mockito.mock(FinalClass.class);
when(mockFinal.getValue()).thenReturn("mocked");
assertEquals("mocked", mockFinal.getValue());

c) Mocking von statischen Methoden (ab Mockito 3.4 mit mockito-inline)

  • Statische Methoden werden mit dem neuen MockedStatic-Feature gemockt:
import static org.mockito.Mockito.*;

try (MockedStatic<Math> mathMock = mockStatic(Math.class)) {
    mathMock.when(() -> Math.max(2, 3)).thenReturn(42);
    assertEquals(42, Math.max(2, 3));
}
  • Das Mocking gilt nur im Kontext des Try-Blocks!

d) Grenzen und Hinweise

  • Statisches/finales Mocking sollte sparsam eingesetzt werden.
  • Besser ist es, Code möglichst testbar und lose gekoppelt zu schreiben.
  • Für Legacy-Code kann diese Technik sehr nützlich sein.

5. Zusammenfassung & Praxistipps

  • Void-Methoden lassen sich mit doNothing(), doThrow() und Verifikation kontrollieren.
  • Exceptions können gezielt für Rückgabewerte und void-Methoden gemockt werden.
  • Mit spy kann man echtes Verhalten und gezieltes Mocking kombinieren.
  • Finale Klassen und statische Methoden lassen sich mit zusätzlichem Setup mocken (ab Mockito 3.x).
  • Nutze Advanced Mocking-Techniken gezielt – sie machen viele Tests möglich, die sonst schwierig wären, aber sollten nicht zum Standard werden.
