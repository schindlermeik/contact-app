Handout: Mocking und JUnit kombinieren

1. Annotationen für das Zusammenspiel von Mockito und JUnit 5

a) @ExtendWith(MockitoExtension.class)

  • Mit dieser Annotation wird die Mockito-Integration für JUnit 5 aktiviert.
  • Sie sorgt dafür, dass Mockito-Annotationen wie @Mock und @InjectMocks funktionieren.

Beispiel:

import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.junit.jupiter.MockitoExtension;

@ExtendWith(MockitoExtension.class)
class MyTest {
    // ...
}

b) @Mock

  • Mit @Mock werden automatisch Mock-Objekte erzeugt und initialisiert.
  • Praktisch und übersichtlich, vor allem bei mehreren Mocks.

Beispiel:

@Mock
PaymentService paymentService;

c) @InjectMocks

  • Mit @InjectMocks wird ein echtes Objekt erzeugt, und alle Felder, für die es einen passenden Mock gibt, werden automatisch "injiziert".
  • Damit werden Abhängigkeiten für die zu testende Klasse automatisch gemockt.

Beispiel:

@InjectMocks
OrderService orderService;
  • In diesem Beispiel werden z. B. ein Mock für PaymentService und ein Mock für ShippingService in den echten OrderService injiziert.

2. Setup mit JUnit 5 (Jupiter)

a) Kompletter Aufbau einer Testklasse

import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

import static org.mockito.Mockito.*;
import static org.junit.jupiter.api.Assertions.*;

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    PaymentService paymentService;

    @Mock
    ShippingService shippingService;

    @InjectMocks
    OrderService orderService;

    @Test
    void testOrderSuccess() {
        when(paymentService.pay(any())).thenReturn(true);
        when(shippingService.ship(any())).thenReturn("Shipped!");

        String result = orderService.placeOrder(new Order());

        assertEquals("Shipped!", result);
        verify(paymentService).pay(any());
        verify(shippingService).ship(any());
    }
}

Vorteile:

  • Mocks werden automatisch erzeugt.
  • Die getestete Klasse erhält ihre Abhängigkeiten „fertig gemockt“.
  • Kein aufwändiges Setup im Code nötig.

3. Mocking in Parameterisierten Tests

  • Auch bei parameterisierten Tests lassen sich Mocks problemlos einsetzen.
  • Typisch ist die Kombination mit JUnit 5-Annotationen wie @ParameterizedTest und @ValueSource, @CsvSource etc.

Beispiel: Mocks im parameterisierten Test

import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.ValueSource;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;
import org.junit.jupiter.api.extension.ExtendWith;

import static org.mockito.Mockito.*;

@ExtendWith(MockitoExtension.class)
class EmailValidatorTest {

    @Mock
    EmailService emailService;

    @ParameterizedTest
    @ValueSource(strings = {"user@example.com", "test@test.org"})
    void shouldAcceptValidEmails(String email) {
        when(emailService.isValid(email)).thenReturn(true);

        boolean result = emailService.isValid(email);

        assertTrue(result);
        verify(emailService).isValid(email);
    }
}

Hinweis:

  • Die Mocks werden bei jedem Testdurchlauf frisch erzeugt, also gibt es keine Nebeneffekte zwischen den Testfällen.
  • Auch mit anderen Quellen (@CsvSource, @MethodSource) möglich.

4. Tipps & Best Practices

  • Mocks und @InjectMocks immer zusammen mit @ExtendWith(MockitoExtension.class) verwenden.
  • Bei vielen Mocks: Klar benennen, damit die Testfälle verständlich bleiben.
  • Mocking in parameterisierten Tests: Mocks werden pro Durchlauf sauber behandelt.
  • In komplexeren Fällen: Setup und Verifikation in eigene Hilfsmethoden auslagern, damit der Test übersichtlich bleibt.
