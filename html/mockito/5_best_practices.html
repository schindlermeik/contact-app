Handout: Best Practices und typische Fallstricke beim Mocking

1. Übermäßiges Mocking vermeiden

Was ist „übermäßiges Mocking“?

  • Zu viele Mocks im Test machen die Tests schwer lesbar und pflegbar.
  • Wenn für jede kleine Abhängigkeit ein Mock erstellt wird, ist oft das Design der getesteten Komponente zu komplex oder zu stark gekoppelt.

Warum ist das problematisch?

  • Tests werden kompliziert und fehleranfällig.
  • Änderungen an der Implementierung führen schnell zu fehlschlagenden Tests („fragile Tests“).
  • Man testet am Ende nicht mehr das gewünschte Verhalten, sondern nur das eigene Mocking-Setup.

Wie viel Mocking ist richtig?

  • Mocke nur das, was notwendig ist: Externe Systeme, langsame oder schwer kontrollierbare Abhängigkeiten (z. B. Datenbank, Netzwerk, Zeit).
  • Nicht mocken: Einfache Datenobjekte (POJOs), einfache Utility-Methoden, eigene Logik, die nicht schwer zu kontrollieren ist.
  • Faustregel: Mocks für Außenwelt, echte Instanzen für Eigenlogik.

2. Mocks als Helfer für bessere Isolation

Wofür sind Mocks wirklich gedacht?

  • Mocks sollen Isolation schaffen:

    • Ein Unit-Test prüft EINE Komponente, alle Abhängigkeiten werden durch Mocks ersetzt.
    • So lässt sich das Verhalten gezielt und reproduzierbar testen, unabhängig vom Rest des Systems.

  • Mocks ermöglichen das Testen von Edge Cases, z. B. wenn eine Methode eine bestimmte Exception werfen oder ein Service nicht erreichbar sein soll.

Beispiel: Isolierter Test mit Mock

@Mock
PaymentService paymentService;

@Test
void testOrderFailsOnPaymentError() {
    when(paymentService.pay(any())).thenThrow(new PaymentException());
    assertThrows(PaymentException.class, () -> orderService.placeOrder(order));
}
  • Hier wird nur die externe Zahlung gemockt – das Order-Verhalten bleibt echt.

3. Verständliche und wartbare Testfälle

Gute Eigenschaften verständlicher Tests

  • Klarer, sprechender Testname: Was wird getestet, welches Verhalten?
  • Kompaktes Setup: So wenig Vorbereitung wie möglich, aber so viel wie nötig.
  • Keine unnötigen Mocks oder Testdaten: Nur das mocken, was nötig ist.
  • Testdaten und Erwartungen sichtbar: Werte sind direkt im Test erkennbar, keine versteckten Konstruktionen.
  • Kommentiere komplexe Stellen und fasse im Testnamen das Ziel des Tests zusammen.

Beispiel: Klarer Test mit Mock

@Test
void shouldSendWelcomeMailWhenUserRegisters() {
    when(emailService.sendWelcomeMail("max@mail.de")).thenReturn(true);
    boolean success = userService.register("max@mail.de");
    assertTrue(success);
    verify(emailService).sendWelcomeMail("max@mail.de");
}

4. Typische Fehler beim Mocking

a) Zu viel Interaktionsverifikation

  • Jede einzelne Methode wird mit verify() geprüft, auch wenn es nicht relevant ist.
  • Besser: Prüfe nur kritische Interaktionen.

b) Mocking von Eigenlogik

  • Methoden aus der zu testenden Klasse selbst werden gemockt (z. B. über Spies).
  • Problem: Man testet nicht mehr den echten Code.
  • Besser: Nur externe Abhängigkeiten mocken.

c) Statische Methoden und Konstruktoren unreflektiert mocken

  • Mit modernem Mockito zwar möglich, aber kann auf Designfehler hinweisen (zu enge Kopplung, schwer testbar).
  • Besser: Design anpassen, um statische Methoden/Singletons zu vermeiden.

d) Vergessen, Verhalten von Mocks zu definieren

  • Mock gibt standardmäßig null/0/leere Werte zurück – Tests laufen „grün“, obwohl eigentlich Fehlerfälle nicht geprüft werden.

e) Mocks werden nicht korrekt „resetet“

  • In Setup-Methoden (@BeforeEach) immer neue Mocks erzeugen, nicht von Test zu Test wiederverwenden.

f) Testdaten im Mocking-Setup verstecken

  • Testdaten sind nur im Setup sichtbar, nicht im eigentlichen Test – erschwert das Verständnis.

5. Zusammenfassung (Best Practices auf einen Blick)

  • Mocke nur das, was notwendig ist – meist externe Systeme.
  • Halte Tests verständlich, kompakt und wartbar.
  • Benutze sprechende Testnamen und klare Testdaten.
  • Verifiziere nur die wichtigsten Interaktionen.
  • Vermeide Design-Probleme, die umfangreiches Mocking nötig machen.
  • Wenn ein Test viel Mocking braucht, prüfe das Design der Komponente.

Checkliste: Ist mein Mocking sinnvoll?

  • Mocke ich nur externe Abhängigkeiten?
  • Sind die Tests auch für Kollegen schnell verständlich?
  • Sind alle verwendeten Mocks und Testdaten im Test sichtbar?
  • Vermeide ich statisches/finales Mocking, wenn es nicht zwingend nötig ist?
  • Werden Mocks frisch für jeden Test erzeugt?
