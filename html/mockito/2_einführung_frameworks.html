Handout: Einführung in Mocking-Frameworks

1. Überblick über gängige Frameworks

a) Mockito

  • Mockito ist das am weitesten verbreitete Mocking-Framework für Java.

  • Es ist einfach zu erlernen, integriert sich nahtlos mit JUnit und bietet eine klare, lesbare Syntax.

  • Kernfunktionen:

    • Erzeugen von Mocks für beliebige Interfaces und Klassen
    • Konfigurieren des Verhaltens (when(...).thenReturn(...))
    • Überprüfen von Methodenaufrufen (verify(...))
    • Unterstützt Annotationen (@Mock, @InjectMocks)
    • Erweiterbar mit MockitoExtension für JUnit 5

  • Beispiel:

    MyService mockService = Mockito.mock(MyService.class);
when(mockService.getData()).thenReturn("Test");
verify(mockService).getData();

  • Weitere Infos: Mockito Doku

b) EasyMock

  • EasyMock ist ein weiteres, seit Jahren bewährtes Mocking-Framework.

  • Bietet vor allem eine alternative Syntax (Aufzeichnen/Abspielen-Prinzip: "record–replay").

  • Kann besonders hilfreich sein, wenn bereits bestehende Projekte darauf setzen.

  • Kernfunktionen:

    • Automatisches Erzeugen von Mocks
    • Definieren von erwarteten Methodenaufrufen vor Ausführung ("expect"–"replay")
    • Überprüfen, ob alle erwarteten Interaktionen stattgefunden haben

  • Beispiel:

    MyService mockService = EasyMock.createMock(MyService.class);
EasyMock.expect(mockService.getData()).andReturn("Test");
EasyMock.replay(mockService);
// ... Code ausführen ...
EasyMock.verify(mockService);

  • Weitere Infos: EasyMock Doku

c) JMockit

  • JMockit ist ein mächtiges Framework, das auch anspruchsvollere Mocking-Szenarien unterstützt.

  • Kann finale, statische Methoden und sogar Konstruktoren mocken (was mit Mockito nicht ohne Weiteres geht).

  • Wird vor allem in Legacy-Projekten mit schwer testbarem Code verwendet.

  • Kernfunktionen:

    • Mocken von finalen/statischen Methoden und Konstruktoren
    • Record/Replay-API und deklarative Mocks
    • Integration mit JUnit und TestNG

  • Beispiel:

    new Expectations(MyService.class) {{
    MyService.staticMethod(); result = "Mocked";
}};

  • Weitere Infos: JMockit Doku

2. Auswahlkriterien für Mocking-Frameworks

Kriterium Mockito EasyMock JMockit
Einfach zu erlernen Ja Ja Mittel
Gute Doku/Community Sehr gut Gut Mittel
Integration mit JUnit 5 Ja, nahtlos Ja Ja
Unterstützung statischer/finaler Methoden Eingeschränkt * Eingeschränkt * Ja
Für Legacy-Code geeignet Teilweise Teilweise Ja (sehr)
Erweiterbarkeit Hoch Mittel Hoch
Zukunftssicherheit Sehr gut Gut Eher gering

* Für das Mocken von finalen/statischen Methoden mit Mockito wird z.B. das Zusatzmodul mockito-inline benötigt.

Fazit:

  • Mockito ist für die meisten modernen Projekte die beste Wahl (einfach, flexibel, weit verbreitet).
  • EasyMock kann sinnvoll sein, wenn es bereits im Projekt verwendet wird oder der „record–replay“-Stil bevorzugt wird.
  • JMockit eignet sich vor allem für Legacy-Code oder sehr spezielle Anforderungen (z.B. statische Methoden, Konstruktoren).

3. Integration ins Projekt

a) Maven-Abhängigkeiten

Mockito (JUnit 5):

<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-core</artifactId>
    <version>5.2.0</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-junit-jupiter</artifactId>
    <version>5.2.0</version>
    <scope>test</scope>
</dependency>

EasyMock:

<dependency>
    <groupId>org.easymock</groupId>
    <artifactId>easymock</artifactId>
    <version>5.1.0</version>
    <scope>test</scope>
</dependency>

JMockit:

<dependency>
    <groupId>org.jmockit</groupId>
    <artifactId>jmockit</artifactId>
    <version>1.49</version>
    <scope>test</scope>
</dependency>

b) IDE-Integration

  • Alle drei Frameworks werden von modernen IDEs (IntelliJ IDEA, Eclipse) erkannt und unterstützt.
  • JUnit-Tests mit Mocks können per Rechtsklick in der IDE oder mit Build-Tools wie Maven/Gradle ausgeführt werden.
  • Manche IDEs bieten Wizards oder Assistenten zum Erzeugen von Mocks (z.B. in IntelliJ).

c) Best Practices

  • Framework wählen, das zur Team-Erfahrung und zum Code passt.
  • Moderne Projekte: Mockito bevorzugen, JUnit 5 nutzen.
  • Für Legacy-Code: Prüfe, ob JMockit oder EasyMock Vorteile bieten.
  • Tests und Mocks immer sauber voneinander trennen – Setup/Initialisierung in @BeforeEach-Methoden oder per Annotation (@Mock).

4. Zusammenfassung

  • Es gibt verschiedene Mocking-Frameworks – das richtige hängt von den Projektanforderungen ab.
  • Für neue Projekte ist Mockito die Standardempfehlung (einfach, flexibel, große Community).
  • Integration ist mit Maven, Gradle oder IDEs sehr unkompliziert.
  • Gute Doku und regelmäßige Updates sind bei Mockito und EasyMock gewährleistet.
