Handout: Integration von Mocking in den Entwicklungsprozess

1. Mocking im Rahmen von TDD (Test Driven Development)

a) Was ist TDD?

  • Test Driven Development (TDD) ist eine Entwicklungsstrategie, bei der man zuerst einen Test schreibt, der fehlschlägt, dann den Code implementiert, sodass der Test besteht, und schließlich den Code verbessert (refaktoriert).

  • Die TDD-Schritte:

    1. Red: Schreibe einen Test, der fehlschlägt
    2. Green: Schreibe den minimalen Code, damit der Test besteht
    3. Refactor: Optimiere den Code und die Tests

b) Rolle von Mocking in TDD

  • Mocks ermöglichen Isolierung: Mit Mocks können Abhängigkeiten schon im allerersten Test simuliert werden, auch wenn sie noch nicht existieren.

  • Entwicklung gegen Schnittstellen: Entwickler können gegen geplante Schnittstellen testen und entwickeln, bevor die tatsächliche Implementierung da ist.

  • Beispiel: Angenommen, eine Methode ruft einen (noch nicht existierenden) Zahlungsservice auf:

    @Test
void testOrderTriggersPayment() {
    PaymentService paymentService = mock(PaymentService.class);
    OrderService orderService = new OrderService(paymentService);
    orderService.placeOrder(new Order());
    verify(paymentService).pay(any(Order.class));
}

  • Tipp: Durch Mocking bleibt der Fokus auf der zu testenden Logik, nicht auf den Abhängigkeiten.

2. Mock-basierte Tests in CI/CD-Pipelines

a) Warum Mock-basierte Tests automatisieren?

  • Mock-basierte Unit-Tests sind schnell, robust und liefern direktes Feedback.
  • Sie sind perfekt geeignet für die Integration in automatisierte Build- und Deployment-Prozesse (CI/CD).

b) Wie funktioniert das in der Praxis?

  • Jeder Commit oder Pull-Request löst eine Pipeline aus (z. B. mit Jenkins, GitHub Actions, GitLab CI).

  • Die Pipeline führt alle Unit-Tests (inkl. Mock-Tests) automatisch aus:

    • Maven/Gradle-Task (mvn test, gradle test)

  • Fehlgeschlagene Tests blockieren den Merge oder das Deployment.

  • Testberichte (und ggf. Coverage-Berichte) werden automatisch erstellt und dem Team angezeigt.

Beispiel: GitHub Actions YAML für ein Java-Projekt

name: Java CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Set up JDK 21
      uses: actions/setup-java@v4
      with:
        java-version: '21'
    - name: Build and Test with Maven
      run: mvn clean test
  • In solchen automatischen Abläufen werden auch alle mit Mocks arbeitenden Tests ausgeführt und geprüft.

3. Code Coverage mit gemockten Komponenten

a) Was ist Code Coverage?

  • Code Coverage misst, wie viel Prozent des Anwendungscodes beim Testen tatsächlich ausgeführt werden.
  • Tools: JaCoCo (Java Code Coverage Library), Cobertura u. a.

b) Was ist beim Mocking zu beachten?

  • Mocking hilft, eigene Logik gezielt zu testen, ohne von echten Abhängigkeiten beeinflusst zu werden.

  • Code Coverage zeigt dann an, wie gut die Logik der eigenen Klasse getestet ist – nicht die der gemockten Abhängigkeiten!

  • Wichtig:

    • Die Testabdeckung ist nur sinnvoll für den selbst geschriebenen Code, nicht für das Verhalten von Mocks!
    • Zu viele Mocks können zu „künstlich hoher“ Coverage führen, wenn kritische Pfade durch Abhängigkeiten nicht getestet werden.

Praxis-Tipps:

  • Nutze Coverage-Berichte, um Lücken im eigenen Code zu finden.
  • Falsches Vertrauen in hohe Coverage vermeiden – prüfe auch Testqualität!
  • Überlege, ob Integrationstests für das Zusammenspiel mit echten Abhängigkeiten nötig sind.

4. Zusammenfassung und Empfehlungen

  • Mocking ist ein Schlüsselelement in TDD und beschleunigt die Entwicklung durch frühe, isolierte Tests.
  • Automatisiere alle Mock-Tests in deiner CI/CD-Pipeline, damit Fehler schnell gefunden werden.
  • Nutze Code Coverage-Tools, aber interpretiere die Ergebnisse mit Bedacht: Gute Coverage heißt nicht automatisch gute Tests!
  • Denke immer auch an Integrationstests, um das Zusammenspiel mit echten Komponenten zu prüfen.

Checkliste für den Entwicklungsprozess

  • Werden Mock-basierte Tests automatisch bei jedem Commit ausgeführt?
  • Sind die Coverage-Berichte regelmäßig verfügbar und werden sie vom Team geprüft?
  • Wird Mocking als Werkzeug zur Isolierung und für frühzeitiges Testen genutzt?
  • Gibt es auch (wo nötig) Integrationstests mit echten Abhängigkeiten?
