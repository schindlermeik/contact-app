Handout: Fehlerbehandlung und Transaktionen

Um robuste und zuverlässige Datenbankanwendungen zu entwickeln, sind zwei Konzepte unerlässlich: eine effektive Fehlerbehandlung und der richtige Umgang mit Transaktionen.

5.1 Umgang mit SQLExceptions

Bei der Interaktion mit einer Datenbank können viele Dinge schieflaufen. Die Verbindung könnte unterbrochen werden, die SQL-Syntax könnte fehlerhaft sein oder die Datenbank könnte eine Integritätsverletzung melden (z.B. ein Duplikat in einem Unique-Feld). All diese Probleme werden in Java als SQLException-Objekte behandelt.

Jede Methode, die eine Datenbankaktion durchführt (z.B. getConnection(), executeQuery(), executeUpdate()), deklariert, dass sie eine SQLException werfen kann. Das bedeutet, du musst diese Ausnahmen entweder mit einem try-catch-Block behandeln oder sie weiterwerfen.

Ein SQLException-Objekt enthält wichtige Informationen:

  • Fehlermeldung: Eine menschenlesbare Beschreibung des Fehlers (getMessage()).
  • SQL-State: Ein fünfstelliger Code, der eine genauere, standardisierte Beschreibung des Fehlertyps liefert (getSQLState()).
  • Vendor-Code: Ein datenbankspezifischer Fehlercode (getErrorCode()).

Beispiel für die Fehlerbehandlung:

import java.sql.*;

public class ExceptionHandling {

    public static void main(String[] args) {
        Connection connection = null;
        try {
            connection = DriverManager.getConnection("jdbc:mysql://localhost:3306/meine_datenbank", "benutzer", "passwort");
            Statement statement = connection.createStatement();
            
            // Absichtlich fehlerhafter SQL-Befehl
            String sql = "SELECT * FROM nicht_existente_tabelle";
            ResultSet resultSet = statement.executeQuery(sql);
            
        } catch (SQLException e) {
            // Hier wird der Fehler behandelt
            System.err.println("Es ist ein Datenbankfehler aufgetreten!");
            System.err.println("Fehlermeldung: " + e.getMessage());
            System.err.println("SQL State: " + e.getSQLState());
            System.err.println("Vendor Code: " + e.getErrorCode());
            
        } finally {
            // Unabhängig vom Erfolg oder Misserfolg wird die Verbindung geschlossen
            if (connection != null) {
                try {
                    connection.close();
                } catch (SQLException e) {
                    System.err.println("Fehler beim Schließen der Verbindung: " + e.getMessage());
                }
            }
        }
    }
}

5.2 Transaktionen mit JDBC (setAutoCommit, commit, rollback)

Eine Transaktion ist eine logische Arbeitseinheit, die aus einer oder mehreren SQL-Anweisungen besteht. Eine Transaktion muss entweder komplett ausgeführt werden (sie wird committet) oder komplett rückgängig gemacht werden (sie wird gerollt) – es gibt nichts dazwischen. Dieses Prinzip wird als ACID (Atomicity, Consistency, Isolation, Durability) bezeichnet und ist für die Datenintegrität von größter Bedeutung.

Standardmäßig ist in JDBC der AutoCommit-Modus aktiviert. Das bedeutet, dass jede einzelne SQL-Anweisung automatisch als separate Transaktion behandelt und sofort committet wird.

Um mehrere Anweisungen als eine einzige Transaktion zu behandeln, musst du den AutoCommit-Modus deaktivieren:

connection.setAutoCommit(false);

Sobald AutoCommit deaktiviert ist, musst du Transaktionen manuell steuern:

  • connection.commit(): Schließt die Transaktion erfolgreich ab. Alle seit dem letzten commit oder rollback ausgeführten Anweisungen werden permanent in der Datenbank gespeichert.
  • connection.rollback(): Macht die Transaktion rückgängig. Alle seit dem letzten commit oder rollback vorgenommenen Änderungen werden verworfen.

Beispiel für eine Transaktion:

Stellen wir uns eine Banküberweisung vor, die aus zwei Schritten besteht:

  1. Geld vom Quellkonto abziehen.
  2. Geld auf das Zielkonto einzahlen.

Beide Schritte müssen entweder beide erfolgreich sein oder beide fehlschlagen.

import java.sql.*;

public class TransactionExample {

    public static void main(String[] args) {
        Connection connection = null;
        try {
            connection = DriverManager.getConnection("jdbc:mysql://localhost:3306/meine_datenbank", "benutzer", "passwort");
            
            // 1. AutoCommit-Modus deaktivieren
            connection.setAutoCommit(false);
            
            // Angenommen: Kontostände von Konto-ID 1 und 2
            double betrag = 100.00;
            int vonKontoId = 1;
            int zuKontoId = 2;
            
            // 2. Ersten Befehl ausführen: Geld abziehen
            String sqlAbziehen = "UPDATE konto SET guthaben = guthaben - ? WHERE id = ?";
            try (PreparedStatement stmtAbziehen = connection.prepareStatement(sqlAbziehen)) {
                stmtAbziehen.setDouble(1, betrag);
                stmtAbziehen.setInt(2, vonKontoId);
                stmtAbziehen.executeUpdate();
            }
            
            // 3. Zweiten Befehl ausführen: Geld einzahlen
            String sqlEinzahlen = "UPDATE konto SET guthaben = guthaben + ? WHERE id = ?";
            try (PreparedStatement stmtEinzahlen = connection.prepareStatement(sqlEinzahlen)) {
                stmtEinzahlen.setDouble(1, betrag);
                stmtEinzahlen.setInt(2, zuKontoId);
                stmtEinzahlen.executeUpdate();
            }
            
            // 4. Bei Erfolg: Transaktion committen
            connection.commit();
            System.out.println("Überweisung erfolgreich abgeschlossen.");
            
        } catch (SQLException e) {
            // 5. Bei Fehlern: Transaktion rückgängig machen
            try {
                if (connection != null) {
                    connection.rollback();
                    System.err.println("Überweisung fehlgeschlagen. Transaktion rückgängig gemacht.");
                }
            } catch (SQLException rollbackEx) {
                System.err.println("Fehler beim Rollback: " + rollbackEx.getMessage());
            }
            e.printStackTrace();
            
        } finally {
            // 6. Verbindung wieder schließen und AutoCommit zurücksetzen
            try {
                if (connection != null) {
                    connection.setAutoCommit(true); // Guter Stil: Ursprünglichen Zustand wiederherstellen
                    connection.close();
                }
            } catch (SQLException closeEx) {
                System.err.println("Fehler beim Schließen der Verbindung: " + closeEx.getMessage());
            }
        }
    }
}

5.3 Ressourcen-Sicherheit durch Try-with-Resources

Das manuelle Schließen von Ressourcen in einem finally-Block kann fehleranfällig und unübersichtlich sein. Java bietet seit Version 7 den try-with-resources-Block, der dieses Problem elegant löst.

Der try-with-resources-Block stellt sicher, dass alle Ressourcen, die innerhalb der Klammern deklariert werden und die das AutoCloseable-Interface implementieren (wie Connection, Statement, PreparedStatement und ResultSet), am Ende des Blocks automatisch geschlossen werden.

Beispiel mit try-with-resources:

import java.sql.*;

public class TryWithResourcesExample {

    public static void main(String[] args) {
        
        try (Connection connection = DriverManager.getConnection("jdbc:mysql://localhost:3306/meine_datenbank", "benutzer", "passwort");
             PreparedStatement preparedStatement = connection.prepareStatement("SELECT * FROM mitarbeiter")) {
            
            try (ResultSet resultSet = preparedStatement.executeQuery()) {
                // Mit dem ResultSet arbeiten
                while (resultSet.next()) {
                    System.out.println(resultSet.getString("vorname"));
                }
            }
            // Das ResultSet wird hier automatisch geschlossen
            
        } catch (SQLException e) {
            // Der Catch-Block behandelt alle auftretenden SQLExceptions
            e.printStackTrace();
        }
        // Connection und PreparedStatement werden hier automatisch geschlossen
    }
}

Dieses Vorgehen ist die empfohlene Best Practice für die Arbeit mit JDBC, da es den Code sauberer und sicherer macht. Du musst dich nicht mehr selbst um das manuelle Schließen der Ressourcen kümmern.
