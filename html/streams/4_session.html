Handout: Funktionsweise von Streams

1. Lazy Evaluation und Pipeline-Prinzip

Lazy Evaluation bedeutet: Stream-Operationen werden erst ausgeführt, wenn eine Terminal-Operation (wie collect(), forEach(), count()) aufgerufen wird. Bis dahin „merkt“ sich Java nur die Kette der Operationen.

Pipeline-Prinzip:

  • Stream-Operationen werden als Verkettung (Pipeline) notiert.
  • Die Daten „fließen“ von einer Operation zur nächsten – aber: Die Verarbeitung startet erst mit der Terminal-Operation.

Beispiel:

List<String> namen = Arrays.asList("Anna", "Bernd", "Clara");

namen.stream()
    .filter(s -> {
        System.out.println("Filter: " + s);
        return s.length() > 3;
    })
    .map(s -> {
        System.out.println("Map: " + s);
        return s.toUpperCase();
    });
// Bis hier passiert GAR NICHTS!

Erst jetzt wird die Pipeline aktiviert:

List<String> ergebnis = namen.stream()
    .filter(s -> {
        System.out.println("Filter: " + s);
        return s.length() > 3;
    })
    .map(s -> {
        System.out.println("Map: " + s);
        return s.toUpperCase();
    })
    .collect(Collectors.toList());
  • Ergebnis: Die Ausgaben zeigen, dass Filter und Map erst bei collect() ausgeführt werden.

2. Reihenfolge und Kurzschluss-Operationen

a) Reihenfolge der Operationen

  • Die Reihenfolge der Operatoren beeinflusst das Ergebnis und die Performance.
  • Filter möglichst früh platzieren, um Folgeoperationen nur für relevante Daten auszuführen.

b) Kurzschluss-Operationen (Short-circuiting)

Manche Terminal-Operationen beenden den Stream sofort, wenn das Ergebnis feststeht:

  • anyMatch(), allMatch(), noneMatch()

    • Sobald feststeht, dass die Bedingung erfüllt oder verletzt ist, wird die Verarbeitung gestoppt.

    • Beispiel:

      Stream.of("a", "b", "c")
      .anyMatch(s -> s.equals("b")); // stoppt nach "b"

  • findFirst(), findAny()

    • Beenden, sobald ein Element gefunden wurde.

Vorteil: Das spart Rechenzeit, besonders bei großen Datenmengen oder aufwändigen Operationen.

3. Unterschied zwischen intermediären und terminalen Operationen

a) Intermediäre (zwischenliegende) Operationen

  • Geben wieder einen Stream zurück (z. B. filter(), map(), sorted())
  • Sind „lazy“: Werden erst bei Terminal-Operation ausgeführt
  • Können beliebig verkettet werden

b) Terminal-Operationen

  • Beenden die Pipeline und liefern ein Ergebnis (z. B. collect(), forEach(), count())
  • Starten tatsächlich die Verarbeitung
  • Können nur einmal pro Stream ausgeführt werden (danach ist der Stream „verbraucht“)

Beispiel:

// Intermediär:
Stream<String> stream = namen.stream().filter(s -> s.length() > 4);

// Terminal:
List<String> ergebnis = stream.collect(Collectors.toList());

4. Übungen zum Selbststudium

  1. Schreibe eine Stream-Pipeline mit mindestens drei intermediären und einer terminalen Operation. Beobachte mit Ausgaben, wann welcher Teil tatsächlich ausgeführt wird.

  2. Verwende anyMatch() und prüfe, wie viele Elemente tatsächlich verarbeitet werden, indem du im Lambda einen Zähler erhöhst oder eine Ausgabe machst.

  3. Experimentiere mit dem Platzieren von filter() früh oder spät in der Pipeline. Was fällt dir auf, wenn du z. B. erst sorted() und dann filter() verwendest – und umgekehrt?

5. Merksätze

  • Streams sind „lazy“: Nichts passiert ohne Terminal-Operation!

  • Die Reihenfolge der Operationen beeinflusst Effizienz und Ergebnis.

  • Kurzschluss-Operationen können Streams frühzeitig beenden.

  • Unterschied:

    • Intermediär = Zwischenstopp, gibt wieder einen Stream zurück.
    • Terminal = Ziel, liefert Ergebnis und beendet den Stream.
