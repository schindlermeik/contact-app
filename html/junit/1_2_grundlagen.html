Handout: Grundlegende Konzepte von JUnit

1. Was ist JUnit?

a) Historie und Entwicklung

  • JUnit ist das bekannteste Framework für Unit-Tests in Java.
  • Entwickelt ab 1997 von Erich Gamma und Kent Beck, Pioniere der Test-First- und Extreme Programming-Bewegung.
  • JUnit wurde als leichtgewichtiges, einfach zu nutzendes Testframework konzipiert und hat den Siegeszug des automatisierten Testens in der Java-Welt entscheidend geprägt.
  • Die Entwicklung von JUnit ist eng mit der Entstehung und Verbreitung von Test Driven Development (TDD) verbunden.

b) JUnit-Versionen: Ein kurzer Überblick

  • JUnit 3

    • Tests wurden durch Vererbung einer Basisklasse geschrieben.
    • Einrichtung und Ausführung noch etwas umständlich.

  • JUnit 4

    • Einführung von Annotationen (@Test, @Before, @After usw.), keine Basisklasse mehr nötig.
    • Deutlich flexibler und moderner in der Anwendung.

  • JUnit 5 (auch „Jupiter“ genannt)

    • Modularer Aufbau, viele neue Features und Verbesserungen.
    • Bessere Unterstützung für neue Java-Versionen, Lambda-Ausdrücke und Erweiterungen.
    • Besteht aus drei Hauptkomponenten: JUnit Platform, JUnit Jupiter (neue APIs), JUnit Vintage (Abwärtskompatibilität).
    • Aktuell der De-facto-Standard für neue Projekte.

2. Aufbau eines Testfalls

a) Testmethoden

  • Jede Methode, die als Test dienen soll, erhält die Annotation @Test.
  • Die Methode sollte möglichst keine Parameter und void als Rückgabetyp haben.

Beispiel:

import org.junit.jupiter.api.Test;

class CalculatorTest {
    @Test
    void testAddition() {
        Calculator calc = new Calculator();
        int result = calc.add(2, 3);
        // Assertion folgt...
    }
}

b) Assertions

  • Assertions überprüfen, ob das tatsächliche Ergebnis dem erwarteten Ergebnis entspricht.

  • Schlägt eine Assertion fehl, wird der Test als fehlgeschlagen markiert.

  • Typische Assertions (JUnit 5):

    • assertEquals(expected, actual)
    • assertTrue(condition)
    • assertFalse(condition)
    • assertThrows(Exception.class, () -> { ... })

Beispiel:

import static org.junit.jupiter.api.Assertions.*;

@Test
void testAddition() {
    Calculator calc = new Calculator();
    int result = calc.add(2, 3);
    assertEquals(5, result); // Erwartung: 2 + 3 = 5
}

c) Setup und Teardown (Vorbereitung und Aufräumen)

  • Für wiederkehrende Vorbereitungen gibt es spezielle Methoden:

    • @BeforeEach: Wird vor jedem Test ausgeführt (z.B. Initialisierung).
    • @AfterEach: Wird nach jedem Test ausgeführt (z.B. Aufräumen).

  • Es gibt auch @BeforeAll und @AfterAll für einmalige Aktionen pro Testklasse.

Beispiel:

import org.junit.jupiter.api.*;

class CalculatorTest {
    Calculator calc;

    @BeforeEach
    void setUp() {
        calc = new Calculator(); // vor jedem Test ein neues Calculator-Objekt
    }

    @AfterEach
    void tearDown() {
        // z.B. Ressourcen freigeben, falls nötig
    }

    @Test
    void testAddition() {
        assertEquals(5, calc.add(2, 3));
    }
}

d) Testklassen-Struktur

  • Eine Testklasse testet in der Regel eine Produktivklasse.
  • Konvention: Name der Testklasse = Name der Produktivklasse + Test (z.B. Calculator → CalculatorTest)
  • Testklassen liegen meist im Verzeichnis src/test/java.

Beispiel für die Struktur eines Projekts:

projekt-root/
  ├─ src/
  │   ├─ main/
  │   │   └─ java/
  │   │       └─ com/beispiel/Calculator.java
  │   └─ test/
  │       └─ java/
  │           └─ com/beispiel/CalculatorTest.java

e) Benennungskonventionen für Tests

  • Testmethoden sollten aussagekräftige Namen haben, die das getestete Verhalten beschreiben.

  • Gängige Muster:

    • methodName_state_expectedResult
    • shouldDoSomething_WhenCondition

  • Beispiele:

    • add_TwoPositiveNumbers_ReturnsSum
    • shouldThrowException_WhenInputIsNull

Vorteil: Gute Namen erleichtern das Verständnis und die Wartung der Tests.

3. Zusammenfassung

  • JUnit ist das Standard-Framework für Unit-Testing in Java und mittlerweile in Version 5 (Jupiter) weit verbreitet.
  • Testfälle bestehen aus Testmethoden mit Assertions, werden klar benannt und sauber strukturiert.
  • Mit Setup- und Teardown-Methoden kann man Tests noch übersichtlicher und wartbarer gestalten.

Checkliste für eigene JUnit-Tests:

  • Habe ich für jede wichtige Methode mindestens einen Test?
  • Sind meine Testnamen selbsterklärend?
  • Nutze ich Assertions, um Erwartungen klar zu prüfen?
  • Ist mein Setup einfach und verständlich?

Tipp: Nutze die umfangreiche Dokumentation von JUnit (JUnit 5 User Guide) und experimentiere mit einfachen Testklassen, um ein Gefühl für das Framework zu bekommen!
