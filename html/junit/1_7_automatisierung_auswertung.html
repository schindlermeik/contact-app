Handout: Auswertung und Automatisierung von Unit-Tests

1. Test-Reports und deren Auswertung

a) Was sind Test-Reports?

  • Test-Reports sind strukturierte Auswertungen über den Status und Verlauf der ausgeführten Tests.

  • Sie zeigen:

    • Welche Tests bestanden sind (grün)
    • Welche Tests fehlgeschlagen oder fehlerhaft sind (rot)
    • Detaillierte Fehlermeldungen, Stacktraces, ggf. Screenshots

b) Wie entstehen Test-Reports?

  • Test-Reports werden automatisch von Build- und Test-Tools erzeugt, z.B. Maven, Gradle oder der IDE.
  • Sie liegen oft im XML- oder HTML-Format vor und werden im Projektordner gespeichert.

Beispiel: Maven + Surefire Plugin

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>3.2.5</version>
</plugin>
  • Nach dem Ausführen von mvn test findest du die Berichte unter target/surefire-reports/ (z.B. als .txt oder .xml)

HTML-Reports (besser lesbar): Mit Plugins wie Surefire Report Plugin oder JaCoCo für Testabdeckung.

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-report-plugin</artifactId>
    <version>3.2.5</version>
</plugin>
  • Nach mvn surefire-report:report gibt es eine HTML-Auswertung unter target/site/surefire-report.html

c) Auswertung von Test-Reports

  • Prüfe: Welche Tests sind fehlgeschlagen? Warum?
  • Nutze die Fehlerausgaben und Stacktraces für das gezielte Debugging.
  • Vergleiche regelmäßig die Ergebnisse, um Trends oder wiederkehrende Fehler zu erkennen.
  • In Team-Projekten: Prüfe, ob neue Fehler im Code von Kollegen auftreten.

Praxis-Tipp: Ein regelmäßiger Blick in die Reports hilft, „schleichende“ Fehler und Test-Regressionen frühzeitig zu erkennen.

2. Integration in CI/CD-Pipelines (z.B. Jenkins, GitHub Actions)

a) Was ist CI/CD?

  • CI (Continuous Integration) bedeutet: Alle Änderungen am Code werden regelmäßig in ein zentrales Repository integriert und automatisch getestet.
  • CD (Continuous Delivery/Deployment): Der Übergang vom Test zur automatischen Bereitstellung auf einer Zielumgebung.

b) Warum Unit-Tests automatisieren?

  • Fehler werden sofort sichtbar, nicht erst nach Tagen oder Wochen.
  • Codequalität bleibt konstant hoch, Rückschritte (Regressionen) werden verhindert.
  • Teammitglieder arbeiten immer mit geprüftem, funktionierendem Code.

c) Wie funktioniert die Integration?

1. Beispiel: Jenkins

  • Jenkins ist ein verbreiteter Open-Source-Server für CI/CD.

  • Typischer Workflow:

    1. Entwickler pusht Code in Git-Repository (z.B. GitHub, GitLab).
    2. Jenkins startet automatisch einen Build-Job.
    3. Unit-Tests laufen im Buildprozess (mvn test).
    4. Test-Reports werden gespeichert und als HTML im Jenkins-Interface angezeigt.
    5. Fehler/Erfolge sind für alle sichtbar; fehlschlagende Builds werden hervorgehoben.

  • Jenkins kann E-Mail-Benachrichtigungen oder Chat-Alerts bei fehlgeschlagenen Tests verschicken.

2. Beispiel: GitHub Actions

  • GitHub Actions ist ein integrierter CI/CD-Dienst auf GitHub.
  • Mit einer Workflow-Datei (.github/workflows/build.yml) werden Unit-Tests nach jedem Push/PR automatisch ausgeführt.

Beispiel für Maven-Projekt:

name: Java CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Set up JDK 21
      uses: actions/setup-java@v4
      with:
        java-version: '21'
    - name: Build with Maven
      run: mvn clean install
  • Die Tests laufen auf jedem GitHub-Server nach jedem Commit, die Ergebnisse sind online sichtbar.
  • Testfehler blockieren das Mergen von Pull Requests.

d) Tipps und Best Practices

  • Mache Unit-Tests zum festen Bestandteil jedes Builds.
  • Test-Reports immer im Build-Log/Interface anzeigen (HTML, Badge, Status-Icon).
  • Fehlgeschlagene Builds niemals ignorieren! Erst lösen, dann weiterentwickeln.
  • Nutze „Badges“ im Repository (z.B. ), um den Teststatus sichtbar zu machen.

3. Zusammenfassung

  • Test-Reports dokumentieren den Zustand deiner Software nach jedem Testlauf – prüfe sie regelmäßig!
  • Die Integration von Tests in CI/CD-Pipelines sorgt für schnelle Rückmeldung, bessere Zusammenarbeit und weniger Fehler im Code.
  • Nutze moderne Tools (Maven, Jenkins, GitHub Actions), um Testprozesse zu automatisieren und transparent zu machen.

Checkliste für deine Projekte

  • Werden Test-Reports automatisch erzeugt?
  • Sind sie für das Team einsehbar?
  • Läuft die Testausführung nach jedem Commit/Push?
  • Werden fehlgeschlagene Tests sofort sichtbar und behandelt?
  • Ist das Build-Tool (Maven/Gradle) richtig konfiguriert?
