Handout: Best Practices für Unit-Testing

1. Isolierte Tests

Was bedeutet „isolation“ im Unit-Testing?

  • Isolierte Tests prüfen immer nur eine einzige Code-Einheit (Methode, Funktion oder Klasse) – ohne externe Abhängigkeiten (z.B. Datenbanken, Netzwerke, Dateisystem, andere Module).
  • Sie sorgen dafür, dass Fehlerursachen schnell und eindeutig gefunden werden können.

Wie erreicht man Isolation?

  • Test-Doubles wie Mocks, Stubs, Fakes oder Spies verwenden.
  • Keine echten Datenbankverbindungen, keine HTTP-Requests, keine Zeit- oder Zufallsabhängigkeiten im Unit-Test!
  • Möglichst keine Seiteneffekte (z.B. keine globalen Variablen verändern).

Beispiel: Statt in einem Test wirklich Daten in eine Datenbank zu speichern, wird der Datenbankzugriff gemockt, sodass der Test nur prüft, ob die Methode aufgerufen wird.

2. Testabdeckung (Coverage) und ihre Aussagekraft

Was ist Testabdeckung?

  • Testabdeckung ist ein Maß dafür, wie viel Prozent des Codes durch Tests ausgeführt werden.
  • Meist gemessen als „Line Coverage“ (wie viele Zeilen werden beim Testen durchlaufen?) oder „Branch Coverage“ (wie viele Verzweigungen werden abgedeckt?).

Aussagekraft und Grenzen

  • Hohe Testabdeckung ist grundsätzlich gut, garantiert aber keine Fehlerfreiheit!
  • Auch 100 % Coverage sagt nichts über die Qualität der Tests aus – evtl. sind sie nur „leere“ Tests, die kaum etwas prüfen.
  • Wichtiger als die Zahl ist, dass die wichtigsten und kritischsten Anwendungsfälle durch sinnvolle Tests abgedeckt werden.

Praxis-Tipp: Setze Coverage-Tools wie JaCoCo ein, aber verlasse dich nicht blind auf die Zahlen. Überprüfe regelmäßig, ob die getesteten Szenarien wirklich aussagekräftig sind.

3. Was sollte (nicht) getestet werden?

Was sollte getestet werden?

  • Alle Logik, die Fehler verursachen kann (z.B. Berechnungen, Validierungen, Bedingungen)
  • Edge Cases / Sonderfälle (z.B. leere Listen, Null-Werte, negative Zahlen)
  • Fehlerbehandlung (z.B. Exceptions bei ungültigen Eingaben)
  • Öffentliche Methoden einer Klasse (Public API)

Was sollte man NICHT testen?

  • Framework-Code und Standardbibliotheken (z.B. „Testet nicht, ob ArrayList.add() funktioniert!“)
  • Private Methoden (Sie sollten über öffentliche Methoden abgedeckt werden)
  • Dinge, die sich häufig ändern, aber keinen Wert bringen (z.B. triviale Getter/Setter)
  • Zufällige Daten oder Zeit ohne Kontrolle/Mocking

4. Typische Fehler und Anti-Patterns beim Testen

Anti-Pattern Beschreibung Besser so:
Zu große Tests Test prüft mehrere Methoden auf einmal Pro Testfall nur EIN Verhalten prüfen
Nicht-deterministische Tests Ergebnisse hängen von Zufall/Zeit/System ab Werte mocken oder kontrolliert setzen
Copy-Paste-Tests Viel doppelter Code, schwer wartbar Setup-Code wiederverwenden (@BeforeEach)
Schwammige Testnamen Test „irgendwas“ – Name sagt nichts aus Aussagekräftige Namen mit Verhalten & Fall
Ungetestete Randfälle Test nur Standardfälle, nicht Edge Cases Immer auch Fehlerfälle, Grenzfälle testen
Zu viel oder zu wenig Mocking Alles wird gemockt, auch Unnötiges Nur das mocken, was nötig ist
Testdaten im Test versteckt Testdaten schwer verständlich/hart codiert Testdaten sprechend und sichtbar wählen

Tipp: Halte Tests klein, eindeutig und unabhängig. Jeder Test sollte genau einen Aspekt des Verhaltens prüfen.

5. Test Driven Development (TDD) – Kurze Einführung

Was ist TDD?

  • TDD (Test Driven Development) ist eine Methode der agilen Softwareentwicklung:

    • 1. Schreibe zuerst einen (fehlenden) Test.
    • 2. Schreibe dann nur so viel Code, dass der Test gerade so besteht.
    • 3. Refaktoriere den Code und die Tests bei Bedarf.
    • Dann: Wiederhole von vorne.

Kernidee: Man entwickelt iterativ und in sehr kleinen Schritten. Jeder neue Code wird sofort durch einen Test abgesichert.

Vorteile von TDD

  • Hilft, die Anforderungen und das gewünschte Verhalten zu klären.
  • Führt zu besser testbarem und oft besser strukturiertem Code.
  • Minimiert die Fehlerquote, weil jeder Schritt sofort geprüft wird.

Beispiel: Mini-TDD-Zyklus

  1. Test schreiben

    @Test
void testAdd() {
    Calculator calc = new Calculator();
    assertEquals(5, calc.add(2, 3));
}

  2. Code schreiben

    int add(int a, int b) { return a + b; }

  3. Refaktorieren (z.B. Code sauberer machen, doppelte Logik entfernen)

6. Zusammenfassung (Best Practices auf einen Blick)

  • Schreibe isolierte, unabhängige Tests.
  • Strebe sinnvolle Testabdeckung an, nicht nur hohe Zahlen.
  • Teste Logik, Fehlerfälle und Sonderfälle – nicht triviale Frameworksachen.
  • Vermeide typische Test-Anti-Patterns.
  • TDD ist eine wertvolle Technik für nachhaltige Entwicklung, aber nicht Pflicht für jeden.

Checkliste für bessere Unit-Tests

  • Ist mein Test wirklich unabhängig und isoliert?
  • Teste ich kritische Logik und Fehlerfälle?
  • Ist der Testname klar und verständlich?
  • Könnte ein neuer Entwickler die Testfälle schnell erfassen?
  • Nutze ich sinnvolle Testdaten und überprüfe ich wichtige Szenarien?
