Handout: Einführung in Unit-Testing

1. Was sind Unit-Tests?

Unit-Tests (auf Deutsch: Modultests oder Komponententests) sind automatisierte Tests, die einzelne, klar abgegrenzte Einheiten eines Programms – meistens Methoden oder Funktionen – auf ihre Korrektheit prüfen. Ziel ist es, möglichst unabhängig vom Rest des Systems zu überprüfen, ob der Code wie erwartet funktioniert.

2. Motivation und Ziele von Unit-Tests

a) Fehler frühzeitig erkennen

  • Vorteil: Fehler werden oft schon während der Entwicklung entdeckt, nicht erst beim Testen der Gesamtanwendung.
  • Beispiel: Eine Methode, die Zahlen addieren soll, gibt plötzlich bei bestimmten Eingaben ein falsches Ergebnis zurück. Ein Unit-Test schlägt dann sofort fehl und zeigt, dass etwas nicht stimmt.

b) Dokumentation und Verständlichkeit des Codes

  • Unit-Tests dokumentieren das Verhalten von Methoden.
  • Andere Entwickler können durch das Lesen der Tests schnell verstehen, wie eine Methode zu verwenden ist und was sie leisten soll.
  • Beispiel: Ein Test prüft, dass eine Funktion beim Aufruf mit einer negativen Zahl eine Ausnahme (Exception) wirft. Daraus kann man schließen, dass negative Zahlen nicht erlaubt sind.

c) Grundlage für Refactoring

  • Refactoring: Das Überarbeiten von Code, um ihn zu verbessern, ohne seine Funktionalität zu verändern.
  • Mit vorhandenen Unit-Tests kann man sicherstellen, dass der Code nach dem Umbau immer noch korrekt funktioniert.
  • Vorteil: Entwickler trauen sich, Verbesserungen vorzunehmen, weil sie sofort sehen, wenn unbeabsichtigte Fehler entstehen.

3. Abgrenzung: Unit-Test vs. Integrationstest vs. Systemtest

Es gibt verschiedene Arten automatisierter Tests. Sie unterscheiden sich durch ihren Umfang und das, was sie testen.

a) Unit-Test (Modultest, Komponententest)

  • Testet: Einzelne Einheiten (meist Methoden/Funktionen/Klassen)
  • Ziel: Isoliert prüfen, ob eine Komponente korrekt funktioniert
  • Hilfsmittel: Oft werden Abhängigkeiten durch Mocks/Fakes ersetzt, damit wirklich nur die Methode selbst getestet wird.

Beispiel:

@Test
void testSum() {
    Calculator calc = new Calculator();
    assertEquals(4, calc.sum(2, 2));
}

b) Integrationstest

  • Testet: Das Zusammenspiel mehrerer Komponenten oder Module
  • Ziel: Überprüfen, ob Schnittstellen und das Zusammenspiel verschiedener Teile funktionieren
  • Beispiel: Eine Klasse ruft einen Service auf, der wiederum mit einer Datenbank kommuniziert – im Integrationstest wird das Gesamtergebnis überprüft.

Skizze (Text):

[Komponente A] → [Service B] → [Datenbank] Der Integrationstest prüft, ob Daten von A über B korrekt in die Datenbank gelangen und gelesen werden können.

c) Systemtest

  • Testet: Das gesamte Anwendungssystem als Ganzes
  • Ziel: Prüfen, ob die gesamte Anwendung aus Sicht des Benutzers wie gewünscht funktioniert
  • Beispiel: Ein automatisierter Test prüft, ob eine Webanwendung nach dem Login die Startseite korrekt anzeigt und Daten darstellt.

4. Warum sind Unit-Tests in der modernen Softwareentwicklung wichtig?

  • Sie ermöglichen kontinuierliche Weiterentwicklung und helfen, technische Schulden zu vermeiden.
  • Sie fördern die Modularisierung des Codes, weil testbarer Code meist besser strukturiert ist.
  • Sie sind ein zentraler Baustein moderner Entwicklungsprozesse (Stichwort: Continuous Integration, DevOps, agile Methoden).
  • Gut getestete Software ist robuster, zuverlässiger und langlebiger.

5. Tipps für die Praxis

  • Schreibe Unit-Tests möglichst zeitnah zum Code (am besten direkt beim Entwickeln).
  • Teste sowohl typische als auch außergewöhnliche Fälle (z.B. auch ungültige Eingaben).
  • Halte Unit-Tests einfach, kurz und unabhängig voneinander.
  • Dokumentiere mit sprechenden Testnamen, was genau geprüft wird.

6. Zusammenfassung (Kurzfassung zum Mitnehmen)

  • Unit-Tests prüfen einzelne Code-Einheiten automatisch auf Korrektheit.
  • Sie helfen, Fehler früh zu erkennen, dienen als Dokumentation und unterstützen beim Refactoring.
  • Sie unterscheiden sich von Integrationstests (Zusammenspiel) und Systemtests (Gesamtsystem).
  • Sie sind eine wichtige Grundlage für moderne, nachhaltige Softwareentwicklung.
