Handout: Schreiben von Unit-Tests mit JUnit

1. Grundlegende Assertions (assertEquals, assertTrue, etc.)

Was sind Assertions?

  • Assertions (deutsch: „Behauptungen“) sind Methoden, mit denen überprüft wird, ob ein tatsächliches Ergebnis mit einem erwarteten Wert übereinstimmt.
  • Wenn eine Assertion fehlschlägt, markiert JUnit den Test als fehlgeschlagen.

Die wichtigsten Assertions in JUnit 5

Assertion Beschreibung Beispiel
assertEquals(expected, actual) Prüft auf Gleichheit assertEquals(5, calc.add(2, 3));
assertNotEquals(expected, actual) Prüft auf Ungleichheit assertNotEquals(0, calc.add(2, 3));
assertTrue(condition) Erwartet, dass die Bedingung wahr ist assertTrue(list.isEmpty());
assertFalse(condition) Erwartet, dass die Bedingung falsch ist assertFalse(str.isEmpty());
assertNull(value) Erwartet, dass der Wert null ist assertNull(obj.getSomething());
assertNotNull(value) Erwartet, dass der Wert nicht null ist assertNotNull(user.getId());
assertArrayEquals(exp, act) Prüft, ob zwei Arrays gleich sind assertArrayEquals(new int[]{1,2}, arr);
assertSame(expected, actual) Prüft, ob zwei Objekte identisch (selbes Objekt) sind assertSame(obj1, obj2);
assertNotSame(expected, actual) Prüft, ob zwei Objekte nicht identisch sind assertNotSame(obj1, obj2);

Tipp: Man kann jeder Assertion einen eigenen Fehlermeldungs-String hinzufügen, z.B. assertEquals(5, calc.add(2,3), "Addition sollte 5 ergeben");

2. Tests für verschiedene Datentypen

  • Primitives: Zahlen, Booleans, Zeichen z.B. assertEquals(7, multiply(2,3)+1);

  • Strings: z.B. assertEquals("Hallo", greeter.greet("Hallo"));

  • Arrays: z.B. assertArrayEquals(new int[]{1,2,3}, list.toArray());

  • Listen/Sets/Maps: Mit Hilfsmethoden (contains, size, etc.) prüfen:

    assertEquals(3, myList.size());
assertTrue(mySet.contains("Java"));

  • Eigene Objekte: Prüfe auf relevante Felder oder implementiere equals() für deine Klasse:

    assertEquals(new Person("Max", 22), myService.getPerson(0));

3. Testing von Exceptions (assertThrows)

Warum Exceptions testen?

  • Es ist wichtig zu prüfen, ob Methoden bei ungültigen Eingaben oder unerwarteten Situationen korrekt eine Exception werfen.

Beispiel für das Testen von Exceptions:

import static org.junit.jupiter.api.Assertions.*;

@Test
void testException() {
    assertThrows(IllegalArgumentException.class, () -> {
        myService.divide(10, 0);
    }, "Division durch 0 sollte IllegalArgumentException werfen");
}
  • Der Test besteht, nur wenn tatsächlich die erwartete Exception auftritt.

4. Parameterisierte Tests

Motivation

  • Erlaubt, einen Test mit mehreren Werten auszuführen – ideal, um viele Fälle kompakt zu prüfen.

Beispiel mit JUnit 5:

  1. Abhängigkeit für JUnit Jupiter Params (junit-jupiter-params) in pom.xml hinzufügen:
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter-params</artifactId>
    <version>5.10.2</version>
    <scope>test</scope>
</dependency>
  1. Parameterisierten Test schreiben:
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.ValueSource;

@ParameterizedTest
@ValueSource(strings = {"", " ", "\t"})
void testIsBlank(String input) {
    assertTrue(myUtil.isBlank(input));
}
  • Der Test wird dreimal ausgeführt, einmal pro Wert in ValueSource.

Weitere Varianten:

  • @CsvSource (mehrere Parameter pro Testdurchlauf)
  • @MethodSource (liefert beliebige Werte)

5. Wiederverwendung von Setup-Code (@BeforeEach, @AfterEach)

  • Verhindert Duplikate und sorgt für „frische“ Objekte bei jedem Testlauf.

Beispiel:

import org.junit.jupiter.api.*;

class MyServiceTest {
    MyService service;

    @BeforeEach
    void setUp() {
        service = new MyService();
    }

    @AfterEach
    void tearDown() {
        // Aufräumarbeiten, z.B. Datenbank schließen
    }

    @Test
    void testSomething() {
        assertNotNull(service);
    }
}

Erklärung:

  • @BeforeEach: Wird vor jedem Test ausgeführt, ideal für Initialisierung.
  • @AfterEach: Wird nach jedem Test ausgeführt, z.B. um Ressourcen wieder freizugeben.

6. Gute Praxis: Beispiel für eine vollständige Testklasse

import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {
    Calculator calc;

    @BeforeEach
    void setUp() {
        calc = new Calculator();
    }

    @Test
    void testAddition() {
        assertEquals(7, calc.add(3,4));
    }

    @Test
    void testDivisionByZeroThrowsException() {
        assertThrows(ArithmeticException.class, () -> calc.divide(5,0));
    }

    @ParameterizedTest
    @ValueSource(ints = {0, 1, 42})
    void testIsEven(int number) {
        assertEquals(number % 2 == 0, calc.isEven(number));
    }
}

7. Zusammenfassung und Tipps

  • Assertions prüfen, ob Ergebnisse wie erwartet sind.
  • Teste verschiedene Datentypen und Szenarien (auch Fehlerfälle!).
  • Nutze parameterisierte Tests für verschiedene Werte.
  • Wiederverwendbarer Setup-Code macht Tests sauber und wartbar.
  • Benenne Tests klar und aussagekräftig.

Empfohlene Übung:

  • Schreibe für eine eigene kleine Klasse (z.B. Rechner, Konto, StringHelper)

    • mindestens 3 normale Tests,
    • einen Test für eine Exception,
    • einen parameterisierten Test.
