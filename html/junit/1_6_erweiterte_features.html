Handout: Erweiterte Features von JUnit

1. Tags und Test-Filter (@Tag, @Disabled)

a) Was sind Tags?

  • Tags ermöglichen es, Tests zu gruppieren und gezielt auszuführen.
  • Sie sind besonders nützlich in großen Projekten oder bei der Unterscheidung zwischen schnellen, langsamen, integrativen oder experimentellen Tests.

Beispiel:

import org.junit.jupiter.api.Tag;
import org.junit.jupiter.api.Test;

@Tag("fast")
@Test
void testFastOperation() {
    // Testcode
}

@Tag("slow")
@Test
void testSlowOperation() {
    // Testcode für aufwändigen Test
}
  • Tests mit bestimmten Tags lassen sich über Build-Tools oder IDE-Features gezielt ausführen, z.B. nur "fast"-Tests.

Praxis-Tipp: Tags helfen, „Nachts nur die langen“, „im CI nur die schnellen“ Tests laufen zu lassen!

b) Test-Filter

  • Test-Frameworks und Build-Tools (wie Maven oder Gradle) erlauben das Filtern und Selektieren von Tests anhand von Tags.

  • Beispiel (Maven):

    mvn test -Dgroups=fast

c) Tests gezielt deaktivieren (@Disabled)

  • @Disabled markiert einen Test oder eine Testklasse als vorübergehend deaktiviert (wird übersprungen, z.B. bei noch nicht implementierten Features oder instabilen Tests).

Beispiel:

import org.junit.jupiter.api.Disabled;
import org.junit.jupiter.api.Test;

@Disabled("Noch nicht implementiert")
@Test
void testComingSoon() {
    // ...
}

2. Test Suites

  • Test Suites fassen mehrere Testklassen zusammen, sodass sie gemeinsam gestartet werden können.
  • Besonders praktisch für Integrationstests, Regressionstests oder spezifische Testläufe.

Beispiel (JUnit 5):

import org.junit.platform.suite.api.SelectClasses;
import org.junit.platform.suite.api.Suite;

@Suite
@SelectClasses({CalculatorTest.class, UserServiceTest.class})
public class AllTestsSuite {
    // Diese Klasse bleibt leer!
}
  • In der Suite werden alle angegebenen Testklassen nacheinander ausgeführt.
  • Für größere Projekte lassen sich so eigene Test-Läufe (z.B. „Alle wichtigen Tests“) erstellen.

3. Zeitbegrenzung von Tests (@Timeout)

  • Langlaufende oder hängende Tests können Entwicklung und CI-Prozesse blockieren.
  • Mit @Timeout kann man für einzelne Tests eine maximale Ausführungszeit definieren.

Beispiel:

import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.Timeout;

import java.util.concurrent.TimeUnit;

@Test
@Timeout(value = 500, unit = TimeUnit.MILLISECONDS)
void testMustFinishQuickly() {
    // Code, der nicht länger als 500ms laufen darf
}
  • Läuft der Test länger, wird er automatisch als fehlgeschlagen markiert.
  • Praxis-Tipp: Auch auf Nebenläufigkeit (Threads) und Deadlocks achten!

4. Testen von Multithreading-Code

a) Warum ist das besonders?

  • Tests, die Threads oder parallele Ausführung nutzen, sind schwerer vorhersehbar und oft fehleranfällig.
  • Typische Fehler: Deadlocks, Race Conditions, nicht deterministische Ergebnisse.

b) Tipps zum Testen von Multithreading-Code

  1. Verwende @Timeout, damit Tests nicht ewig hängen bleiben!

  2. Synchronisiere auf Testergebnisse:

    • Warten, bis Threads wirklich fertig sind (z.B. mit CountDownLatch, Future, ExecutorService).

  3. Teste verschiedene Szenarien mehrfach (z.B. in Schleifen), um Race Conditions zu finden.

  4. Nutze Assertions innerhalb von Threads, achte aber darauf, dass Fehler gemeldet werden.

Beispiel:

import org.junit.jupiter.api.*;
import java.util.concurrent.*;

@Test
@Timeout(2) // Sekunden
void testParallelCalculation() throws Exception {
    ExecutorService executor = Executors.newFixedThreadPool(2);
    Future<Integer> f1 = executor.submit(() -> longCalculation(5));
    Future<Integer> f2 = executor.submit(() -> longCalculation(10));
    assertEquals(5, f1.get());
    assertEquals(10, f2.get());
    executor.shutdown();
}
  • Hier werden zwei Threads gestartet und das Ergebnis später per get() abgefragt.

Praxis-Tipps:

  • Achte darauf, Ressourcen (Threads, Executor) nach dem Test zu schließen!
  • Nicht alles lässt sich mit Unit-Tests abdecken – für komplexe Nebenläufigkeit sind zusätzliche Tools/Frameworks hilfreich (z.B. Awaitility für asynchrone Tests).

5. Zusammenfassung und Empfehlungen

  • Nutze Tags zur besseren Organisation und gezielten Ausführung von Tests.
  • Deaktiviere Tests temporär mit @Disabled – aber halte die Liste kurz und dokumentiere den Grund!
  • Mit Test Suites lassen sich mehrere Testklassen bündeln und effizient ausführen.
  • Timeouts schützen vor blockierenden Tests, besonders bei Multithreading.
  • Multithreading-Tests sind anspruchsvoll: Synchronisation, Fehlerbehandlung und Zeitmanagement sind entscheidend.

Checkliste für die Praxis

  • Habe ich wichtige Tests getaggt, um sie gezielt starten zu können?
  • Gibt es deaktivierte Tests? Wenn ja: Warum?
  • Benutze ich Test Suites für größere Testläufe?
  • Sind meine Tests schnell – oder brauche ich sinnvolle Timeouts?
  • Habe ich Multithreading-Code mit ausreichend Wiederholungen und Zeitbeschränkungen geprüft?
