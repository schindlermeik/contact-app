Handout: Architektur und Kernkonzepte von JPA

Dieses Handout taucht tiefer in die Kernarchitektur und die grundlegenden Konzepte von JPA ein.

2.1 Entity, EntityManager, Persistence Context

Dies sind die drei zentralen Bausteine, die das Herzstück jeder JPA-Anwendung bilden.

  • Entity (Entität): Eine Entität ist ein einfaches Java-Objekt, das eine Zeile in einer Datenbanktabelle repräsentiert. Um ein normales Java-Objekt zu einer Entität zu machen, wird es mit der Annotation @Entity gekennzeichnet. Jede Entität muss eine eindeutige ID haben, die mit der Annotation @Id markiert wird.

    Beispiel:

    @Entity
@Table(name = "kunden")
public class Kunde {
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Long id;

    private String name;
    private String email;

    // Getters and Setters
    // ...
}

    In diesem Beispiel ist die Klasse Kunde eine Entität, die der Tabelle kunden zugeordnet ist. Das Feld id ist der Primärschlüssel.

  • EntityManager: Der EntityManager ist die zentrale API-Schnittstelle von JPA. Er ist für die Verwaltung des Lebenszyklus von Entitäten zuständig. Mit ihm führen Sie alle CRUD-Operationen (Create, Read, Update, Delete) auf Entitäten durch. Der EntityManager ist quasi das "Tor" zur Datenbank.

    Typische Methoden des EntityManager:

    • persist(Object entity): Speichert eine neue Entität in der Datenbank (entspricht einem INSERT).
    • find(Class entityClass, Object primaryKey): Sucht eine Entität anhand ihrer ID.
    • merge(Object entity): Aktualisiert den Zustand einer Entität in der Datenbank (entspricht einem UPDATE). Wird auch verwendet, um eine detached Entität wieder in den managed Zustand zu überführen.
    • remove(Object entity): Löscht eine Entität aus der Datenbank.
    • createQuery(...): Erstellt eine Abfrage mit JPQL (Java Persistence Query Language).

  • Persistence Context (Persistenzkontext): Der Persistenzkontext ist ein Cache, der alle verwalteten (managed) Entitäten für eine bestimmte EntityManager-Instanz enthält. Wenn Sie eine Entität über den EntityManager laden oder erstellen, wird diese Entität in den Persistenzkontext aufgenommen.

    Wichtige Eigenschaften des Persistenzkontextes:

    • Identität: Innerhalb eines Persistenzkontextes gibt es für dieselbe Datenbankzeile nur ein einziges Entitätsobjekt. Wenn Sie eine Entität mit find() laden und sie sich bereits im Kontext befindet, gibt der EntityManager einfach die bestehende Instanz zurück, ohne die Datenbank erneut abzufragen.
    • Automatische Synchronisation: Der Persistenzkontext verfolgt alle Änderungen an den verwalteten Entitäten. Am Ende einer Transaktion (z.B. beim commit) synchronisiert der EntityManager diese Änderungen automatisch mit der Datenbank. Sie müssen also nicht explizit update-Methoden aufrufen.

2.2 Die Rolle der Persistence Unit (persistence.xml)

Die Persistence Unit ist eine logische Gruppierung von Entitätsklassen und Konfigurationseinstellungen. Sie wird in der Datei persistence.xml definiert, die sich typischerweise im Verzeichnis META-INF Ihrer Anwendung befindet.

Die persistence.xml ist das Konfigurationszentrum für JPA. Sie ist essenziell, um dem JPA-Provider (z.B. Hibernate) mitzuteilen, welche Entitäten er verwalten soll und wie er sich mit der Datenbank verbinden muss.

Typische Inhalte der persistence.xml:

  • <persistence-unit name="mein_persistenzeinheit">: Der Name der Persistenzeinheit. Dieser Name wird verwendet, um einen EntityManager zu erstellen.

  • <provider>org.hibernate.jpa.HibernatePersistenceProvider</provider>: Spezifiziert den JPA-Provider, den Sie verwenden.

  • <class>com.example.Kunde</class>: Listet alle Entitätsklassen auf, die zu dieser Persistenzeinheit gehören. Alternativ kann man in bestimmten Umgebungen (z.B. in der Jakarta EE) auch das Scannen des Classpaths aktivieren.

  • <properties>: Hier werden die Konfigurationseigenschaften für den Datenbankzugriff definiert.

    Beispiel-Konfiguration (mit Hibernate):

    <persistence xmlns="http://xmlns.jcp.org/xml/ns/persistence"
             xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
             xsi:schemaLocation="http://xmlns.jcp.org/xml/ns/persistence http://xmlns.jcp.org/xml/ns/persistence/persistence_2_1.xsd"
             version="2.1">

    <persistence-unit name="meinePU" transaction-type="RESOURCE_LOCAL">
        <provider>org.hibernate.jpa.HibernatePersistenceProvider</provider>
        <class>com.example.Kunde</class>
        <properties>
            <property name="javax.persistence.jdbc.url" value="jdbc:mysql://localhost:3306/meinedb"/>
            <property name="javax.persistence.jdbc.user" value="root"/>
            <property name="javax.persistence.jdbc.password" value="passwort"/>
            <property name="javax.persistence.jdbc.driver" value="com.mysql.cj.jdbc.Driver"/>

            <property name="hibernate.hbm2ddl.auto" value="update"/>
            <property name="hibernate.show_sql" value="true"/>
        </properties>
    </persistence-unit>
</persistence>

2.3 Lebenszyklus von Entities

Eine Entität kann verschiedene Zustände durchlaufen. Das Verständnis dieser Zustände ist entscheidend für die korrekte Verwendung von JPA.

  1. New / Transient (Neu / Flüchtiger Zustand):

    • Die Entität wurde mit new instanziiert, aber noch nicht mit einem EntityManager verknüpft.
    • Sie hat noch keine Primärschlüssel-ID.
    • In diesem Zustand ist das Objekt flüchtig und existiert nur im Speicher der Anwendung, nicht in der Datenbank oder im Persistenzkontext.

    Übergang: entityManager.persist(entity)

  2. Managed (Verwaltet):

    • Die Entität ist an einen EntityManager und dessen Persistenzkontext gebunden.
    • Änderungen an der Entität werden vom EntityManager verfolgt und am Ende der Transaktion automatisch in die Datenbank synchronisiert.
    • Sie können Entitäten in diesen Zustand versetzen, indem Sie persist() für eine neue Entität aufrufen oder eine Entität mit find() aus der Datenbank laden.

    Übergang:

    • entityManager.detach(entity) oder
    • Schließen des EntityManager oder der Transaktion.

  3. Detached (Abgetrennt):

    • Die Entität wurde einst vom EntityManager verwaltet, ist aber nicht mehr mit einem Persistenzkontext verbunden.
    • Änderungen an der Entität in diesem Zustand werden nicht automatisch in der Datenbank gespeichert.
    • Die Entität hat immer noch ihre ID.

    Übergang:

    • entityManager.merge(entity): Bringt eine detached Entität wieder in den managed Zustand. JPA vergleicht die Entität mit dem Zustand in der Datenbank und führt bei Änderungen einen UPDATE durch.

  4. Removed (Entfernt):

    • Die Entität ist zum Löschen markiert.
    • Der EntityManager hat die Methode remove() für diese Entität aufgerufen.
    • Die tatsächliche Löschung in der Datenbank erfolgt erst, wenn die Transaktion committet wird.

    Übergang: entityManager.remove(entity)
