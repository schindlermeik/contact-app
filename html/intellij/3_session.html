Handout: Debugging und Versionskontrolle in IntelliJ IDEA

1. Debugging in IntelliJ IDEA

1.1. Setzen und Verwalten von Breakpoints

Ein Breakpoint (Haltepunkt) stoppt die Ausführung deines Programms an einer bestimmten Stelle, damit du den aktuellen Zustand untersuchen kannst.

So setzt du einen Breakpoint:

  • Klicke links neben die Codezeile (auf den grauen Rand) – ein roter Punkt erscheint.
  • Alternativ: Cursor in die Zeile und Strg + F8 (Windows/Linux) bzw. ⌘ + F8 (macOS) drücken.

Breakpoints verwalten:

  • Klicke mit der rechten Maustaste auf einen Breakpoint für weitere Optionen (z. B. „Condition“, um bedingte Breakpoints zu setzen).
  • Alle Breakpoints lassen sich in der „Breakpoints“-Ansicht (Strg + Shift + F8 / ⌘ + Shift + F8) verwalten.

1.2. Debugging starten

Programm im Debug-Modus starten:

  • Klicke auf das Käfer-Symbol (grün) oder Shift + F9.
  • Alternativ: Rechtsklick auf die Datei → „Debug 'Klassenname.main()'“.

Ablauf:

  • Das Programm läuft bis zum nächsten Breakpoint und hält dort an.
  • Die Debug-Ansicht öffnet sich unten in IntelliJ.

1.3. Schrittweise Ausführung des Codes

Im Debug-Modus kannst du dein Programm Zeile für Zeile oder blockweise durchgehen:

Aktion Windows/Linux macOS Beschreibung
Step Over F8 F8 Nächste Zeile ausführen, Methode überspringen
Step Into F7 F7 In die aufgerufene Methode „hineinspringen“
Step Out Shift + F8 Shift + F8 Methode beenden und zurückspringen
Resume Program F9 ⌘ + Option + R Fortsetzen bis zum nächsten Breakpoint

1.4. Überprüfen von Variablen und Ausdrücken

Variablen beobachten:

  • Im Debug-Fenster siehst du links alle aktuellen Variablenwerte.
  • Mit Doppelklick auf eine Variable kannst du ihren Wert inspizieren.

Werte „on the fly“ prüfen:

  • Fahre mit der Maus über eine Variable im Editor – ein Popup zeigt den aktuellen Wert.

Expressions auswerten:

  • Klicke im Debug-Fenster auf „Evaluate Expression“ (Lupe) oder nutze Alt + F8.
  • Hier kannst du beliebige Java-Ausdrücke zur Laufzeit auswerten (z. B. Methoden aufrufen, Berechnungen testen).

1.5. Praktische Tipps

  • Du kannst Breakpoints aktivieren/deaktivieren, ohne sie zu löschen (Rechtsklick → Enable/Disable).
  • Conditional Breakpoints: Nur auslösen, wenn eine Bedingung erfüllt ist (z. B. counter == 10).
  • Log Breakpoints: Statt das Programm zu stoppen, wird eine Nachricht ins Log geschrieben.

2. Versionskontrolle (Git) in IntelliJ IDEA

2.1. Einrichtung und Konfiguration

Git auf deinem Rechner installieren:

  • https://git-scm.com/downloads

Git in IntelliJ aktivieren:

  • Öffne dein Projekt.
  • Menü: VCS → Enable Version Control Integration → wähle „Git“.
  • Bei Bedarf Git-Repository initialisieren (git init wird automatisch ausgeführt).

Git-Benutzername und E-Mail festlegen:

  • Menü: File → Settings → Version Control → Git
  • Dort ggf. Username/Email eintragen.

2.2. Grundlegende Git-Operationen

Änderungen (Changes) anzeigen lassen:

  • Ansicht „Version Control“ (unten im Fenster, Tab „Commit“ oder „Local Changes“).
  • Dort siehst du alle geänderten, neuen oder gelöschten Dateien.

a) Committen

Speichert Änderungen in deinem lokalen Repository.

  • Shortcut: Strg + K (Windows/Linux), ⌘ + K (macOS)
  • Oder: Rechtsklick im Changes-Tab → „Commit…“
  • Commit-Nachricht eingeben, ggf. einzelne Dateien auswählen.
  • Klicke „Commit“ oder „Commit and Push“, um die Änderungen auch gleich zu versenden.

b) Änderungen Pushen und Pullen

  • Push (Hochladen):

    • Überträgt deine Commits auf den zentralen Server (z. B. GitHub).
    • Shortcut: Strg + Shift + K (Windows/Linux), ⌘ + Shift + K (macOS)

  • Pull (Herunterladen):

    • Holt Änderungen anderer aus dem Remote-Repository.
    • Menü: VCS → Git → Pull, oder Klick auf das Pull-Symbol.

c) Branches erstellen und wechseln

  • Klicke unten rechts auf den aktuellen Branch-Namen.
  • Wähle „New Branch…“, gib einen Namen ein und bestätige.
  • Zum Wechseln einfach einen anderen Branch anklicken und „Checkout“ wählen.

d) Ansicht der Code-Änderungen (Diff)

  • Im Changes/Commit-Fenster wähle eine Datei aus und klicke auf das „Show Diff“-Symbol (zwei Pfeile).
  • Zeigt alte (links) und neue (rechts) Version direkt an, Unterschiede sind farbig hervorgehoben.

2.3. Praktische Tipps

  • .gitignore:

    • Lege eine .gitignore-Datei im Projekt an, um bestimmte Dateien/Ordner von der Versionskontrolle auszuschließen (z. B. *.class, /out, .idea).

  • Konflikte lösen:

    • IntelliJ bietet grafische Tools zum Zusammenführen („Merge“) von Konflikten, die beim Pull entstehen können.

  • History und Blame:

    • Mit Rechtsklick auf eine Datei → „Git → Show History“ siehst du alle Änderungen.
    • „Annotate/Blame“ zeigt, welche Zeile wann von wem geändert wurde.

Übungsaufgaben (zum Selbststudium)

  1. Setze mindestens drei Breakpoints in deinem Code und starte das Debugging.
  2. Nutze Step Over und Step Into, um den Ablauf deiner Anwendung nachzuvollziehen.
  3. Prüfe und verändere während des Debuggens Variablenwerte im Debugger-Fenster.
  4. Lege ein Git-Repository für dein Projekt an und committe erste Änderungen.
  5. Erstelle einen neuen Branch, wechsle zu ihm, mache Änderungen und merge zurück in den Haupt-Branch.
  6. Nutze die Diff-Ansicht, um Änderungen zwischen zwei Versionen zu sehen.

Tipp: Die Git-Integration von IntelliJ bietet fast alle Standard-Workflows. Für komplexere Szenarien (z. B. Rebase, Cherry-Pick) lohnt ein Blick in die Online-Doku oder ein separates Git-Tool.

Mit diesen Kenntnissen kannst du Code gezielt debuggen und deine Projekte sicher versionieren. Viel Erfolg beim Ausprobieren!
