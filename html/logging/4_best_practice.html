Handout: Best Practices & typische Fehler beim Logging

4.1 Do’s & Don’ts beim Logging

Sensible Daten nicht loggen

  • Do: Achte darauf, dass in Logs keine vertraulichen oder personenbezogenen Daten landen.

  • Beispiele sensibler Daten:

    • Passwörter
    • Kreditkartennummern
    • Bankdaten
    • vollständige Adressen
    • Zugangstoken/API-Keys

  • Warum? Logs werden oft zentral gespeichert und von vielen gelesen – ein Datenleck durch zu „redselige“ Logs ist ein echtes Risiko!

Falsch:

logger.info("User {} logged in with password {}", username, password);

Richtig:

logger.info("User {} logged in", username);

Logger nur einmal pro Klasse initialisieren

  • Do: Erzeuge pro Klasse EINEN Logger als private static final Variable.
  • Warum? Effizient, konsistent und du bekommst im Log den richtigen Klassennamen.
  • Don’t: Logger nicht dynamisch/mehrfach erzeugen!

Richtig:

private static final Logger logger = LoggerFactory.getLogger(MyClass.class);

Falsch:

Logger logger = LoggerFactory.getLogger(...); // innerhalb jeder Methode neu!

Keine aufwendigen Berechnungen in Log-Ausdrücken

  • Problem: Code in einem Log-Statement wird immer ausgeführt, auch wenn das Log-Level gar nicht aktiviert ist! Das kann unnötig Performance kosten.
  • Best Practice: Nutze Platzhalter ({}) im Log-Statement – die Berechnung findet nur statt, wenn das Level auch wirklich geloggt wird.

Falsch:

logger.debug("List: " + list.stream().map(...).collect(Collectors.joining(",")));

Richtig:

logger.debug("List: {}", list); // oder
if (logger.isDebugEnabled()) {
    logger.debug("List: " + list.stream().map(...).collect(Collectors.joining(",")));
}

4.2 Best Practices beim Einsatz von Log-Levels

Warnungen und Fehler nicht für „normale“ Abläufe nutzen

  • ERROR und WARN sind für echte Fehler und Probleme reserviert, nicht für „normale“ Betriebszustände!
  • Sonst: Logfiles werden unübersichtlich, Monitoring schlägt ständig falschen Alarm.

Falsch:

logger.error("Benutzer hat sich erfolgreich angemeldet."); // KEIN Fehler!

Richtig:

logger.info("Benutzer {} angemeldet", username);

Wann TRACE, DEBUG, INFO sinnvoll sind

  • TRACE: Nur für sehr detaillierte Ablaufverfolgung, z. B. Schleifen, Algorithmen – meist nur in Entwicklung/Test sinnvoll.
  • DEBUG: Für technische Detailinfos, die bei Fehleranalyse helfen (z. B. Variablenwerte, Entscheidungen im Code).
  • INFO: Für normale Betriebsinformationen (Start, Stopp, Status, Nutzeraktionen).
  • WARN: Für unerwartete, aber nicht kritische Situationen (z. B. Netzwerk kurz weg, Config fehlt, wird Default genommen).
  • ERROR: Für alles, was kritisch ist, nicht mehr „normal“ läuft (Exceptions, Datenverlust, Absturz).

Beispiel:

logger.trace("Verarbeite Element {}", element);
logger.debug("Wert von a: {}", a);
logger.info("Service gestartet");
logger.warn("Config nicht gefunden, nutze Default");
logger.error("Datei konnte nicht geladen werden!", e);

4.3 Hinweis auf Security (Log Injection)

Was ist Log Injection?

  • Angreifer schleusen manipulierte Daten in Logs ein, z. B. mit speziellen Zeichen, Escape-Sequenzen oder sogar Schadcode.

  • Ziel:

    • Manipulation der Logausgabe (z. B. fälschlicher Eintrag „ERROR“ erzeugen)
    • Im schlimmsten Fall: Ausnutzen von Schwachstellen (siehe Log4Shell 2021)

Fallbeispiel: Log4Shell

  • 2021 wurde eine schwere Sicherheitslücke in Log4j bekannt, bei der Angreifer beliebigen Code auf Servern ausführen konnten – allein durch speziell formatierte Log-Nachrichten!
  • Ursache: Ungeprüfte/unsichere Verarbeitung von Benutzereingaben im Logging.

Schutzmaßnahmen:

  • Immer SLF4J/Logback aktuell halten!
  • Keine Benutzereingaben oder externe Daten ungefiltert ins Log schreiben.
  • Keine Pattern/Lookups erlauben, die zu gefährlichem Verhalten führen könnten.
  • Im Zweifel: Logging-Eingaben „säubern“ (Escape/Validierung).
  • Niemals Logs als „vertrauenswürdig“ ansehen!

Zusammenfassung & Tipps

  • Schreibe Logs mit Bedacht – sie sind Werkzeug und Risiko zugleich!
  • Halte dich an die Trennung der Log-Levels.
  • Logger immer korrekt initialisieren.
  • Log-Ausdrücke so effizient wie möglich halten.
  • Sensible und externe Daten nur sehr bewusst loggen.
  • Sicherheitsupdates bei Logging-Frameworks immer schnellstmöglich einspielen!

Selbststudium:

  • Prüfe dein Projekt: Werden sensible Daten geloggt?
  • Welches Log-Level verwenden deine wichtigsten Statements? Gibt es Verbesserungsbedarf?
  • Weißt du, wie du eine Schwachstelle im Logging erkennen würdest?
