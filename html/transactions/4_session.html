Handout: Transaktionen und Panache in Quarkus

4.1 Transaktionale Methoden mit @Transactional in Quarkus

Warum @Transactional in Quarkus?

  • Die Annotation @Transactional sorgt dafür, dass alle Datenbankoperationen innerhalb einer Methode in einer Transaktion ausgeführt werden.
  • In Quarkus wird damit das Transaktionsmanagement von JPA/Hibernate für dich übernommen – egal ob du mit klassischen Entities oder Panache arbeitest.

Praxisbeispiel mit Panache

import io.quarkus.hibernate.orm.panache.PanacheEntity;
import javax.transaction.Transactional;
import javax.enterprise.context.ApplicationScoped;

@Entity
public class User extends PanacheEntity {
    public String name;
}

@ApplicationScoped
public class UserService {

    @Transactional
    public void createUser(String name) {
        User user = new User();
        user.name = name;
        user.persist();  // Persist läuft in einer Transaktion
    }

    @Transactional
    public void transferName(Long fromId, Long toId) {
        User from = User.findById(fromId);
        User to = User.findById(toId);
        if (from != null && to != null) {
            to.name = from.name;
            to.persist();
        }
    }
}

Was passiert hier?

  • Jeder Aufruf einer @Transactional-Methode läuft automatisch innerhalb einer neuen (oder bestehenden) Transaktion.
  • Tritt ein Fehler auf (Exception), wird ein Rollback ausgelöst.

4.2 Automatisches vs. manuelles Transaktionsmanagement

Automatisches Transaktionsmanagement

Empfohlen und Standard in Quarkus!

  • Mit der @Transactional-Annotation startet und beendet Quarkus die Transaktion automatisch.
  • Bei Erfolg: Commit, bei Fehler: Rollback.
  • Entwickler muss sich nicht um die Details kümmern.

Beispiel:

@Transactional
public void doWork() {
    // beliebige persistente Operationen...
}

Vorteile:

  • Weniger Fehlerquellen, da keine Vergesslichkeit beim Commit/Rollback.
  • Kürzerer, klarer Code.

Manuelles Transaktionsmanagement

Nicht empfohlen, aber möglich.

  • Du kannst (wie bei JPA) auch manuell Transaktionen starten und beenden, aber das ist in Quarkus selten notwendig und wird eher für Spezialfälle genutzt.
  • Panache und Quarkus sind so gebaut, dass 99% aller Anwendungsfälle mit automatischem Management abgedeckt sind.

Manuelles Beispiel (nur ausnahmsweise):

import javax.persistence.EntityManager;
import javax.persistence.PersistenceContext;
import javax.transaction.UserTransaction;

@PersistenceContext
EntityManager em;

@Resource
UserTransaction utx;

public void manualTxExample() {
    try {
        utx.begin();
        // ... DB-Operationen ...
        utx.commit();
    } catch(Exception e) {
        utx.rollback();
    }
}

Achtung:

  • Viel aufwändiger und fehleranfällig.
  • Automatisches Management ist fast immer die bessere Wahl.

4.3 Besonderheiten und Tipps bei Panache

A) Panache und Transaktionen – Besonderheiten

  • CRUD-Operationen (persist, delete, update, etc.) funktionieren nur zuverlässig innerhalb einer Transaktion.
  • @Transactional gehört auf jede Methode, die Daten ändern (persist, update, delete).
  • Lesende Methoden (z. B. reine find-Aufrufe) brauchen keine Transaktion, können aber für Konsistenz in manchen Szenarien trotzdem davon profitieren.

B) Fehler- und Rollback-Verhalten

  • Bei Unchecked Exceptions (RuntimeException und Unterklassen) wird die Transaktion automatisch zurückgerollt.
  • Für Checked Exceptions (z. B. IOException) kann das Verhalten mit @Transactional(rollbackOn = Exception.class) angepasst werden.

C) Typische Fehlerquellen vermeiden

  • Keine persistierenden Operationen außerhalb von @Transactional-Methoden! Sonst kommt es zu LazyInitializationException, fehlenden Commits oder unerwartetem Verhalten.
  • Transaktionen kurz halten! Lange laufende Transaktionen können Deadlocks, Sperren und Performanceprobleme verursachen.
  • Kein Netzwerk oder UI innerhalb der Transaktion! Halte Transaktionen auf die notwendigen Datenbankoperationen beschränkt.

D) Transaktionen und REST

  • In Quarkus wird für jede HTTP-Request-Methode keine automatische Transaktion gestartet!
  • Du musst selbst mit @Transactional steuern, welche Endpunkte transaktional arbeiten sollen.

Beispiel für eine REST-Resource:

@Path("/users")
@ApplicationScoped
public class UserResource {

    @POST
    @Transactional
    public Response addUser(User user) {
        user.persist();
        return Response.ok().build();
    }
}

E) Erweiterte Tipps

  • Kombiniere Service-Schichten: Geschäftslogik sollte in Service-Klassen gekapselt und dort transaktional gesteuert werden.
  • Testing: Nutze @Transactional auch in Testmethoden, um Testdaten nach dem Testlauf wieder aufgeräumt zu bekommen.

Zusammenfassung

  • @Transactional ist in Quarkus das zentrale Werkzeug für Transaktionsmanagement – insbesondere mit Panache.
  • Das automatische Transaktionsmanagement deckt fast alle Anwendungsfälle ab und ist weniger fehleranfällig als manuelles Management.
  • Achte auf die richtige Platzierung von @Transactional – immer auf Methoden, die persistente Änderungen ausführen.
  • Halte Transaktionen kurz und fokussiert – das sorgt für Performance und weniger Fehler.
