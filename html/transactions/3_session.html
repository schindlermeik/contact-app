Handout: Transaktionen mit JPA/Hibernate

3.1 Das Transaktionsmodell von JPA/Hibernate

Was ist eine Transaktion in JPA/Hibernate?

  • JPA (Java Persistence API) ist eine standardisierte API für ORM (Object-Relational Mapping) in Java.
  • Hibernate ist eine weitverbreitete Implementierung von JPA.
  • Datenbankoperationen werden grundsätzlich in Transaktionen ausgeführt – egal ob Einzeloperationen oder komplexe Geschäftslogik.

Wie funktionieren Transaktionen mit JPA?

  • Das Herzstück ist der EntityManager.
  • Mit ihm wird der Lebenszyklus von Entitäten und Transaktionen gesteuert.
  • Der EntityManager arbeitet im Zusammenhang mit einem Persistence Context (Verwaltungsbereich für Objekte im RAM).

Typischer Ablauf:

  1. EntityManager starten
  2. Transaktion beginnen
  3. Operationen durchführen (persist, merge, remove, find, etc.)
  4. Transaktion committen (oder bei Fehler rollback)
  5. EntityManager schließen

Beispiel (klassisch, ohne Framework):

EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();

try {
    tx.begin();
    em.persist(new User("Anna"));
    tx.commit();
} catch (Exception e) {
    tx.rollback();
} finally {
    em.close();
}

3.2 Verwendung von @Transactional (in Java EE, Spring, Quarkus etc.)

Warum @Transactional?

  • Annotation @Transactional kapselt den Start und das Ende einer Transaktion automatisch.
  • Sie sorgt dafür, dass alle Datenbank-Operationen in der Methode in einer Transaktion laufen.
  • Sehr beliebt in modernen Frameworks (Java EE/Jakarta EE, Spring, Quarkus).

Beispiel:

Java EE / Jakarta EE:

import javax.transaction.Transactional;

@Stateless // EJB-Bean
public class UserService {

    @PersistenceContext
    EntityManager em;

    @Transactional
    public void createUser(User user) {
        em.persist(user);
        // Noch mehr DB-Operationen ...
    }
}

Spring:

import org.springframework.transaction.annotation.Transactional;

@Service
public class UserService {

    @Autowired
    EntityManager em;

    @Transactional
    public void register(User user) {
        em.persist(user);
        // Noch mehr DB-Operationen ...
    }
}

Quarkus:

import javax.transaction.Transactional;

@ApplicationScoped
public class UserService {

    @Transactional
    public void saveUser(User user) {
        user.persist();
        // Auch Panache-Methoden laufen dann in einer Transaktion
    }
}

Was passiert?

  • Das Framework beginnt automatisch eine Transaktion, bevor die Methode startet.
  • Bei erfolgreichem Durchlauf: Commit.
  • Bei Exception: Rollback.
  • Entwickler muss sich nicht um explizite Transaktionssteuerung kümmern.

3.3 Abgrenzung: Container-Managed vs. Application-Managed Transactions

A) Container-Managed Transactions (CMT)

  • Das Framework (Container) steuert den Transaktionsablauf.
  • Typisch bei Enterprise-Java-Anwendungen (Java EE, Spring, Quarkus).
  • Mit Annotationen wie @Transactional oder EJB-Annotations (@Stateless).

Vorteile:

  • Einfach, wenig Boilerplate.
  • Der Entwickler konzentriert sich auf die Geschäftslogik.

Beispiel:

@Stateless
public class PaymentService {
    @Transactional
    public void transfer(...) { ... }
}

B) Application-Managed Transactions (AMT)

  • Entwickler steuert den EntityManager und die Transaktionen selbst.
  • Besonders in einfachen Setups, bei Migrationsprojekten, bei sehr feingranularer Kontrolle.

Vorteile:

  • Volle Kontrolle über Transaktionsstart und -ende.
  • Sinnvoll für spezielle, nicht standardkonforme Szenarien.

Beispiel:

EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();
tx.begin();
// ... Datenbankoperationen
tx.commit();

3.4 Rollback-Strategien und Fehlerbehandlung

Rollback in @Transactional-Methoden

  • Standard: Bei Unchecked Exceptions (RuntimeException und deren Unterklassen) erfolgt automatisch ein Rollback.
  • Checked Exceptions (z. B. IOException) führen meist nicht automatisch zu einem Rollback – außer, du konfigurierst es explizit!

Spring & Jakarta EE:

@Transactional(rollbackFor = Exception.class)
public void doSomething() { ... }
  • Das bewirkt: Auch bei geprüften Ausnahmen wird zurückgerollt.

Manuelles Rollback:

  • Kann auch durch das Werfen einer Exception forciert werden.

Typische Fehler und Best Practices

  • Fang Exceptions nicht zu früh ab! Lasse Fehler so lange wie möglich „nach oben wandern“, damit Transaktionsmanagement sauber greifen kann.

  • Halte Transaktionen so kurz wie möglich. (Vermeide: unnötige Schleifen, Netzwerkaufrufe, UI-Operationen in Transaktionen.)

  • Keine Nebenwirkungen außerhalb der Transaktion! E-Mails versenden, Events triggern etc. sollte erst nach Commit passieren, da es sonst zu Inkonsistenzen kommen kann.

  • Nutze Logging, um Rollbacks und Exception-Flows nachvollziehen zu können.

  • Achte auf Nebenläufigkeit und Deadlocks: Lange Transaktionen oder konkurrierende Updates können zu Problemen führen.

Zusammenfassung

  • JPA/Hibernate-Transaktionen werden meist mit @Transactional gesteuert – Framework übernimmt Commit und Rollback.
  • Container-Managed (z. B. mit Quarkus, Spring, Java EE) ist Standard – weniger Fehlergefahr, mehr Produktivität.
  • Für feingranulare Spezialfälle kannst du Application-Managed Transactions nutzen.
  • Rollback bei Fehlern ist (bei unchecked Exceptions) Standard – bei checked Exceptions ggf. rollbackFor explizit setzen.
  • Best Practices: Transaktionen klein halten, Fehler sauber behandeln, keine Nebenwirkungen außerhalb der Transaktion.
