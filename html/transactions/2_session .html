Handout: Transaktionen in klassischen JDBC-Anwendungen

2.1 Manuelles Transaktionsmanagement (setAutoCommit, commit, rollback)

JDBC und Transaktionen – Grundlagen

In JDBC ist das Transaktionsmanagement standardmäßig manuell. Das bedeutet:

  • Jede SQL-Anweisung wird (ohne Änderung der Einstellungen) sofort nach Ausführung gespeichert (Auto-Commit-Modus).
  • Möchtest du mehrere Anweisungen als eine Transaktion ausführen, musst du Auto-Commit deaktivieren und die Transaktion selbst steuern.

Wichtige Methoden

  • setAutoCommit(false) Schaltet den Auto-Commit-Modus ab – ab jetzt müssen Transaktionen explizit beendet werden.
  • commit() Schreibt alle seit dem letzten Commit gemachten Änderungen dauerhaft in die Datenbank.
  • rollback() Macht alle Änderungen seit dem letzten Commit rückgängig.

Typischer Ablauf in JDBC:

  1. Verbindung öffnen
  2. Auto-Commit deaktivieren
  3. SQL-Anweisungen ausführen
  4. Commit, wenn alles erfolgreich
  5. Rollback bei Fehler
  6. Verbindung schließen

2.2 Beispiel: Transaktionen in JDBC (Try-Catch, Fehlerfälle)

Praxisbeispiel: Angenommen, du willst eine Banküberweisung abbilden:

  1. Betrag von Konto A abziehen
  2. Betrag zu Konto B hinzufügen

Das Ganze soll entweder komplett klappen oder gar nicht.

import java.sql.*;

public class JdbcTransactionExample {
    public static void main(String[] args) {
        String url = "jdbc:postgresql://localhost:5432/bankdb";
        String user = "demo";
        String password = "geheim";

        Connection conn = null;
        try {
            conn = DriverManager.getConnection(url, user, password);
            // Transaktion starten
            conn.setAutoCommit(false);

            // 1. Betrag von Konto A abziehen
            PreparedStatement withdraw = conn.prepareStatement(
                "UPDATE konto SET saldo = saldo - ? WHERE id = ?");
            withdraw.setDouble(1, 100);
            withdraw.setInt(2, 1);
            withdraw.executeUpdate();

            // 2. Betrag zu Konto B hinzufügen
            PreparedStatement deposit = conn.prepareStatement(
                "UPDATE konto SET saldo = saldo + ? WHERE id = ?");
            deposit.setDouble(1, 100);
            deposit.setInt(2, 2);
            deposit.executeUpdate();

            // Alles erfolgreich? Dann commit!
            conn.commit();
            System.out.println("Überweisung erfolgreich!");

        } catch (SQLException e) {
            // Fehler? Dann alles rückgängig machen
            if (conn != null) {
                try {
                    conn.rollback();
                    System.out.println("Fehler aufgetreten – Transaktion rückgängig gemacht!");
                } catch (SQLException ex) {
                    System.err.println("Rollback fehlgeschlagen: " + ex.getMessage());
                }
            }
            e.printStackTrace();
        } finally {
            // Ressourcen schließen
            if (conn != null) {
                try { conn.setAutoCommit(true); conn.close(); } catch (SQLException e) { }
            }
        }
    }
}

Wichtige Hinweise zum Beispiel:

  • Auto-Commit muss vor der ersten Anweisung deaktiviert werden.
  • Alle SQL-Statements werden innerhalb desselben Connection-Objekts ausgeführt.
  • Ein Fehler (z. B. Konto nicht gefunden, SQL-Fehler) führt zum Rollback.
  • Nach Abschluss sollte Auto-Commit wieder aktiviert und die Verbindung geschlossen werden.
  • Immer Ressourcen korrekt schließen (am besten try-with-resources).

2.3 Typische Fehler und Best Practices

Häufige Fehlerquellen:

  1. Auto-Commit vergessen zu deaktivieren → Jede Operation wird einzeln ausgeführt, keine echte Transaktion!

  2. Verbindung nicht korrekt geschlossen → „Connection Leaks“: Datenbank-Server verweigert irgendwann neue Verbindungen.

  3. Fehler bei Rollback nicht abgefangen → Ein Fehler beim Rollback kann zu weiteren Problemen führen; daher immer try-catch beim Rollback.

  4. Gemischte Ressourcen/Verbindungen → Alle Anweisungen müssen auf derselben Connection laufen!

  5. Transaktionen zu groß oder zu lang → Zu lange offene Transaktionen erhöhen das Risiko für Deadlocks, Sperren und Performanceprobleme.

Best Practices:

  • Immer explizit Transaktionen beginnen und beenden (commit/rollback).
  • Nutze try-with-resources für Statements und ResultSets – das schließt Ressourcen zuverlässig.
  • Fasse Transaktionen möglichst klein und auf das Notwendigste zusammen.
  • Fehler immer abfangen und sauber behandeln.
  • Setze Auto-Commit nach der Transaktion zurück auf true (falls die Connection wiederverwendet wird).
  • Nutze Logging, um Transaktionsverläufe und Fehler nachvollziehen zu können.

Zusammenfassung

  • Transaktionsmanagement ist in JDBC manuell – das gibt maximale Kontrolle, erfordert aber sauberes Arbeiten!
  • Mit setAutoCommit(false), commit() und rollback() steuerst du den Ablauf.
  • Fehler und unsaubere Handhabung können zu Datenverlust, Inkonsistenzen oder „Connection Leaks“ führen.
  • Mit Best Practices und sauberem Code ist sicheres Arbeiten garantiert.
