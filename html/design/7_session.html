Handout: Praxis und Übungen – Modularisierung in Java

7.1 Entwurf einer modularen Beispielanwendung

Aufgabenstellung:

  • Entwirf ein Softwaresystem, das aus mehreren klar abgegrenzten Modulen besteht, z. B.:

    • Ein Plugin-System (z. B. für ein Notizbuch oder eine Mini-IDE)
    • Ein Mini-Microservice-System (z. B. „User“- und „Order“-Service)

Vorgehen:

  • Identifiziere Kernfunktionen und trenne sie in eigenständige Module (z. B. „core“, „plugin-api“, „plugin-impl“)
  • Definiere die Schnittstellen/Verträge (APIs) für die Kommunikation zwischen den Modulen
  • Dokumentiere, welche Module voneinander abhängen und wie die Kommunikation abläuft (z. B. via Service-Interfaces, Events, REST)

Beispiel für Modulaufteilung (Plugin-System):

  • app-core (zentrale Steuerung, Startpunkt)
  • plugin-api (Schnittstellen für Erweiterungen)
  • plugin-impl-xyz (verschiedene konkrete Plugins)

7.2 Umsetzung eines Projekts mit mehreren Modulen (inkl. JPMS)

Aufgabe:

  • Setze ein kleines Java-Projekt mit mindestens 3 Modulen um.
  • Verwende JPMS (ab Java 9) mit eigenen module-info.java-Dateien.

Schritte:

  1. Projektstruktur anlegen: Erstelle ein Multi-Module-Projekt (z. B. mit Maven).
  2. Module definieren: z. B. user.core, user.api, user.impl
  3. module-info.java anlegen: Exportiere nur die benötigten Packages, importiere gezielt andere Module.
  4. Abhängigkeiten setzen: In der pom.xml und in module-info.java.
  5. Kompilieren und ausführen: Baue das Gesamtprojekt und teste, ob die Module korrekt miteinander arbeiten.

Beispiel:

// In module user.api
module user.api {
    exports com.example.user.api;
}

// In module user.impl
module user.impl {
    requires user.api;
    provides com.example.user.api.UserService with com.example.user.impl.UserServiceImpl;
}

7.3 Simulieren und Auflösen zyklischer Abhängigkeiten

Aufgabe:

  • Erzeuge bewusst eine zyklische Abhängigkeit zwischen zwei Modulen und behebe sie wieder.

Vorgehen:

  1. Simuliere eine zyklische Abhängigkeit, z. B.:

    • modul-a → modul-b
    • modul-b → modul-a
    • (JPMS meldet einen Fehler!)

  2. Löse die Abhängigkeit auf:

    • Extrahiere gemeinsame Schnittstellen in ein drittes Modul (z. B. common.api)
    • Lasse beide ursprünglichen Module von diesem neuen API-Modul abhängen.

Lernziel:

  • Verstehe, warum zyklische Abhängigkeiten problematisch sind, und wie sie mit sauberem API-Design aufgelöst werden können.

7.4 Refactoring einer monolithischen Codebasis zu modularer Struktur

Aufgabe:

  • Nimm eine einfache monolithische Beispielanwendung (z. B. alles in einem Package) und gliedere sie in sinnvolle Module um.

Schritte:

  1. Analysiere die bestehende Codebasis: Welche Verantwortlichkeiten gibt es?
  2. Bestimme Modulgrenzen: Trenne fachliche (User, Bestellung, Produkt) und technische (Persistence, Service) Aspekte.
  3. Extrahiere Schnittstellen: Definiere, was die Module voneinander brauchen (APIs).
  4. Lagere Implementierungen aus: Verschiebe konkrete Klassen in Implementierungs-Module.
  5. Lege module-info.java an: Exportiere/importiere gezielt Packages.
  6. Schreibe und führe Modultests und Integrationstests aus.

7.5 Optional: Integration von externen Modulen/Libraries (Third-Party/Ökosystem)

Aufgabe:

  • Baue ein Modul, das eine externe Bibliothek (z. B. Jackson für JSON, JDBC für Datenbanken) verwendet.

Vorgehen:

  1. Abhängigkeit deklarieren: In der pom.xml des entsprechenden Moduls.
  2. module-info.java anpassen: Füge ggf. ein requires für das externe Modul hinzu (Achtung: Nicht alle Drittanbieter-Libs sind JPMS-fähig!).
  3. Isoliere die externe API: Kapsle die Benutzung in einem eigenen Adapter/Service, damit du sie später leichter ersetzen kannst.
  4. Testen: Schreibe Tests, die sicherstellen, dass die Integration korrekt funktioniert.

Reflexionsfragen

  1. Wie würdest du ein einfaches Plugin-System in Java strukturieren?
  2. Was sind die größten Herausforderungen beim Umstieg von monolithisch zu modular?
  3. Warum sind zyklische Abhängigkeiten im Java Module System ein Problem?
  4. Welche Vorteile bietet die Integration externer Bibliotheken als separate, klar gekapselte Module?

Zusammenfassung

  • Praxisübungen stärken das Verständnis für echte Modularisierung.
  • Modulentwurf, JPMS-Einsatz, API-Design und das bewusste Vermeiden von Zyklik sind zentrale Fähigkeiten.
  • Refactoring alter Codebasen und die Integration von Drittanbieter-Libraries runden das Know-how ab.
  • Der Schritt von der Theorie zur Praxis ist entscheidend für die Entwicklung nachhaltiger, modularer Softwaresysteme.
