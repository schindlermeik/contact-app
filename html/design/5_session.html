Handout: Build, Deployment und Versionierung modularer Anwendungen

5.1 Modulares Bauen mit Maven

Was ist Maven?

  • Ein weit verbreitetes Build-Management- und Projektverwaltungstool für Java.
  • Unterstützt die Strukturierung in Module (Subprojekte) und verwaltet Abhängigkeiten, Builds und Tests.

Modulares Bauen mit Maven

  • In Maven können Module als eigenständige Projekte verwaltet werden, die jeweils ihre eigene pom.xml besitzen.
  • Die Abhängigkeiten (Dependencies) zwischen Modulen werden explizit in der pom.xml definiert.
  • Maven erkennt und baut die Module in der richtigen Reihenfolge.

Beispiel-Struktur:

my-app/
├── pom.xml            # Parent-POM
├── user-module/
│   └── pom.xml
├── order-module/
│   └── pom.xml
└── common-module/
    └── pom.xml

Parent-POM (my-app/pom.xml):

<modules>
    <module>user-module</module>
    <module>order-module</module>
    <module>common-module</module>
</modules>
  • Die Module referenzieren sich gegenseitig über <dependency>-Blöcke in ihren jeweiligen pom.xml.

5.2 Mehrmodulprojekte (Multi-Module Builds)

Was ist ein Multi-Module Build?

  • Ein Maven-Projekt, das aus mehreren Modulen (Subprojekten) besteht.
  • Der Parent-Build orchestriert den Bau aller Module.
  • Ideal für große Anwendungen, bei denen Funktionalität sauber getrennt werden soll.

Vorteile:

  • Klare Trennung und Kapselung von Funktionalität.
  • Getrennte Abhängigkeiten, Versionen und Releases für jedes Modul.
  • Einfache Wiederverwendung von Modulen in anderen Projekten.

Build-Befehl: Im Wurzelverzeichnis ausführen:

mvn clean install
  • Maven baut alle Module automatisch in der richtigen Reihenfolge.

5.3 Versionierung von Modulen, API-Kompatibilität und Semantic Versioning

Warum Versionierung?

  • Module entwickeln sich unabhängig weiter – saubere Versionierung sichert Rückwärtskompatibilität und planbare Updates.

Semantic Versioning (SemVer)

  • Versionsnummern nach dem Schema: MAJOR.MINOR.PATCH (z. B. 2.1.3)

    • MAJOR: Inkompatible API-Änderungen
    • MINOR: Abwärtskompatible Funktions-Erweiterungen
    • PATCH: Fehlerbehebungen, keine API-Änderung

Best Practice:

  • API-Änderungen nur mit neuer Hauptversion (MAJOR) veröffentlichen.
  • Jedes Modul hat eigene Versionsnummer in der pom.xml.

API-Kompatibilität prüfen

  • Tools wie japicmp oder revapi prüfen Änderungen zwischen Modulen/Versionen.
  • Gute Dokumentation der Schnittstellen und Veränderungen ist Pflicht!

5.4 Dynamisches Laden und Entladen von Modulen (z. B. OSGi, Java Module Layer)

Warum dynamisches Laden?

  • In großen, modularen Systemen möchte man Module im laufenden Betrieb hinzufügen, aktualisieren oder entfernen – ohne Systemneustart.

OSGi (Open Services Gateway Initiative)

  • Industriestandard für dynamische Java-Modulsysteme.
  • Module („Bundles“) können zur Laufzeit geladen, entladen und aktualisiert werden.
  • Klare Trennung von API und Implementierung, Service Registry für lose Kopplung.

Beispiel: Ein neues Plugin (als OSGi-Bundle) kann ins System geladen werden und stellt zusätzliche Services bereit.

Java Module Layer (ab Java 9)

  • Das Java Platform Module System erlaubt über ModuleLayer die dynamische Erstellung von Modulen im laufenden Programm.
  • Möglich, z. B. für Plug-in-Architekturen oder für Systemsicherheit (Module mit verschiedenen Rechten laden).

Zusammenfassung

  • Maven erleichtert das modulare Bauen, Testen und Deployen großer Java-Anwendungen.
  • Multi-Module-Builds ermöglichen klare Architektur, bessere Wartbarkeit und Wiederverwendung.
  • Semantic Versioning ist essenziell für planbare Updates und Kompatibilität.
  • Dynamisches Laden und Entladen (z. B. mit OSGi oder Java Module Layer) steigert die Flexibilität und Erweiterbarkeit von Softwaresystemen.
