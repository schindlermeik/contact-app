Handout: Modul-Design und Strukturierung in Java

3.1 Modul-Begriff im Kontext von Java (Packages, JARs, Module seit Java 9)

Packages

  • Seit Java 1.0: Logische Gruppierung verwandter Klassen, Interfaces und Sub-Packages.
  • Kapselung auf Klassenebene mit public, protected, package-private, private.
  • Keine echte „physische“ Abgrenzung oder Abhängigkeitskontrolle auf Paketebene.

JARs (Java Archives)

  • Seit Java 1.1: Bündelung (Archivierung) von Klassen und Ressourcen zu wiederverwendbaren Komponenten.
  • Häufig als „Bibliotheks-Modul“ eingesetzt, aber technisch nur ein Container.
  • Abhängigkeiten werden nicht explizit beschrieben – alles ist von allem zugreifbar, wenn im Classpath.

Module (ab Java 9 – JPMS)

  • Neue Spracheinheit: „Modul“ kapselt Packages und legt explizit fest, was nach außen sichtbar ist.
  • Module werden durch module-info.java beschrieben.
  • Ermöglicht echte Kapselung und gezielte Abhängigkeitskontrolle auf Modul-Ebene.

Beispiel-Entwicklung:

  • Früher: Anwendung → mehrere JARs → viele Packages, aber keine klare Abgrenzung oder Kontrolle
  • Heute: Anwendung → mehrere Module (je Modul: mehrere Packages, ein JAR, explizite Schnittstellen und Abhängigkeiten)

3.2 Das Java Platform Module System (JPMS)

3.2.1 module-info.java und Sichtbarkeit von Paketen

  • Jedes Modul besitzt (im Wurzelverzeichnis) eine module-info.java-Datei.

  • Beispiel:

    module shop.core {
    exports com.myshop.products;
    exports com.myshop.orders;
    requires shop.utils;
}

  • Mit exports werden Packages explizit nach außen sichtbar gemacht – der Rest ist gekapselt!

  • Ohne exports ist ein Package nur innerhalb des eigenen Moduls sichtbar.

3.2.2 Export und Import von Paketen

  • Export (exports): Steuert, welche Packages für andere Module sichtbar sind.

    exports com.myshop.products;

  • Import (requires): Steuert, welche anderen Module benötigt werden.

    requires shop.utils;

  • Sichtbarkeit auf Klassenebene: Nur Klassen aus exportierten Packages sind für andere Module sichtbar. Interne Packages bleiben privat.

  • Restriktiver Export: Seit Java 9 kann gezielt für einzelne Module exportiert werden:

    exports com.myshop.internal to shop.admin; // Nur für bestimmtes Modul sichtbar

3.2.3 Starke Kapselung vs. klassische Packages

  • Klassische Packages:

    • Jedes public-Element ist für jedes andere Package in jedem JAR im Classpath sichtbar.
    • Keine explizite Kontrolle über die Nutzung – „public“ heißt wirklich öffentlich.

  • Starke Kapselung im Modul-System:

    • „public“ ist nur innerhalb des eigenen Moduls und exportierter Packages wirklich öffentlich.
    • Klassen aus nicht-exportierten Packages sind für andere Module nicht zugreifbar – auch wenn sie „public“ sind.
    • Vorteil: Weniger unbeabsichtigte Abhängigkeiten, bessere Wartbarkeit, mehr Kontrolle.

3.3 Best Practices beim Zuschnitt von Modulen

Domain Driven Design (DDD)

  • Module an der Fachlichkeit ausrichten: z. B. „benutzer“, „bestellung“, „produkt“.
  • Jedes Modul kapselt ein konsistentes Stück Geschäftslogik.

Feature Slices

  • Module entlang von Features schneiden (z. B. „warenkorb“, „bezahlung“), nicht nur entlang technischer Layer.
  • Vorteil: Alles, was zu einem Feature gehört, liegt beisammen.

Layered Architecture

  • Klassische Schichtenarchitektur: Module für „core“, „api“, „web“, „service“, „persistence“ etc.
  • Trennung nach Verantwortlichkeiten, oft hilfreich für größere Teams.

Weitere Best Practices

  • Module nicht zu groß („Gott-Modul“), nicht zu klein („Nano-Modul“).
  • Klare Schnittstellen definieren, so wenig wie möglich exportieren.
  • Zyklische Abhängigkeiten vermeiden!
  • Namensgebung: Klar, eindeutig, am besten sprechend für die Domäne.

3.4 Typische Fehler und Stolperfallen bei der Modulbildung

1. Zu große oder zu kleine Module

  • „Gott-Modul“: Alles in einem Modul → keine Vorteile der Modularität.
  • „Nano-Modul“: Jedes Feature/Klasse ein eigenes Modul → Overhead, schwer verständlich.

2. Zyklische Abhängigkeiten

  • Module A benötigt B und B benötigt A → schlecht wartbar, erschwert Testen und Deployment.
  • Vermeidung: Architekturreview, API-Design, technische Checks (Maven Enforcer, JDeps).

3. Falsche oder zu großzügige Exporte

  • Zu viele Packages exportiert → Kapselung verloren, API wird schwer änderbar.
  • Empfehlung: Nur das Nötigste exportieren.

4. Unklare Schnittstellen

  • Unpräzise oder „unsaubere“ APIs führen zu Missverständnissen und Kopplung.
  • Tipp: Schnittstellen als Verträge behandeln; sauber dokumentieren.

5. Unbedachte Abhängigkeiten von externen Bibliotheken

  • Fremdbibliotheken direkt ins eigene API „leaken“ lassen – erschwert Wartung/Migration.
  • Empfehlung: Externe Abhängigkeiten möglichst intern halten oder explizit im API-Design kennzeichnen.

Zusammenfassung

  • Modul-Design in Java beginnt bei der klaren Trennung von Packages, JARs und echten Modulen (ab Java 9).
  • Mit JPMS (Java Platform Module System) lassen sich Abhängigkeiten und Sichtbarkeit exakt steuern – Kapselung und Kontrolle werden gestärkt.
  • Gute Module orientieren sich an Domäne, Features oder Schichten – nie am Zufall!
  • Typische Fehler sind zu große/zu kleine Module, zyklische Abhängigkeiten und unsaubere Schnittstellen.
  • Eine kluge Modulaufteilung legt die Grundlage für wartbare, skalierbare und testbare Softwaresysteme.
